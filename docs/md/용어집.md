## 1. 컴퓨터 구조 기초

### 시스템 콜 (System Call) {#syscall}

- **정의**: 일반 프로그램(앱)이 운영체제(OS) 커널에게 "이 일 좀 대신 해 줘" 라고 부탁하는 공식 통로 — *앱과 커널 사이의 근본 인터페이스*다. `open` 으로 파일 열기, `read`/`write` 로 디스크 읽기/쓰기, `mmap`/`brk` 로 메모리 영역 늘리기, `clone` 으로 스레드 만들기 등이 모두 시스템 콜이다. 동작 방식은 CPU 를 특별한 명령(x86-64 의 `syscall`, ARM 의 `svc`)으로 *트랩(trap)* 시켜 유저 모드에서 커널 모드로 권한을 올리는 것이다. 이 모드 전환에는 비용이 들어 — 보통 수백 ns ~ 수 µs — 같은 일을 잘게 쪼개 시스템 콜을 남발하면 느려진다. **흔한 오해**: `malloc()` 은 시스템 콜이 아니라 C 라이브러리 함수다. 메모리가 부족할 때만 내부적으로 `brk`/`mmap` 시스템 콜을 호출한다.
- **비유**: 식당에서 손님(앱)이 주방(OS)에 "물 좀 더 주세요" 라고 말하는 행위. 손님이 주방에 직접 들어가지는 않고, 정해진 호출 창구로만 요청한다.
- **왜 중요한가**: 모든 자원(메모리·CPU 시간·디스크·네트워크)은 OS 가 관리하므로, 앱이 무슨 사고를 치든 그 흔적은 결국 시스템 콜 어딘가에 남는다. `strace` 로 어떤 시스템 콜에서 멈췄는지(예: OOM 직전 `mmap` 실패, 데드락 시 `futex` 대기) 보는 것이 장애 분석의 출발점이 된다.
  > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [syscalls(2) — Linux manual page](https://man7.org/linux/man-pages/man2/syscalls.2.html) · [System call — Wikipedia](https://en.wikipedia.org/wiki/System_call)

### 운영체제 · 커널 · 유저 공간 {#os-kernel-userspace}

- **정의**:
  - **OS (운영체제)**: 컴퓨터의 모든 자원(CPU·메모리·디스크·네트워크)을 관리하고, 앱들이 안전하게 공유하도록 중재하는 거대한 프로그램. 리눅스·윈도우·macOS 가 OS 다.
  - **커널 (Kernel)**: OS 의 핵심 부분으로, 가장 강한 권한(CPU 의 "특권 모드", x86 에서는 Ring 0)을 가지고 하드웨어를 직접 다루는 코드. "리눅스 커널" 이라고 부르는 그것. 부팅 시 메모리에 올라와 종료까지 상주한다.
  - **유저 공간 (User Space)**: 일반 앱이 실행되는 영역(x86 에서는 Ring 3). 권한이 약해서 하드웨어를 직접 못 만지고, [시스템 콜](#syscall)로 커널에 부탁해야 한다. 두 공간은 *별도의 메모리 영역*으로 분리되어, 유저 앱의 버그가 커널을 직접 깨뜨리지 못하게 막는다(보호, protection).
- **비유**: 운영체제 = 빌딩 전체 시스템. 커널 = 시설관리실(전기·수도·엘리베이터 직접 통제). 유저 공간 = 사무실(직원이 일하지만 시설은 못 만짐).
- **왜 중요한가**: "내가 짠 앱" 은 유저 공간에서 돈다. 메모리 할당·스레드 생성·파일 쓰기 등은 다 커널에 시스템 콜로 부탁하는 거다. 그래서 *앱 로그* 와 *커널 로그(`dmesg`)* 두 군데를 다 봐야 장애 전모를 본다 — 예컨대 OOM Killer 가 프로세스를 죽인 기록은 커널 로그에만 남는다.
  > 더 깊이: [이론 지식 §1 컴퓨터 구조 기초](이론_지식.md)
- **출처**: [User space and kernel space — Wikipedia](https://en.wikipedia.org/wiki/User_space_and_kernel_space)

### 폰 노이만 구조 (von Neumann architecture) {#von-neumann}

- **정의**: 1945년 수학자 John von Neumann 이 「First Draft of a Report on the EDVAC」에서 정리·회람한 컴퓨터 모델로, "코드(명령어)와 데이터를 같은 메모리에 두고, CPU 가 그 메모리에서 한 단어씩 가져와 실행한다" 는 *저장 프로그램(stored-program)* 설계다. 핵심 구성요소는 CPU(연산장치 + 제어장치)·메모리·입출력(I/O)이며, 이들을 *버스(bus)* 가 잇는다. 코드와 데이터가 같은 버스를 공유하기 때문에 CPU 가 명령어를 읽는 동안 데이터를 동시에 못 옮기는 *폰 노이만 병목(von Neumann bottleneck)* 이 생긴다. **참고**: 이 구조의 착상에는 EDVAC 팀의 Eckert·Mauchly 기여도 컸으나, 회람된 보고서에 von Neumann 의 이름만 실려 그의 이름이 붙었다.
- **비유**: 요리사(CPU)와 레시피 + 재료를 같은 책상(메모리) 위에 올려놓고, 요리사가 책상에서 한 줄씩 레시피를 읽으며 재료를 가져다 쓰는 모습.
- **왜 중요한가**: 우리가 지금 쓰는 거의 모든 컴퓨터(노트북·서버·스마트폰)가 이 모델을 따르므로, "코드가 곧 메모리이고, 메모리가 모자라면 코드 실행도 멈춘다" 는 사실이 자연스러워진다. OOM 이 왜 시스템 마비로 이어지는지의 근본 원인.
  > 더 깊이: [이론 지식 §1.2 폰 노이만 병목](이론_지식.md)
- **출처**: [First Draft of a Report on the EDVAC — Wikipedia](https://en.wikipedia.org/wiki/First_Draft_of_a_Report_on_the_EDVAC) · [Von Neumann architecture — Wikipedia](https://en.wikipedia.org/wiki/Von_Neumann_architecture)

### CPU (Central Processing Unit) {#cpu}

- **정의**: 컴퓨터의 "두뇌". 메모리에 적힌 명령어를 *가져오기(fetch) → 해석(decode) → 실행(execute)* 의 명령어 사이클로 하나씩 처리하는 하드웨어 칩. 클록(clock)에 맞춰 동작하며 클록 속도(예: 3 GHz = 초당 30억 사이클)가 한 코어의 기본 처리 속도를 정한다. 오늘날 CPU 는 보통 여러 개의 *코어(core)* 를 담고, 각 코어는 하이퍼스레딩(SMT)으로 2개의 *논리 CPU* 로 보일 수 있다. **흔한 오해**: 클록만 높다고 빠른 게 아니다 — 캐시·분기예측·코어 수가 실제 성능을 좌우한다.
- **비유**: 식당의 요리사. 주문(명령어)을 받아 정해진 동작(더하기·곱하기·메모리 읽기 등)을 수행한다.
- **왜 중요한가**: CPU 가 100% 점유되면 다른 작업이 못 들어간다. 이번 미션의 CPU Spike 가 그 결과다. `top` 의 `%CPU` 가 코어 수 × 100% 까지 올라갈 수 있다는 점([CPU-bound](#cpu-vs-io-bound) 참고)도 코어 개념에서 나온다.
  > 더 깊이: [이론 지식 §3 CPU](이론_지식.md)
- **출처**: [Central processing unit — Wikipedia](https://en.wikipedia.org/wiki/Central_processing_unit)

### 스루풋 (Throughput) vs 지연시간 (Latency) {#throughput-vs-latency}

- **정의**:
  - **스루풋(throughput)**: 단위 시간당 처리 가능한 작업량 (예: 초당 요청 1만 건, RPS).
  - **지연시간(latency)**: 작업 하나가 시작해서 끝나는 데 걸리는 시간 (예: 요청 한 건당 5ms).
  - 둘은 단순 역수가 아니다. 파이프라인·병렬 처리가 있으면 *지연시간이 길어도 스루풋은 높을 수* 있다(공장 컨베이어처럼). 부하가 높아져 큐가 쌓이면 스루풋은 한계에 머물고 지연시간만 폭증한다(Little's Law: 평균 대기 건수 = 스루풋 × 지연시간).
- **비유**: 식당에서 스루풋 = "1시간에 손님 몇 명 받나", 지연시간 = "손님 한 명이 음식 받기까지 얼마나 걸리나".
- **왜 중요한가**: 둘이 다른 개념인데 흔히 헷갈린다. CPU 코어가 많아지면 스루풋이 늘지만 한 작업의 지연시간은 그대로일 수 있다. 관제 지표에서 "처리량은 멀쩡한데 p99 지연만 튄다" 같은 패턴을 읽으려면 이 구분이 필수다.
- **출처**: [Latency (engineering) — Wikipedia](https://en.wikipedia.org/wiki/Latency_%28engineering%29)

### 시간 단위 (ns / µs / ms / s) {#time-units}

- **정의**:
  - 1 초 (s) = 1,000 밀리초 (ms)
  - 1 밀리초 (ms) = 1,000 마이크로초 (µs)
  - 1 마이크로초 (µs) = 1,000 나노초 (ns)
  - 즉 1 초 = 1,000,000,000 나노초. 각 접두어는 SI 표준 10진 접두어(milli=10⁻³, micro=10⁻⁶, nano=10⁻⁹)다.
- **비유**: 1초가 1년이라면, 1ms = 약 8시간 46분, 1µs = 약 30초, 1ns = 약 0.03초.
- **왜 중요한가**: CPU 한 클록 사이클(약 0.3 ns)과 HDD 탐색(약 10 ms)의 차이는 **약 3,300만 배**다. 메모리·캐시·디스크·네트워크의 지연시간 격차가 이 척도 위에서 벌어지므로, "작은 단위라고 무시하면 안 된다" — 어디서 시간을 잡아먹는지 직관을 잡는 기준자다.
  > 더 깊이: [이론 지식 §1.2 폰 노이만 병목](이론_지식.md)
- **출처**: [SI prefixes — BIPM](https://www.bipm.org/en/measurement-units/si-prefixes) · [Latency Numbers Every Programmer Should Know — GitHub gist (jboner)](https://gist.github.com/jboner/2841832)

### 동시성 (Concurrency) vs 병렬성 (Parallelism) {#concurrency-vs-parallelism}

- **정의**:
  - **동시성(concurrency)**: 여러 일을 *논리적으로 겹쳐* 다루는 구조. 코어가 1개여도 [스케줄러](#scheduler)가 짧은 시간 단위로 [스레드](#thread)들을 번갈아 돌리면(시분할, time-slicing) *겉으로는* 동시 같다. "동시에 진행 중인 일을 *다루는* 능력"이지 반드시 같은 순간 실행을 뜻하진 않는다.
  - **병렬성(parallelism)**: *진짜로* 같은 시각에 여러 일이 실행되는 상태. 물리적으로 멀티코어(또는 멀티 CPU)에서만 가능하다.
- **비유**: 동시성 = 1명의 요리사가 라면·김밥·떡볶이를 30초마다 번갈아 만들기. 병렬성 = 요리사 3명이 진짜 동시에 만들기.
- **왜 중요한가**: 데드락은 *동시성* 이 만드는 문제이지 *병렬성* 이 만드는 게 아니다. 1 코어 머신에서도 멀티스레드면 데드락 가능. 집합 관계는 동시성 ⊃ 병렬성(병렬성은 동시성의 한 실현 방식).
  > 더 깊이: [이론 지식 §4.1 동시성 vs 병렬성](이론_지식.md)
- **출처**: [Concurrency (computer science) — Wikipedia](https://en.wikipedia.org/wiki/Concurrency_%28computer_science%29)

### CPU-bound vs I/O-bound {#cpu-vs-io-bound}

- **정의**:
  - **CPU-bound 작업**: 대부분의 시간을 *계산* 에 쓰는 작업. 이미지 처리, 압축, 암호화, 수치 시뮬레이션. 더 빠른/더 많은 CPU 코어를 주면 빨라진다.
  - **I/O-bound 작업**: 대부분의 시간을 *기다림* 에 쓰는 작업. HTTP 요청, DB 쿼리, 파일 읽기/쓰기, 네트워크 송수신. CPU 를 늘려도 별 소용없고, 대기를 겹치게(오버랩) 만들어야 빨라진다.
  - 구분법: 작업 중 `%CPU` 가 거의 100% 면 CPU-bound, CPU 는 한가한데 응답만 느리면 I/O-bound 쪽이다.
- **비유**: CPU-bound = 시험 문제를 계속 푸는 학생. I/O-bound = 식당에서 주문하고 음식 나오기를 기다리는 손님.
- **왜 중요한가**: 어느 쪽인지에 따라 최적의 병렬화 전략이 다르다. CPU-bound 는 [멀티프로세싱](#py-multiprocessing) 이 답이고, I/O-bound 는 [asyncio](#asyncio) 나 멀티스레드가 답. Python 의 [GIL](#gil) 은 한 번에 한 스레드만 바이트코드를 실행시켜 CPU-bound 멀티스레드의 병렬 가속을 막지만, 스레드가 I/O 대기 중에는 GIL 을 풀어주므로 I/O-bound 는 멀티스레드로도 잘 겹쳐진다.
  > 더 깊이: [이론 지식 §4.8 어떤 모델을 언제 쓰는가](이론_지식.md)
- **출처**: [CPU-bound — Wikipedia](https://en.wikipedia.org/wiki/CPU-bound) · [Global interpreter lock (GIL) — Python Glossary](https://docs.python.org/3/glossary.html#term-global-interpreter-lock)

### bit · byte · 이진수 (binary) {#bit-byte}

- **정의**:
  - **bit (비트)**: 0 또는 1 — 컴퓨터의 가장 작은 정보 단위. binary digit 의 줄임말.
  - **byte (바이트)**: 거의 모든 현대 시스템에서 8 bit 묶음(8비트 묶음을 명확히 가리킬 땐 *옥텟, octet*). 영어 한 글자(ASCII)가 1 byte 이며, 대부분 아키텍처에서 *메모리 주소를 매기는 최소 단위*다.
  - **이진수 (binary)**: 0/1 만으로 모든 숫자·문자·이미지·동영상을 표현하는 방식. 8 bit 한 바이트는 0~255 (= 2⁸ = 256가지)를 나타낸다.
- **비유**: bit = 전기 스위치 1개(on/off). byte = 스위치 8개 묶음. 컴퓨터는 결국 스위치 수십억 개로 모든 일을 한다.
- **왜 중요한가**: KB·MB·GB 모두 byte 의 배수. `top`, `ps`, `free` 가 보여주는 모든 메모리 수치는 결국 byte 단위다. 4 KB = 4096 byte 라는 사실이 [페이지](#page) 크기와도 연결된다.
  > 더 깊이: [이론 지식 §2.1 프로세스의 가상 메모리 레이아웃](이론_지식.md)

## 1.5 셸 · 터미널 · 로그 (비전공자 보충)

이 섹션은 미션 도중 마주치는 *명령행 환경* 의 기본기를 비전공자 관점에서 한 번 정리한다.

### 셸 (Shell) · bash · zsh {#shell}

- **정의**: 사용자가 키보드로 친 명령어를 받아 OS 에 전달·실행하는 *프로그램*. 명령행을 토큰으로 쪼개고(파싱), [환경변수](#env-var)·와일드카드(`*`, `?`)·따옴표를 *치환·확장* 한 뒤, 내장 명령(`cd`, `export` 등)은 직접 처리하고 외부 명령은 [fork+exec](#fork-exec) 로 새 프로세스를 띄워 돌린다. 리눅스 기본은 `bash`, macOS 는 2019년 Catalina 부터 기본 셸을 `zsh` 로 바꿨다. 흔한 오해: "셸 = 터미널" 이 아니다. 셸은 명령 해석기 *프로그램* 이고, 터미널은 그걸 보여주는 *창* 이다([터미널](#terminal) 참고). 또 다른 오해: 셸은 단순 입력 전달기가 아니라 [파이프](#pipe-redirect)·조건 분기·반복문을 갖춘 *프로그래밍 언어* 다.
- **비유**: 호텔 컨시어지. "5층 가게 해 주세요" 같은 사람의 요청을 받아 엘리베이터·청소·룸서비스 같은 실제 시스템에 전달.
- **왜 중요한가**: 우리가 치는 `ps`, `top`, `curl`, `kill` 모두 셸이 받아서 [fork+exec](#fork-exec) 로 새 [프로세스](#process)를 만들어 돌린다. 셸 자체도 하나의 프로세스([PID](#pid) 있음)라서, 셸이 죽으면 그 아래 자식 프로세스들도 영향을 받는다(이게 [데몬](#daemon) `nohup` 이 필요한 이유와 직결).

> 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [Shell — Wikipedia (Unix shell)](https://en.wikipedia.org/wiki/Unix_shell) · [Use zsh as the default shell on your Mac — Apple Support](https://support.apple.com/en-us/102360)

### 터미널 (Terminal) {#terminal}

- **정의**: 셸을 띄우고 그 입출력을 화면에 그려 주는 *창* 프로그램. macOS Terminal, Windows Terminal, GNOME Terminal 등이 모두 터미널 *에뮬레이터* — 옛날의 물리 단말기를 소프트웨어로 흉내 낸 것이다. 그 원형이 1978년 DEC 가 내놓은 VT100 으로, ANSI 이스케이프 시퀀스(커서 이동·색상 제어용 제어문자열)를 사실상 표준화시켰고 오늘날 거의 모든 에뮬레이터가 "VT100 호환" 을 표방한다. 내부적으로 터미널 에뮬레이터는 셸과 *의사 터미널(pseudo-terminal, pty)* 이라는 커널 장치 쌍으로 연결되어, 사람이 친 키 입력을 셸의 [stdin](#std-streams) 으로, 셸의 [stdout/stderr](#std-streams) 를 화면으로 중계한다.
- **비유**: 셸이 컨시어지라면 터미널은 그 컨시어지가 앉아있는 *데스크*. SSH 로 원격 서버에 접속하는 것은 "다른 데스크에서 같은 컨시어지(셸)와 대화" 하는 것.
- **왜 중요한가**: 터미널은 셸을 *보여주는 도구* 일 뿐 셸 자체는 아니다. 둘을 구분해야 "터미널을 닫으면 왜 [`nohup`](#daemon) 없이는 프로세스도 죽나" 같은 질문에 답할 수 있다 — 터미널이 닫히면 그 pty 가 사라지며 셸과 자식들에게 `SIGHUP`([시그널](#signal))이 전달되기 때문이다.

> 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [VT100 — Wikipedia](https://en.wikipedia.org/wiki/VT100) · [pty(7) — Linux manual page](https://man7.org/linux/man-pages/man7/pty.7.html)

### CLI vs GUI {#cli-gui}

- **정의**:
  - **CLI (Command Line Interface)**: 텍스트 명령어를 한 줄씩 입력해 컴퓨터를 다루는 방식. [셸](#shell)이 대표. 명령이 텍스트라서 그대로 스크립트 파일에 적어 *재현·자동화* 할 수 있는 게 핵심 강점이다.
  - **GUI (Graphical User Interface)**: 아이콘·창·마우스 같은 그래픽 요소로 다루는 방식. 윈도우 탐색기, macOS Finder. 직관적이지만 동작을 그대로 기록·반복하기는 어렵다.
- **비유**: CLI = 점원에게 말로 주문. GUI = 메뉴판 사진 보고 손가락으로 가리키기.
- **왜 중요한가**: 서버 운영·자동화·스크립트는 거의 CLI. 이번 미션의 모든 도구(`ps`, `top`, `vmstat`, `dmesg`, `curl`) 가 CLI 이며, OOM/CPU 장애 분석은 결국 CLI 출력을 [파이프](#pipe-redirect)·`grep` 으로 가공하는 작업이다.

> 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [Command-line interface — Wikipedia](https://en.wikipedia.org/wiki/Command-line_interface) · [Graphical user interface — Wikipedia](https://en.wikipedia.org/wiki/Graphical_user_interface)

### 표준 입출력 (stdin · stdout · stderr) {#std-streams}

- **정의**: 모든 [프로세스](#process)가 시작할 때 자동으로 열려 있는 3개의 데이터 통로. POSIX 가 각각 [파일 디스크립터](#file-descriptor) 0/1/2 번을 차지하도록 규정한다(`<unistd.h>` 의 `STDIN_FILENO`/`STDOUT_FILENO`/`STDERR_FILENO`).
  - **stdin (표준 입력, FD 0)**: 키보드 또는 다른 명령의 출력에서 *들어오는* 데이터.
  - **stdout (표준 출력, FD 1)**: 화면으로 보내는 일반 출력. 터미널을 향할 때는 *줄 단위 버퍼링(line-buffered)*.
  - **stderr (표준 에러, FD 2)**: 화면으로 보내는 에러·경고. *버퍼링하지 않아(unbuffered)* 즉시 출력되며, stdout 과 분리되어 있어 따로 잡을 수 있다.
- **비유**: 식당 카운터에 창구가 3개 — 주문(stdin), 음식(stdout), 컴플레인(stderr).
- **왜 중요한가**: `command > out.log 2> err.log` 처럼 정상 출력과 에러를 분리할 수 있는 건 둘이 다른 FD 라서 가능. monitor.sh 같은 로깅 스크립트가 자주 쓰며, 장애 로그에서 정상 흐름과 에러를 갈라 보는 첫걸음이다. 흔한 함정: 프로그램이 죽기 직전 stdout 버퍼는 비우지 못해도 stderr 의 에러 메시지는 거의 항상 남는다 — 그래서 사후 분석은 stderr 를 먼저 본다.

> 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [stdin(3) — Linux manual page](https://man7.org/linux/man-pages/man3/stdin.3.html) · [stdin — The Open Group Base Specifications](https://pubs.opengroup.org/onlinepubs/9699919799/functions/stdin.html)

### 파이프 · 리다이렉트 (`|`, `>`, `>>`, `<`) {#pipe-redirect}

- **정의**:
  - **파이프 (`|`)**: 왼쪽 명령의 [stdout](#std-streams) 을 오른쪽 명령의 [stdin](#std-streams) 으로 *흘려보낸다*. `ps -ef | grep python` 처럼. 두 명령은 *동시에* 실행되며, 커널의 파이프 버퍼(리눅스 기본 64KiB)가 차면 쓰는 쪽이 잠깐 멈춘다 — 즉 파이프는 [IPC](#ipc) 의 한 형태다.
  - **리다이렉트**: `>` 는 stdout 을 *파일로 덮어쓰기*(파일을 비우고 새로 씀), `>>` 는 *이어쓰기*, `<` 는 파일을 stdin 으로 *읽기*, `2>` 는 stderr 만 따로 보낸다. 흔한 실수: `2>&1` 의 위치 — `cmd > f 2>&1` 은 둘 다 `f` 로 가지만 `cmd 2>&1 > f` 는 stderr 가 *원래 화면* 으로 새어 나간다(평가가 왼쪽→오른쪽 순서라서).
- **비유**: 파이프 = 한 요리사가 만든 반찬을 옆 요리사 도마로 바로 올리는 컨베이어 벨트. 리다이렉트 = 다 만든 음식을 그릇(파일)에 담아 보관.
- **왜 중요한가**: 셸 스크립트의 거의 모든 표현력이 이 두 가지에서 나온다. monitor.log 같은 결과 파일도 결국 리다이렉트의 산물이며, `dmesg | grep -i oom` 같은 한 줄로 장애 단서를 추려내는 게 분석의 일상이다.

> 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [pipe(7) — Linux manual page](https://man7.org/linux/man-pages/man7/pipe.7.html) · [Bash Reference Manual — Redirections (GNU)](https://www.gnu.org/software/bash/manual/html_node/Redirections.html)

### 파일 디스크립터 (File Descriptor, FD) {#file-descriptor}

- **정의**: [프로세스](#process)가 열어 둔 파일·소켓·파이프를 가리키는, OS 가 붙이는 작은 음이 아닌 정수. 0/1/2 는 [stdin/stdout/stderr](#std-streams) 가 차지하고, 새로 연 것은 *현재 사용 가능한 가장 낮은 번호*(보통 3, 4, 5…)를 받는다. 정확히는 FD 는 커널의 *열린 파일 테이블* 항목을 가리키는 프로세스별 인덱스라서, 같은 파일을 두 번 열면 FD 번호가 둘로 다르다. `ls -l /proc/<PID>/fd/` 로 그 프로세스가 지금 무엇을 열고 있는지 다 볼 수 있다.
- **비유**: 도서관에서 빌린 책마다 받는 번호표. 책 자체가 아니라 번호표만 들고 다닌다.
- **왜 중요한가**: 멀티스레드에서 FD 테이블은 프로세스 단위로 *공유* 자원이라 한 [스레드](#thread)가 닫으면 모두에게 영향. "Too many open files"(`EMFILE`) 에러는 한 프로세스가 가질 수 있는 FD 수가 한계(`ulimit -n`, 리눅스 기본 흔히 1024)에 닿았다는 뜻 — FD 누수는 OOM 못지않은 흔한 장애 원인이다.

> 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [open(2) — Linux manual page](https://man7.org/linux/man-pages/man2/open.2.html) · [getrlimit(2) — Linux manual page](https://man7.org/linux/man-pages/man2/getrlimit.2.html)

### exit code (종료 코드) · `$?` {#exit-code}

- **정의**: 명령어가 끝날 때 OS 에 반환하는 0~255 정수. 관례: **0 = 성공, 그 외(1~255) = 실패**. 직전 명령의 종료 코드는 셸 변수 `$?` 에 담긴다. `wait()` 류 호출이 받는 원래 상태값은 더 넓지만, 셸은 하위 8비트만 노출하므로 256 이상은 256 으로 나눈 나머지로 wrap 된다(예: `exit 256` → `$?` 는 0).
- **비유**: 시험지를 내고 받는 점수. 0 점이 *만점*(다 풀었다 = 성공), 다른 숫자는 어디서 막혔는지 코드.
- **왜 중요한가**: 셸의 `if`, `&&`, `||` 가 모두 exit code 로 판단. 프로세스가 [시그널](#signal)로 죽으면 셸은 `128 + 시그널번호` 로 보고한다 — SIGKILL(9) → 137, SIGTERM(15) → 143. 로그에 `Exited with 137` 이 보이면 *누군가(흔히 OOM Killer) SIGKILL 을 보냈다* 의 강력한 단서.

> 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [Bash Reference Manual — Exit Status (GNU)](https://www.gnu.org/software/bash/manual/html_node/Exit-Status.html)

### root · sudo · 권한 (Permission) {#root-sudo}

- **정의**:
  - **root (= superuser, uid 0)**: 시스템 *대부분의 권한* 을 가진 특수 계정. 통상적인 파일 권한 검사를 우회하며, 어떤 프로세스든 [SIGKILL](#signal) 할 수 있다. (엄밀히는 SELinux·capabilities·seccomp 등으로 root 도 제약될 수 있다.)
  - **sudo**: 일반 사용자가 정책 파일(`/etc/sudoers`)이 허용하는 범위에서 *명령을 다른 사용자(기본 root) 권한으로* 실행하는 도구. `sudo apt install ...` 같은 식.
  - **권한**: 파일·디렉토리에 붙은 *r(읽기) · w(쓰기) · x(실행)* 표시. `ls -l` 의 `rwxr-xr-x` 9 칸이 (소유자·그룹·기타) × (r/w/x).
- **비유**: 빌딩 마스터키(root) vs 직원 카드(일반 사용자). 마스터키는 모든 문을 열지만, 직원 카드는 자기 사무실만. 청소부가 잠깐 마스터키를 빌리는 것 = sudo.
- **왜 중요한가**: `dmesg`, 남의 프로세스 [`kill`](#signal), `/proc/<남의 pid>/...` 등 시스템 깊은 곳에 닿으려면 root 가 필요할 때가 많다. 보안 원칙: *기본은 일반 사용자, 필요할 때만 sudo* — 그래야 실수나 침해의 폭발 반경을 줄인다.

> 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [credentials(7) — Linux manual page](https://man7.org/linux/man-pages/man7/credentials.7.html) · [sudo(8) — sudo.ws manual](https://www.sudo.ws/docs/man/sudo.man/)

### 로그 · 로그 레벨 {#log-level}

- **정의**: 프로그램이 동작 중 남기는 텍스트 기록. 파이썬 `logging` 모듈은 5단계 레벨을 숫자값과 함께 정의한다 — 숫자가 클수록 심각하고, 설정한 레벨 *이상* 만 출력된다:
  - **DEBUG (10)**: 개발 중 추적용 (운영에선 보통 꺼둠)
  - **INFO (20)**: 정상 진행 상황 알림
  - **WARNING (30)**: 잘못될 수 있는 상황
  - **ERROR (40)**: 일부 기능 실패
  - **CRITICAL (50)** (= FATAL): 시스템 자체가 위험 — 곧 종료/장애
- **비유**: 일기. 평범한 날(INFO), 좀 이상한 날(WARNING), 사고 난 날(ERROR), 큰 사건(CRITICAL).
- **왜 중요한가**: `MemoryGuard` 가 띄우는 `[CRITICAL]` 은 *가장 높은 위험 레벨(50)* 이므로 장애 1차 단서가 된다. `grep CRITICAL` 만 해도 평소 안 보이던 사건만 솎아낼 수 있다. 반대로 운영에서 레벨을 WARNING 으로 올려두면 DEBUG/INFO 가 묻혀 정작 중요한 로그를 놓치기도 한다.

> 더 깊이: [이론 지식](이론_지식.md)

## 2. 저장 계층 (Storage Hierarchy)

### 레지스터 (Register) {#register}

- **정의**: CPU 칩 *안에* 박혀 있는, 컴퓨터에서 가장 빠른(접근 지연 약 0.3 ns) 저장 공간. 산술·논리 연산의 피연산자와 결과는 결국 전부 레지스터를 거친다. 64비트 x86-64 아키텍처 기준 프로그램이 일반적으로 쓰는 **범용 레지스터(GPR)는 16개**(RAX·RBX·...·RSP 등 8개 + R8~R15 8개)이고, 그 외에 부동소수점·벡터(SSE/AVX)·상태 플래그용 레지스터가 따로 있다. 흔한 오해 하나: "레지스터가 많을수록 무조건 빠르다" — 실제로는 개수보다 컴파일러가 값을 메모리로 내보내지 않고(=레지스터 스필 회피) 얼마나 레지스터 위에 머물게 하느냐가 성능을 좌우한다.
- **비유**: 요리사의 양손. 지금 이 순간 만지고 있는 재료 한두 개만 들 수 있다.
- **왜 중요한가**: CPU 가 진짜로 계산하는 건 레지스터 위에서다. 메모리에서 값을 꺼내 레지스터에 올리고, 계산하고, 다시 메모리에 적는 순환을 끝없이 반복한다. CPU 과점유(spike) 장애를 이해할 때, `top` 의 `%CPU` 가 100%라는 건 결국 이 "꺼내고-계산하고-적는" 사이클이 한 코어를 가득 채우고 있다는 뜻이다.

> 더 깊이: [이론 지식 §1.2 폰 노이만 병목](이론_지식.md)
- **출처**: [Processor register (Wikipedia)](https://en.wikipedia.org/wiki/Processor_register)

### 캐시 · L1 / L2 / L3 (Cache) {#cache}

- **정의**: [레지스터](#register)와 [DRAM](#dram) 사이에 끼워 놓은 작은 고속 메모리. CPU 가 방금 쓴(또는 곧 쓸) 데이터를 가까이 붙잡아 두려는 장치다. **L1 이 가장 작고(코어당 수십 KB, 보통 16~64 KB) 가장 빠르며, L3 가 가장 크고(수십 MB) 가장 느리다.** L1·L2 는 보통 코어 전용이고 L3 는 여러 코어가 공유한다. CPU 는 L1→L2→L3→DRAM 순으로 차례로 찾으며, 찾던 데이터가 캐시에 있으면 "캐시 히트", 없으면 "캐시 미스" 다. 흔한 오해: 캐시는 프로그램이 직접 다루는 게 아니라 하드웨어가 자동으로 채우고 비운다 — 그래서 "지역성(locality)"이 좋은 코드가 빨라진다.
- **비유**: 책상 서랍 3단. 1단 서랍(L1)은 손 뻗으면 닿지만 작고, 3단(L3)은 약간 멀지만 더 많이 들어간다. 책장([DRAM](#dram))까지 가는 건 더 멀다.
- **왜 중요한가**: 같은 데이터를 반복해 쓰면 캐시 덕분에 무지 빠르다. 반대로 메모리 누수로 작업 집합(working set)이 캐시 용량을 넘어 자주 갈리면(= "캐시 미스"가 잦아지면) CPU 가 매번 DRAM 까지 내려가야 하니 같은 일을 해도 수십 배 느려진다.

> 더 깊이: [이론 지식 §1.2 폰 노이만 병목](이론_지식.md)
- **출처**: [Cache hierarchy (Wikipedia)](https://en.wikipedia.org/wiki/Cache_hierarchy)

### DRAM (= RAM) {#dram}

- **정의**: 우리가 흔히 "메모리" 라 부르는 그것. 전원이 꺼지면 내용이 사라지는 **휘발성** 저장소이며, 한 비트를 트랜지스터 1개 + 콘덴서 1개에 저장한다. 콘덴서의 전하는 시간이 지나면 새어 나가기 때문에, DRAM 은 켜져 있는 동안에도 주기적으로 내용을 다시 써 넣는 **리프레시(refresh)** 를 끊임없이 해야 한다. 이름의 "D(Dynamic)" 가 바로 이 동적 리프레시를 가리킨다. 접근 지연은 약 100 ns 로, 캐시보다 한 단계 느린 계층이다.
- **비유**: 책장. 책상보다는 크지만 멀고, 정전되면 책이 다 사라지는 마법의 책장.
- **왜 중요한가**: 모든 실행 중인 프로그램과 데이터가 여기에 있다. `free`, `%MEM` 이 가리키는 게 바로 DRAM 사용량. 가득 차면 OS 가 디스크로 내쫓기 시작한다(= [스왑](#swap-thrashing)). 미션의 OOM(메모리 누수) 장애는 본질적으로 이 DRAM 이 한계에 다다르는 과정이다.

> 더 깊이: [이론 지식 §1.1 모든 것이 메모리에 있다](이론_지식.md)
- **출처**: [Dynamic random-access memory (Wikipedia)](https://en.wikipedia.org/wiki/Dynamic_random-access_memory)

### 디스크 · SSD · HDD {#disk}

- **정의**:
  - **HDD (Hard Disk Drive)**: 회전하는 자기 원판(platter)에 데이터를 자기로 적는 전통적 디스크. 헤드가 물리적으로 이동·탐색하므로 느리다(접근 지연 약 10 ms 수준).
  - **SSD (Solid State Drive)**: NAND 플래시 메모리 기반. **움직이는 부품이 없어** HDD 보다 약 100배 빠르다(랜덤 접근 지연 약 100 µs 수준).
  - 둘 다 전원이 꺼져도 내용이 남는 **비휘발성** 저장소다. 단, SSD 의 NAND 셀은 쓰기 횟수에 수명 한계가 있어 무한히 덮어쓸 수 없다는 점이 DRAM 과 다르다.
- **비유**: 도서관. 책장에 없는 책을 찾으러 가는 곳. SSD 는 가까운 도서관, HDD 는 다른 도시 도서관.
- **왜 중요한가**: [DRAM](#dram) 이 다 차면 OS 가 페이지를 SSD/HDD 의 스왑 영역으로 옮긴다. 옮긴 페이지가 다시 필요하면 SSD/HDD 에서 읽어와야 하니 갑자기 100~10000배 느려진다 (= [쓰래싱](#swap-thrashing)).

> 더 깊이: [이론 지식 §1.2 폰 노이만 병목](이론_지식.md)
- **출처**: [Solid-state drive (Wikipedia)](https://en.wikipedia.org/wiki/Solid-state_drive)

### 페이지 교체 · 스왑 · 쓰래싱 {#swap-thrashing}

- **정의**:
  - **페이지 교체**: [DRAM](#dram) 이 모자라면 OS 가 자주 안 쓰는 [페이지](#page)(보통 4KB 덩어리)를 골라 디스크로 옮기는(swap-out) 행위. 옮긴 페이지가 다시 필요하면 디스크에서 도로 읽어 온다(swap-in) — 이때 [페이지 폴트](#page-fault)가 발생한다.
  - **스왑 (Swap)**: 그렇게 디스크로 옮겨두는 전용 영역. 리눅스에서는 swap 파일/파티션으로 만든다.
  - **쓰래싱 (Thrashing)**: 실제 메모리(RAM)가 과도하게 초과 사용(over-commit)되어, 프로그램의 작업 집합이 물리 메모리에 다 들어가지 못하는 상태. 페이지를 디스크로 옮기자마자 다시 필요해져서 "DRAM ↔ 디스크" 왕복(page fault)만 끝없이 반복하고, CPU 시간 대부분이 진짜 일이 아니라 페이지 폴트 처리에 소모돼 시스템 처리량이 붕괴한다. 임계점을 넘는 순간 페이지 폴트가 급격히 폭증한다는 점이 특징이다.
- **비유**: 책상이 좁아서 안 쓰는 책을 도서관에 반납했는데, 반납하자마자 그 책이 또 필요해져서 다시 빌리러 가는 사람. 결국 책상에서 공부는 못 하고 도서관만 왔다갔다 한다.
- **왜 중요한가**: 쓰래싱이 시작되면 *겉으로는* "메모리는 아직 100%는 아닌데 CPU 도 멀쩡한데 시스템이 멈췄다" 처럼 보인다. monitor.log 의 MEM 이 95% 근처에서 CPU 도 같이 치솟는다면 쓰래싱 의심. 즉 OOM 분석과 CPU 분석이 한 화면에서 겹쳐 보이는 대표적 함정이다.

> 더 깊이: [이론 지식 §2.2 가상 메모리](이론_지식.md)

## 3. 프로세스와 메모리 구조

### 프로세스 (Process) {#process}

- **정의**: 실행 중인 프로그램 한 인스턴스. 디스크에 있는 실행 파일을 OS 가 메모리에 올려서 돌리면, 그 살아 있는 단위 하나가 프로세스. 정확히는 *코드(텍스트) + 데이터 + [가상 주소 공간](#virtual-address-space) + 커널이 관리하는 상태([PCB](#pcb): 레지스터·열린 파일·스케줄링 우선순위 등)* 를 합친 실행 단위다. 흔한 오해 하나: "프로세스 = 프로그램" 이 아니다. 프로그램은 디스크 위의 *수동적인* 파일(데이터)이고, 프로세스는 그것이 메모리에서 *능동적으로 실행되는* 인스턴스다 — 같은 프로그램을 두 번 실행하면 PID 가 다른 프로세스 2개가 된다.
- **비유**: 책장에 꽂힌 요리책(실행 파일) vs 그 레시피로 지금 요리하고 있는 요리사 한 명(프로세스). 같은 책으로 두 요리사가 동시에 요리하면 프로세스 2개.
- **왜 중요한가**: `ps`, `top`, `kill <PID>` 등 우리가 다루는 모든 단위가 프로세스다. 각 프로세스는 자기만의 [가상 주소 공간](#virtual-address-space)을 가지며, OOM/CPU/데드락 미션의 모든 진단이 결국 "어느 프로세스인가"를 특정하는 데서 시작한다.
- > 더 깊이: [이론 지식 §3.1 프로세스 vs 스레드](이론_지식.md)
- **출처**: [Process (computing) — Wikipedia](https://en.wikipedia.org/wiki/Process_%28computing%29)

### PID (Process ID) {#pid}

- **정의**: 실행 중인 모든 [프로세스](#process)에 OS 가 붙이는 고유 번호. 1 부터 시작하며(부팅 직후 init/systemd 가 PID 1), 종료된 프로세스가 회수되면 그 번호는 재사용될 수 있다. 동시에 살아있는 PID 의 최댓값은 `/proc/sys/kernel/pid_max` 로 정해지며, 기본값은 보통 32768(2^15) 이고 64비트 리눅스에서는 최대 4194304(2^22) 까지 상향할 수 있다 — 이 한도가 차면 새 프로세스를 만들지 못한다.
- **비유**: 식당 손님의 대기번호. 같은 번호는 같은 시각에 두 명 못 받는다.
- **왜 중요한가**: 데드락 미션에서 "PID 가 살아있다(`ps -ef | grep`)" 가 1차 증거다. 프로세스가 죽었으면 PID 도 사라진다.
- **출처**: [proc(5) — Linux manual page](https://man7.org/linux/man-pages/man5/proc.5.html)

### PPID · 부모-자식 프로세스 {#parent-child}

- **정의**:
  - **PPID (Parent PID)**: 어떤 [프로세스](#process)를 *만든* 부모 프로세스의 [PID](#pid). `ps -o pid,ppid,comm` 으로 확인.
  - 모든 프로세스는 [fork](#fork-exec) 로 만들어지므로 *반드시* 부모가 있다. 시스템 부팅 직후 만들어지는 `init` 또는 `systemd`(PID 1) 가 모든 프로세스의 시조이며, PID 1 만은 부모가 없다(PPID 0).
  - 자식이 끝나면 부모가 `wait()` (또는 `waitpid()`) 시스템 콜로 회수해야 자식의 [PCB](#pcb) 가 깨끗이 정리된다. 부모가 회수하지 않으면 자식은 [좀비](#zombie-orphan)로 남는다.
- **비유**: 가족 관계도 — 모든 사람에게 부모가 있고, 거슬러 올라가면 한 시조에 닿는다.
- **왜 중요한가**: 부모-자식 관계는 [좀비/고아](#zombie-orphan) 를 이해하는 토대이며, 셸이 `nohup` 으로 자식을 *부모와 분리* 해 살려두는 것도 이 관계를 끊는 행위다.
- **출처**: [wait(2) — Linux manual page](https://man7.org/linux/man-pages/man2/wait.2.html)

### 좀비 · 고아 프로세스 {#zombie-orphan}

- **정의**:
  - **좀비 (Zombie)**: 자식이 종료됐는데 [부모](#parent-child)가 `wait()` 으로 회수하지 않아 [PCB](#pcb) 만 남은 프로세스. [`STAT=Z`](#stat-codes)(`defunct` 로도 표시). CPU·메모리는 거의 0 이지만 종료 코드를 부모가 읽을 때까지 [PID](#pid) 슬롯과 프로세스 테이블 항목을 차지한다. 좀비는 *이미 죽은* 상태라 `kill -9` 로도 못 죽인다 — 부모가 회수하거나, 부모가 죽어 PID 1 이 회수해야 사라진다.
  - **고아 (Orphan)**: 부모가 *먼저* 죽어서 [PPID](#parent-child)가 1번(init/systemd)으로 재지정(reparenting)된 자식. 사고는 아니며 정상 동작. PID 1 이 주기적으로 `wait()` 을 호출해 이들을 회수하므로 고아는 좀비로 남지 않는다.
- **비유**: 좀비 = 출석부에는 있는데 졸업 처리가 안 된 학생(자리 차지). 고아 = 담임이 바뀐 학생(여전히 학생).
- **왜 중요한가**: 좀비가 쌓이면 시스템 PID 가 고갈된다. ⚠ 단, 이번 데드락 미션의 *"살아있는 좀비"* 표현은 진짜 좀비(STAT=Z)가 아니라 *비유적* 표현 — 정상 작동이 멈춘 채 PID 만 유지되는 데드락 프로세스를 말한다.
- **출처**: [Zombie process — Wikipedia](https://en.wikipedia.org/wiki/Zombie_process) · [init — Wikipedia](https://en.wikipedia.org/wiki/Init)

### 데몬 (Daemon) · 서비스 {#daemon}

- **정의**: 부팅 시 자동으로 떠서 *백그라운드로 계속 돌면서* 어떤 서비스를 제공하는 [프로세스](#process). 이름 끝에 `d` 가 붙는 관례(`sshd`, `httpd`, `crond`, `systemd`). 전통적으로 제어 터미널과 분리(setsid 로 새 세션 리더가 됨)되어 부모가 init 인 [고아](#zombie-orphan) 가 되도록 만들어진다. 현대 시스템에서는 systemd 같은 서비스 매니저가 직접 데몬을 자식으로 띄워 관리하는 방식이 일반적이다.
- **비유**: 24시간 편의점 점원. 손님이 오든 안 오든 자리를 지키고 있다가 누가 들어오면 응대.
- **왜 중요한가**: 리눅스의 거의 모든 시스템 서비스(웹서버·DB·SSH·시간동기화)가 데몬으로 실행된다. `agent-leak-app` 도 일종의 데몬처럼 종일 동작하다 정책 위반 시 자체 종료. `nohup ... &` 로 일반 명령을 데몬화할 수 있다.
- **출처**: [daemon(7) — Linux manual page](https://man7.org/linux/man-pages/man7/daemon.7.html)

### 가상 주소 공간 (Virtual Address Space) {#virtual-address-space}

- **정의**: [프로세스](#process)마다 OS 가 만들어주는 "자기 전용 메모리 환상". 실제 [DRAM](#dram) 의 어디에 있든, 프로세스는 0번지부터 거대한 연속 공간을 자기가 다 쓰는 것처럼 느낀다. 64비트 x86-64 리눅스에서 사용자 공간 가상 주소는 보통 하위 47비트(128 TB)까지 쓰며, [MMU](#mmu)와 [페이지 테이블](#page-table-tlb)이 이 가상 주소를 물리 주소로 변환한다. 주의: VSZ 가 크다고 그만큼 실제 메모리를 쓰는 게 아니다 — [lazy allocation](#lazy-allocation) 때문에 대부분은 아직 물리 페이지가 배정되지 않은 "약속"일 뿐이다.
- **비유**: 호텔 객실. 모든 객실은 "방번호 1번" 부터 시작하는 듯한 도면을 갖지만, 실제로는 빌딩의 다른 층, 다른 위치에 있다. 옆방의 도면을 봐도 자기 방 안은 못 본다(메모리 격리).
- **왜 중요한가**: 가상 주소 공간 덕분에 (1) 프로세스 간 격리 (한 앱 버그가 다른 앱 메모리를 못 망침), (2) DRAM 보다 큰 메모리도 마치 있는 것처럼 쓸 수 있음(스왑 활용) 이 가능하다.
- > 더 깊이: [이론 지식 §2.2 가상 메모리](이론_지식.md)
- **출처**: [Virtual memory — Wikipedia](https://en.wikipedia.org/wiki/Virtual_memory)

### Heap · Stack · Data · BSS · Text {#memory-segments}

- **정의 (한 줄씩)**:
  - **Text (Code)**: 프로그램의 기계어 명령. 읽기 전용이며 여러 프로세스가 같은 실행 파일을 돌릴 때 공유될 수 있다.
  - **Data**: `int x = 5;` 처럼 초기값 있는 전역/정적 변수. 실행 파일에 그 값이 그대로 저장된다.
  - **BSS**: `int y;` 처럼 초기값 없는(또는 0 인) 전역/정적 변수. 실행 파일엔 *크기만* 기록되고 로드 시 0 으로 자동 초기화 — 그래서 BSS 는 디스크 용량을 거의 안 먹는다.
  - **Heap (힙)**: 프로그램이 실행 중에 동적으로 할당받는 영역. `malloc()`, `new`, Python 객체 생성 등. 낮은 주소에서 높은 주소 방향으로 자란다(`brk`/`mmap`).
  - **Stack (스택)**: 함수가 호출될 때마다 새 칸(스택 프레임)이 쌓이는 영역. 지역 변수·매개변수·반환 주소가 들어감. 보통 높은 주소에서 낮은 주소 방향으로 자라며, 무한 재귀 등으로 한계를 넘으면 스택 오버플로(SIGSEGV)가 난다.
- **비유**: 식당 주방의 구역들. Text = 메뉴판(못 바꿈), Data/BSS = 주방장이 미리 만들어둔 소스 통, Heap = 손님 주문대로 그때그때 만드는 음식 보관함(계속 변함), Stack = 지금 만드는 한 요리에 쓰는 임시 도마(만들고 치움).
- **왜 중요한가**: 메모리 누수는 *거의 항상* Heap 에서 일어난다. Stack 은 함수 끝나면 자동 정리되니까. 그래서 누수 분석은 Heap 사용량(= [RSS](#vsz-vs-rss))추이만 봐도 충분.
- > 더 깊이: [이론 지식 §2.1 프로세스의 가상 메모리 레이아웃](이론_지식.md)
- **출처**: [Data segment — Wikipedia](https://en.wikipedia.org/wiki/Data_segment)

### malloc / free / 가비지 컬렉터 (GC) {#malloc-free-gc}

- **정의**:
  - `malloc()`: C 언어에서 [Heap](#memory-segments)에 메모리 달라는 함수. ("malloc" = memory allocate) 작은 요청은 `brk` 로 힙을 늘려 처리하고, 큰 요청(보통 128 KB 이상)은 `mmap` 으로 별도 영역을 받아온다.
  - `free()`: 다 쓴 메모리를 반납하는 함수. C 에서는 *손으로* 호출해야 함. free 를 안 하면 누수, 두 번 하면 double-free 버그.
  - **GC (Garbage Collector)**: Python/Java 등 일부 언어가 자동으로 "이 객체 더 이상 쓸 사람 없네" 라고 판단해 메모리를 회수해주는 시스템. CPython 은 *참조 카운팅* 을 기본으로 쓰고, 서로를 참조하는 순환 참조를 따로 처리하는 보조 순환 수집기를 둔다.
- **비유**: 도서관에서 책을 빌리는 것(malloc) ↔ 반납하는 것(free). GC 는 "한 달 동안 안 본 책은 자동 반납됨" 같은 자동화 시스템.
- **왜 중요한가**: GC 가 있어도 *어딘가에서 그 객체를 참조하고 있으면* GC 는 못 치운다. 전역 리스트에 자꾸 append 하면서 안 비우면 GC 도 "이거 누가 쓰고 있는 거 같은데?" 하고 그냥 둔다 → 메모리 누수.
- **출처**: [malloc(3) — Linux manual page](https://man7.org/linux/man-pages/man3/malloc.3.html) · [gc — Garbage Collector interface — Python docs](https://docs.python.org/3/library/gc.html)

### ELF (Executable and Linkable Format) {#elf}

- **정의**: 리눅스를 비롯한 대부분의 유닉스 계열 OS 가 쓰는 표준 실행 파일·공유 라이브러리·오브젝트 파일 형식. 윈도우의 `.exe`(PE 형식) 에 해당. 파일 맨 앞 4바이트가 매직 넘버 `0x7F 'E' 'L' 'F'` 라 `file` 명령이 이걸 보고 ELF 임을 판별한다.
- **비유**: 실행 파일이 담긴 도시락 박스. 표지에 "이 도시락 안에 코드는 여기, 데이터는 여기" 식의 색인이 있어 OS 가 그걸 보고 메모리에 펼친다.
- **왜 중요한가**: `agent-leak-app` 도 ELF 다 (`file agent-leak-app` 으로 확인 가능). 부팅 시 로더가 ELF 의 [Text/Data 영역](#memory-segments)을 [가상 메모리](#virtual-address-space)에 펼친다.
- **출처**: [elf(5) — Linux manual page](https://man7.org/linux/man-pages/man5/elf.5.html)

### 페이지 · 4 KB {#page}

- **정의**: 가상 메모리 ↔ 물리 메모리 매핑을 관리하는 최소 단위. 거의 모든 현대 OS·x86-64 에서 4096 바이트(4 KB). 실행 중인 시스템의 실제 값은 `getconf PAGE_SIZE` 또는 `sysconf(_SC_PAGESIZE)` 로 확인한다. 단, 큰 메모리를 효율적으로 쓰려고 2 MB·1 GB 짜리 *huge page* 를 쓸 수도 있다(기본은 4 KB).
- **비유**: 책의 한 페이지. 책을 한 글자 단위로 도서관에서 빌리고 반납하면 비효율적이니, "페이지 단위로 빌린다" 고 정한 셈.
- **왜 중요한가**: OS 는 메모리를 페이지 단위로 할당·회수·스왑한다. 1 바이트만 써도 4 KB 가 할당된다(= 내부 단편화). 메모리 누수도 결국 페이지 단위 누적.
- **출처**: [getpagesize(2) — Linux manual page](https://man7.org/linux/man-pages/man2/getpagesize.2.html)

### MMU (Memory Management Unit) {#mmu}

- **정의**: CPU 칩 안에 있는 하드웨어로, [프로세스](#process)가 보는 가상 주소를 실제 [DRAM](#dram) 의 물리 주소로 *매번* 자동 변환해주는 장치. 변환 과정에서 [페이지 테이블](#page-table-tlb)을 참조하고, 접근 권한(읽기/쓰기/실행) 위반이나 매핑 부재를 발견하면 [페이지 폴트](#page-fault)를 일으켜 커널에 제어를 넘긴다.
- **비유**: 호텔 객실 번호(가상) → 실제 빌딩 층/방 번호(물리) 를 즉시 알려주는 안내원.
- **왜 중요한가**: 이게 없다면 [가상 메모리](#virtual-address-space)도 없고, 프로세스 격리도 없고, 스왑도 없다. 모든 메모리 시스템의 토대.
- **출처**: [Memory management unit — Wikipedia](https://en.wikipedia.org/wiki/Memory_management_unit)

### 페이지 테이블 · TLB {#page-table-tlb}

- **정의**:
  - **페이지 테이블**: "가상 페이지 N 번은 물리 페이지 M 번에 있다" 같은 매핑 정보가 적힌 표. 메모리에 있으며, 64비트 주소 공간이 워낙 커서 보통 4단계(또는 5단계) 계층 구조로 구성된다.
  - **TLB (Translation Lookaside Buffer)**: 페이지 테이블 검색(여러 단계 메모리 접근)이 너무 자주 일어나니, [MMU](#mmu) 가 최근 매핑 결과를 작은 캐시에 저장. 캐시 히트면 한 번에 변환, 미스면 페이지 테이블을 다시 걷는다(page table walk).
- **비유**: 페이지 테이블 = 호텔 객실 도면(분량이 크다). TLB = 안내원이 머리에 외워둔 최근 객실 위치 메모.
- **왜 중요한가**: 메모리를 새로 많이 할당하면 TLB 가 자주 갈리고(= miss), 변환만으로도 느려진다. 메모리 누수가 성능 저하를 가속하는 또 하나의 이유. [컨텍스트 스위치](#context-switch) 시 TLB 가 통째로 비워지므로 [멀티프로세싱](#py-multiprocessing) 이 [멀티스레딩](#pthread) 보다 비싼 이유이기도 하다.
- **출처**: [Page table — Wikipedia](https://en.wikipedia.org/wiki/Page_table) · [Translation lookaside buffer — Wikipedia](https://en.wikipedia.org/wiki/Translation_lookaside_buffer)

### 페이지 폴트 (Page Fault) {#page-fault}

- **정의**: [프로세스](#process)가 어떤 가상 주소를 건드렸는데, 해당 [페이지](#page)가 아직 [DRAM](#dram) 에 없거나 매핑이 안 돼 있을 때 발생하는 이벤트. CPU 가 커널에게 "이 페이지 좀 채워줘" 라고 호출. 디스크 I/O 가 필요 없으면 **마이너 폴트**(수 마이크로초), 스왑·디스크에서 읽어와야 하면 **메이저 폴트**(수 밀리초 — 약 1000배 느림)로 나뉜다. `ps -o min_flt,maj_flt` 로 카운트를 볼 수 있다.
- **비유**: 호텔 객실 도면에 있는 방인데 실제로는 아직 가구가 없는 상태로 가니, 안내원이 즉시 가구를 옮겨와 채워주는 행위.
- **왜 중요한가**: 페이지 폴트는 *정상* 동작이다. 처음 할당받은 페이지에 처음 쓰는 순간 항상 발생. 하지만 메이저 폴트가 너무 자주 발생하면(예: 스왑된 페이지를 다시 가져오는 [스래싱](#swap-thrashing)) 시스템이 급격히 느려진다. [Copy-on-Write](#cow) 도 페이지 폴트 메커니즘 위에 얹혀 있다.
- **출처**: [Page fault — Wikipedia](https://en.wikipedia.org/wiki/Page_fault)

### lazy allocation (지연 할당) {#lazy-allocation}

- **정의**: [프로세스](#process)가 `malloc(1GB)` 라고 해도 OS 는 진짜 1GB 물리 메모리를 안 준다. *프로세스가 그 페이지를 실제로 읽거나 쓰는 순간*([페이지 폴트](#page-fault)) 에만 물리 페이지를 배정한다(demand paging). 그래서 할당 자체는 거의 즉시 성공하지만 실제 물리 메모리는 천천히 채워진다 — 이 때문에 리눅스는 실제 RAM 보다 더 많은 메모리를 약속하는 *오버커밋(overcommit)* 이 기본이다.
- **비유**: 식당에서 "10인분 예약" 했다고 음식을 미리 다 만들어두지 않는다. 손님이 와서 "이제 주세요" 할 때마다 한 사람분씩 만든다.
- **왜 중요한가**: [VSZ vs RSS](#vsz-vs-rss) 의 차이를 만드는 핵심 메커니즘. 누수 추적 시 *RSS* 만 봐야 하는 이유. 오버커밋이 한계를 넘으면 [OOM Killer](#oom-killer) 가 발동한다.
- **출처**: [proc(5) overcommit_memory — Linux manual page](https://man7.org/linux/man-pages/man5/proc.5.html)

### VSZ vs RSS {#vsz-vs-rss}

- **정의**:
  - **VSZ (Virtual Size)**: [프로세스](#process)가 OS 에 "이만큼 줘" 라고 *요청한* 가상 메모리 총량 (KB 단위). [lazy allocation](#lazy-allocation) 때문에 실제 사용량은 더 작을 수 있다. 매핑된 공유 라이브러리·예약 영역까지 다 포함하므로 보통 RSS 보다 훨씬 크다.
  - **RSS (Resident Set Size)**: 그 중에서 *지금 [DRAM](#dram) 에 진짜 올라와 있는* 양. [스왑](#swap-thrashing)으로 나간 페이지는 빠진다. 단, 여러 프로세스가 공유하는 페이지(공유 라이브러리 등)도 각 프로세스의 RSS 에 중복 계산되므로, RSS 들을 단순 합산하면 실제 물리 메모리 사용량보다 커진다는 함정이 있다.
- **비유**: 호텔 예약(VSZ) vs 지금 실제 묵고 있는 손님 수(RSS). 1000명 예약해도 80명만 와 있으면 RSS 는 80명분.
- **왜 중요한가**: 메모리 누수의 가장 직접적 지표는 RSS 다. VSZ 는 늘었다 줄었다 안 하므로 누수 감지에 둔감하다.
- > 더 깊이: [이론 지식 §2.3 RSS vs VSZ](이론_지식.md)
- **출처**: [ps(1) — Linux manual page](https://man7.org/linux/man-pages/man1/ps.1.html) · [Resident set size — Wikipedia](https://en.wikipedia.org/wiki/Resident_set_size)

### fork / exec {#fork-exec}

- **정의**:
  - **`fork()`**: 현재 [프로세스](#process) 를 *자기 복제* 하는 시스템 콜. 호출 한 번에 거의 똑같은 프로세스 두 개(부모·자식)가 생긴다. 메모리 상태·열린 파일·환경변수 모두 같지만 [PID](#pid) 만 다르다. fork 는 한 번 호출되어 부모에게는 자식의 PID 를, 자식에게는 0 을 반환한다("한 번 호출, 두 번 반환"). 메모리 복제는 [Copy-on-Write](#cow) 로 지연된다.
  - **`exec()`**: 현재 프로세스의 메모리 이미지를 *다른 프로그램으로 교체* 하는 시스템 콜(`execve` 계열). PID 는 유지되지만 코드·데이터·힙·스택이 완전히 새 프로그램으로 바뀐다. 성공하면 원래 코드로 돌아오지 않는다.
- **비유**: fork = 나 자신을 복사해서 쌍둥이를 만든다. exec = 그 쌍둥이가 옷·기억·이름표를 한꺼번에 새것으로 바꾼다(PID 라는 사번만 유지).
- **왜 중요한가**: 셸이 명령어를 실행할 때마다 `fork → exec` 패턴을 쓴다. `bash` 가 `ls` 를 돌리려면 fork 로 자식을 만들고, 자식이 exec 로 ls 가 된다. 멀티프로세싱 구현의 가장 밑바닥.
- > 더 깊이: [이론 지식 §4.2 멀티프로세싱](이론_지식.md)
- **출처**: [fork(2) — Linux manual page](https://man7.org/linux/man-pages/man2/fork.2.html) · [execve(2) — Linux manual page](https://man7.org/linux/man-pages/man2/execve.2.html)

### Copy-on-Write (COW) {#cow}

- **정의**: [fork](#fork-exec) 후에 부모와 자식이 같은 물리 페이지를 *공유* 하다가, 어느 한쪽이 페이지에 *쓰려는 순간* 비로소 페이지를 복사해 둘로 분리하는 최적화. 구현은 공유 페이지를 읽기 전용으로 표시해 두고, 쓰기 시도가 일어나면 [페이지 폴트](#page-fault)(보호 위반)를 유발해 커널이 그 페이지만 복사하는 식이다. 읽기만 하면 영원히 공유.
- **비유**: 같은 책을 둘이 같이 보다가, 한 명이 한 페이지에 메모를 적으려는 순간 그 페이지만 복사해 가져가는 방식.
- **왜 중요한가**: 1 GB 짜리 프로세스를 fork 해도 실제 메모리는 거의 안 늘어난다. 그래서 멀티프로세싱이 *생각보다* 메모리 효율적이다. 단, 시간이 흐르며 페이지가 분리되면 결국 메모리는 늘어난다.
- **출처**: [Copy-on-write — Wikipedia](https://en.wikipedia.org/wiki/Copy-on-write)

### clone() 시스템 콜 {#clone-syscall}

- **정의**: 리눅스에서 [`fork`](#fork-exec) 도 [`pthread_create`](#pthread) 도 결국 모두 같은 `clone()` 시스템 콜의 다른 옵션이다. `CLONE_VM`·`CLONE_FS`·`CLONE_FILES`·`CLONE_SIGHAND`·`CLONE_THREAD` 등 플래그로 "메모리도 공유? 파일도 공유? 시그널 핸들러도 공유?" 를 결정한다. `CLONE_VM` 이 켜지면 두 실행 흐름이 *같은 메모리 공간* 을 본다 — 이게 곧 스레드다.
- **비유**: 한 종류의 "분신술" 시스템에서 분신과 본체가 무엇을 공유할지 옵션으로 고를 수 있는 것. fork = 아무것도 공유 안 함. thread = 거의 다 공유.
- **왜 중요한가**: 리눅스에서 프로세스와 스레드의 경계가 *유연* 하다는 의미. 그래서 `ps -L` 이 스레드 별 정보를 보여줄 수 있다. 커널 입장에서는 같은 [`task_struct`](#pcb).
- > 더 깊이: [이론 지식 §4.3 멀티스레딩](이론_지식.md)

## 4. 종료 · 보호 정책 · 환경변수

### OOM Killer · `oom_score` {#oom-killer}

- **정의**:
  - **OOM Killer**: 물리 메모리(DRAM)와 스왑까지 모두 차서 커널이 더는 페이지를 할당해 줄 수 없을 때 발동하는 비상 장치. 한 프로세스를 골라 [SIGKILL](#signal)(9) 로 강제 종료해 메모리를 회수한다. 메모리가 *조금 부족한* 정도가 아니라 **할당 자체가 막힌** 막다른 상황에서만 작동한다는 점이 흔한 오해 포인트다.
  - **`oom_score` / badness**: 커널이 각 프로세스에 매기는 "이 놈을 죽이면 메모리가 많이 회수될까" 점수. 대체로 [RSS](#vsz-vs-rss)(실제 점유 물리 메모리)에 비례해 커진다. `/proc/<pid>/oom_score` 에서 현재 점수를, `/proc/<pid>/oom_score_adj` 에서 사람이 직접 매기는 가중치(보정값)를 볼 수 있다. `oom_score_adj` 는 **-1000 ~ +1000** 범위이며, **-1000 으로 두면 그 프로세스는 사실상 OOM 대상에서 제외**(절대 안 죽임)된다.
- **비유**: 응급 상황에서 함선 무게를 줄여야 할 때, 가장 무거운 짐 한 개를 골라 바다에 던지는 함장. `oom_score_adj = -1000` 은 "이 짐만큼은 절대 못 버린다" 라고 못 박아 두는 것.
- **왜 중요한가**: OOM Killer 가 발동하면 *우리가 죽이고 싶지 않은 중요한 프로세스* 까지 끌려갈 수 있다. 그래서 앱 내부에 MemoryGuard 같은 자체 보호 정책을 둬서 커널 OOM Killer 가 발동하기 *전에* 스스로 죽는다. 이번 미션의 핵심은 "같은 OOM 이라도 **커널이 죽였나(SIGKILL, `dmesg` 에 흔적) 앱이 스스로 죽었나(SELF-TERMINATED 로그)** 를 구분" 하는 것이다.
  > 더 깊이: [이론 지식 §2.4 OOM — 두 가지 트리거](이론_지식.md)
- **출처**: [proc_pid_oom_score(5)](https://man7.org/linux/man-pages/man5/proc_pid_oom_score.5.html) · [proc_pid_oom_score_adj(5)](https://man7.org/linux/man-pages/man5/proc_pid_oom_score_adj.5.html)

### 시그널 · SIGKILL · SIGTERM {#signal}

- **정의**: [프로세스](#process)에게 커널이 전달하는 짧은 비동기 알림. 종류마다 번호와 기본 동작이 정해져 있다.
  - **SIGTERM (15)**: "끝낼 준비해" — 프로세스가 **핸들러로 가로채(catch)** 정상 종료 절차(파일 닫기, 락 풀기 등)를 수행할 수 있다. 핸들러가 없으면 기본 동작은 종료. `kill <pid>` 의 기본 시그널.
  - **SIGKILL (9)**: "지금 즉시 죽여" — **가로채거나 무시·차단할 수 없는** 단 둘뿐인 시그널 중 하나(다른 하나는 SIGSTOP). 프로세스는 정리 코드를 돌릴 틈도 없이 커널에 의해 즉사한다.
  - **SIGINT (2)**: 터미널에서 Ctrl+C 가 보내는 시그널. 기본 동작은 종료이며 SIGTERM 처럼 가로챌 수 있다.
- **비유**: SIGTERM = "퇴근 준비하세요" / SIGKILL = "당장 끌어내, 변명 금지" / SIGINT = "Ctrl+C 누른 사용자의 인내심 끝".
- **왜 중요한가**: MemoryGuard 는 보통 SIGKILL 을 쓰고([Watchdog](#watchdog) 은 SIGTERM 을 쓴다), 이 차이가 곧 정책의 정체를 알려준다. 메모리가 한계에 도달한 상황에선 cleanup 조차 추가 메모리를 요구할 수 있어 SIGKILL(즉사)이 안전하고, CPU 만 문제인 경우엔 SIGTERM(정상 정리)이 적합하다. 로그에서 종료 시그널이 무엇이었는지를 보면 어느 정책이 발동했는지 역추적할 수 있다.
  > 더 깊이: [이론 지식 §3.4 Watchdog — SIGTERM vs SIGKILL](이론_지식.md)
- **출처**: [signal(7)](https://man7.org/linux/man-pages/man7/signal.7.html)

### 환경변수 (Environment Variable) {#env-var}

- **정의**: 프로세스가 자식에게 물려주는 `KEY=VALUE` 형태의 문자열 묶음. 셸에서 `export NAME=value` 로 설정하면, 이후 실행되는 자식은 [`exec`](#fork-exec) 계열 호출의 `envp` 인자를 통해 **현재 환경의 복사본** 을 물려받는다. 프로그램은 C 의 `getenv("NAME")`(파이썬은 `os.environ`) 으로 읽는다. 핵심 오해 정정: 환경변수는 **상속(복사)** 이라 *자식이 바꿔도 부모에게는 반영되지 않고*, 부모가 나중에 바꿔도 이미 떠난 자식에게는 영향이 없다. 현재 실행 중 프로세스의 환경은 `/proc/<pid>/environ` 에서 들여다볼 수 있다.
- **비유**: 신입사원이 출근 첫날 받아 가는 가이드 사본. "사무실 전화번호는 X, 이메일은 Y" 를 미리 알려주지만, 사원이 자기 사본에 낙서해도 회사 원본은 그대로다.
- **왜 중요한가**: `agent-leak-app` 은 `MEMORY_LIMIT`, `CPU_MAX_OCCUPY`, `MULTI_THREAD_ENABLE` 같은 환경변수를 부팅 시 읽어 이후 행동의 임계치·동작 모드로 쓴다. 환경변수만 바꾸면 코드 재컴파일 없이 정책이 바뀌므로, 어떤 값으로 떠 있었는지가 곧 장애 재현 조건이다.
  > 더 깊이: [이론 지식 §2.5 환경변수 MEMORY_LIMIT 의 의미](이론_지식.md)
- **출처**: [environ(7)](https://man7.org/linux/man-pages/man7/environ.7.html)

### Watchdog (감시견) {#watchdog}

- **정의**: 어떤 자원의 사용량을 주기적으로 감시하다가, 임계치를 넘으면 미리 정의된 보호 조치(주로 종료)를 발동하는 패턴. 하드웨어 워치독(타이머를 안 건드리면 보드를 리셋), OS 워치독(리눅스 softlockup/hardlockup 감지기 — 한 태스크가 커널 모드에서 기본 ~20초간 빠져나오지 못하면 경고), 앱 워치독 등 여러 층위에 존재한다. 이번 미션의 `Watchdog` 은 **앱 내부 정책** 으로, 자기 CPU% 가 `CPU_MAX_OCCUPY` 를 넘으면 자신에게 [SIGTERM](#signal) 을 보내 종료한다.
- **비유**: 집의 누전 차단기. 전기를 너무 많이 쓰면 자동으로 차단해 화재를 막는다. 단 이번 미션의 워치독은 "옆집 분전반(커널)" 이 아니라 "내 방에 직접 단 차단기(앱)" 다.
- **왜 중요한가**: "내가 폭주하면 알아서 자살" 패턴은 시스템 전체를 지키는 방어선이자 이번 미션의 핵심 정책 중 하나다. 로그에 `[Watchdog] CPU threshold exceeded … SIGTERM` 이 보이면, 그것은 커널 softlockup 감지기가 아니라 **앱이 스스로 발동한 정책** 임을 구분할 수 있어야 한다.
  > 더 깊이: [이론 지식 §3.4 Watchdog — CPU 과점유 보호 정책](이론_지식.md)

## 5. 스레드 · 스케줄링

### 스레드 (Thread) {#thread}

- **정의**: [프로세스](#process) 안에서 따로 실행되는 작업 단위. 같은 프로세스의 메모리·파일·코드(텍스트/힙/전역 변수)를 공유하지만 자기만의 실행 흐름과 [Stack](#memory-segments), 레지스터 집합(특히 프로그램 카운터)을 가짐. 리눅스에서 스레드는 별도의 특별한 객체가 아니라 *주소 공간 등을 공유하는 또 하나의 실행 단위*([`task_struct`](#pcb))로 구현되며, 같은 프로세스의 스레드들은 하나의 스레드 그룹(TGID = 대표 [PID](#pid))에 묶이고 커널 내부적으로는 각자 고유한 TID 를 가진다. 흔한 오해: "스레드는 항상 프로세스보다 가볍고 안전하다" — 생성·전환 비용은 더 싸지만, 메모리를 공유하므로 한 스레드의 버그(잘못된 포인터, 세그폴트)가 프로세스 전체를 죽인다.
- **비유**: 한 식당(프로세스) 안의 여러 요리사(스레드). 같은 주방과 재료(메모리)를 공유하면서 각자 다른 요리를 동시에 만든다.
- **왜 중요한가**: 멀티스레드는 빠르지만, 같은 자원을 동시에 만지면 충돌([race condition](#race-condition), [데드락](#coffman))이 생긴다. 이번 미션 3번째 장애의 원인. 공유 메모리 때문에 한 스레드의 폭주가 전체 프로세스의 [OOM](#oom-killer)·CPU 폭주로 번지는 점도 분석 포인트.
- > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [clone(2) — Linux manual page](https://man7.org/linux/man-pages/man2/clone.2.html)

### PCB · `task_struct` {#pcb}

- **정의**: 프로세스/스레드의 상태(레지스터 값, [PID](#pid)/TID, 메모리 정보, 열린 파일 목록, 스케줄링 상태·우선순위 등)를 담아두는 OS 내부 자료구조. 리눅스에서는 `task_struct` 라는 C 구조체로 표현되며, 모든 실행 단위(프로세스든 스레드든)마다 하나씩 존재한다. 프로세스마다 따로, 스레드마다 따로 만들어진다는 점이 핵심 — 리눅스 커널은 "프로세스"와 "스레드"를 동일한 `task_struct` 로 다루고 자원 공유 범위만 다르게 둔다.
- **비유**: 식당 손님 카드. 손님 번호, 주문 내역, 결제 정보 등을 한 장에 모아둔 것.
- **왜 중요한가**: OS 가 [컨텍스트 스위치](#context-switch)(다음 스레드로 바꾸기)를 할 때 PCB 의 정보를 저장·복원한다. `/proc/<pid>/` 가 노출하는 대부분의 정보(상태·메모리·스레드 목록)가 이 구조체에서 나오므로, 미션의 모니터링·진단도 결국 PCB 를 들여다보는 일이다.
- > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [docs.kernel.org: Data Structures (task_struct)](https://docs.kernel.org/kernel-hacking/locking.html)

### 컨텍스트 스위치 (Context Switch) {#context-switch}

- **정의**: CPU 가 지금 돌리던 [스레드](#thread)를 멈추고 다른 스레드로 바꾸는 행위. 현재 스레드의 모든 레지스터 값을 [PCB](#pcb) 에 저장하고, 새 스레드의 값을 PCB 에서 꺼내 레지스터에 채운다. 발생 계기는 (1) [타임 슬라이스](#time-slice) 소진, (2) 더 높은 우선순위 스레드의 깨어남(선점), (3) 스레드가 자발적으로 I/O·락 대기로 잠드는 경우.
- **비유**: 요리사가 음식 만들기를 잠시 멈추고, 도마/칼/재료 위치를 메모지에 적어둔 뒤, 다른 요리를 시작. 나중에 돌아오면 그 메모지를 보고 정확히 멈춘 자리부터 재개.
- **왜 중요한가**: 컨텍스트 스위치는 공짜가 아니다(직접 비용은 보통 수 µs, 캐시·[TLB](#page-table-tlb) 가 식는 간접 비용까지 더하면 더 커진다). 너무 자주 일어나면 정작 일하는 시간보다 바꾸는 시간이 길어진다. 또 *프로세스 간* 스위치는 주소 공간이 바뀌어 [TLB](#page-table-tlb) 를 비워야(flush) 하므로 *스레드 간* 스위치보다 훨씬 비싸다. CPU 사용률이 높은데 처리량이 안 나오면 `vmstat` 의 `cs`(context switch) 수치를 의심하라.
- > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [Context switch — Wikipedia](https://en.wikipedia.org/wiki/Context_switch)

### IPC (Inter-Process Communication) {#ipc}

- **정의**: 서로 격리된 [프로세스](#process)끼리 데이터를 주고받는 방법들. 파이프·FIFO(명명된 파이프)·소켓·공유 메모리·메시지 큐·세마포어·[시그널](#signal) 등. POSIX 와 System V 두 계열의 표준 메커니즘이 함께 존재한다. 이 중 공유 메모리만이 "복사 없이" 데이터를 주고받고, 나머지는 대체로 커널을 거치며 복사 비용이 든다.
- **비유**: 다른 사무실에 있는 동료에게 메모를 보내거나(파이프), 전화하거나(소켓), 공용 게시판에 적거나(공유 메모리), 사내 방송으로 알리는 것([시그널](#signal)).
- **왜 중요한가**: 멀티프로세스 설계에서 필수. 메모리가 격리돼 있어 변수 공유가 안 되므로 명시적 통신이 필요. [멀티스레딩](#pthread) 에 비해 통신이 *명시적이고 비싼* 게 멀티프로세싱의 트레이드오프. Python [multiprocessing](#py-multiprocessing) 의 `Queue`·`Pipe` 도 결국 OS IPC 위에 얹혀 있다.
- > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [pipe(7) — Linux manual page](https://man7.org/linux/man-pages/man7/pipe.7.html)

### CPU 코어 · 하이퍼스레딩 {#core-smt}

- **정의**:
  - **코어**: CPU 칩 안에 들어 있는 독립적 처리 유닛. 코어 하나당 동시에 한 [스레드](#thread) 실행 가능.
  - **하이퍼스레딩 (Hyper-Threading, SMT)**: 한 물리 코어를 OS 가 보기에 2개의 "논리 프로세서"처럼 보이게 하는 인텔의 SMT(Simultaneous Multithreading) 구현. 두 논리 프로세서는 각자의 아키텍처 상태(레지스터)를 가지지만 실행 유닛·캐시 같은 물리 자원은 *공유*한다. 4코어 + HT = OS 가 보기엔 8 "논리 코어". 흔한 오해: HT 가 성능을 2배로 만들지는 않는다 — 실행 자원을 공유하므로 워크로드에 따라 통상 15~30% 향상에 그친다.
- **비유**: 코어 = 식당 요리사 인원. HT 는 한 명이 두 일을 번갈아/겹쳐 하는 "1.5인분" 효과.
- **왜 중요한가**: `top` 에서 %CPU 가 400% 찍혔다면 "4개 논리 코어를 다 쓰고 있다" 라는 뜻. 시스템 전체 사용률은 (총합 / 코어 수) 로 나눠서 봐야 함. [병렬성](#concurrency-vs-parallelism) 의 상한은 결국 (논리)코어 수이며, CPU-bound 분석에서 "몇 % 까지가 정상인가"의 기준선이 된다.
- > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [Hyper-threading — Wikipedia](https://en.wikipedia.org/wiki/Hyper-threading)

### idle (유휴 시간) {#idle}

- **정의**: [CPU](#cpu) 가 할 일이 없어서 쉬고 있는 시간의 비율. `top` 의 `id` 컬럼. 리눅스는 이 외에도 I/O 완료를 기다리느라 노는 시간을 `wa`(iowait)로 따로 구분하는데, iowait 도 엄밀히는 CPU 가 노는 상태의 한 종류다. 흔한 오해: `id` 가 0 이라고 곧장 "위험"은 아니다 — 의도적으로 코어를 꽉 채워 쓰는 배치 작업이면 정상일 수 있다.
- **비유**: 식당 요리사가 주문이 없어 그냥 서 있는 시간의 비율.
- **왜 중요한가**: CPU 사용률 = 100% − idle%. `monitor.sh` 가 사용률을 구하는 공식이 그것. 미션의 CPU 폭주 판정 임계치도 결국 idle% 가 바닥을 치는 순간으로 정의된다.
- **출처**: [top(1) — Linux manual page](https://man7.org/linux/man-pages/man1/top.1.html)

### 스케줄러 (Scheduler) {#scheduler}

- **정의**: 여러 [스레드](#thread)가 동시에 실행 가능(runnable)할 때, "다음으로 어느 스레드에게 CPU 를 줄까" 를 결정하는 커널의 핵심 모듈. 리눅스는 실시간 정책(`SCHED_FIFO`/`SCHED_RR`)과 일반 정책(`SCHED_OTHER`)을 함께 가지며, 실시간 클래스가 항상 일반 클래스보다 먼저 CPU 를 가져간다.
- **비유**: 식당 매니저. 손님이 줄을 서면 누구를 어느 테이블에 보낼지 결정한다.
- **왜 중요한가**: 스케줄러 정책이 바뀌면 같은 코드도 다른 속도로 돌 수 있다. "왜 내 작업이 안 끝나지?" 의 답이 스케줄러에 있을 때가 많다. CPU 경쟁이 심한 미션 상황에서 어떤 프로세스가 굶주리는지(starvation) 이해하는 출발점.
- > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [sched(7) — Linux manual page](https://man7.org/linux/man-pages/man7/sched.7.html)

### 타임 슬라이스 (Time Slice / Quantum) {#time-slice}

- **정의**: 한 [스레드](#thread)가 [CPU](#cpu) 를 잡고 있을 수 있는 (목표) 시간. 보통 수 ms 단위. 이 시간이 지나면 [스케줄러](#scheduler)가 강제로 다음 스레드로 바꾼다(선점). 리눅스의 공정 스케줄러는 고정된 슬라이스를 쓰지 않고, 실행 가능한 스레드 수가 많아질수록 각자의 슬라이스를 줄이는 방식이라 슬라이스 길이는 동적이다(예: 기본 슬라이스 파라미터는 수 ms 수준).
- **비유**: 회의실 예약 시간. 30분 슬롯이 다 차면 다음 팀에게 양보해야 한다.
- **왜 중요한가**: 타임 슬라이스가 짧으면 응답성↑ [컨텍스트 스위치](#context-switch)↑(오버헤드↑). 길면 처리량↑ 응답성↓. 트레이드오프의 대표 사례.
- **출처**: [CFS Scheduler — The Linux Kernel documentation](https://docs.kernel.org/scheduler/sched-design-CFS.html)

### CFS · `vruntime` {#cfs}

- **정의**:
  - **CFS (Completely Fair Scheduler)**: 리눅스 2.6.23(2007)부터 오랫동안 기본 [스케줄러](#scheduler)였던 일반 작업용 스케줄러. "모든 [스레드](#thread)를 가능한 한 *공정하게* 다루겠다". (참고: 커널 6.6(2023)부터 기본 스케줄러는 CFS 를 계승·개선한 EEVDF 로 교체되었으나, vruntime 기반 공정성·rbtree 구조 등 핵심 개념은 그대로 이어진다.)
  - **`vruntime`**: 각 스레드가 지금까지 받은 CPU 시간을 [nice 값](#nice) 가중치로 보정한 "가상 실행 시간". 작을수록(= 덜 받았을수록) 우선 실행 대상.
- **비유**: CFS = 식당 매니저가 "오늘 가장 적게 음식 받은 손님에게 다음 음식을 준다" 정책. vruntime = 손님이 (가중치로 환산해) 받은 음식 양.
- **왜 중요한가**: 공정성을 추구하므로 Round-Robin 과 외형이 비슷하지만, 일부 스레드가 자발적으로 sleep 하거나 [nice 값](#nice)이 다르면 비율이 달라진다. CPU 를 두고 여러 프로세스가 다툴 때 "왜 이 비율로 나뉘는가"를 설명해 준다.
- > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [Completely Fair Scheduler — Wikipedia](https://en.wikipedia.org/wiki/Completely_Fair_Scheduler)

### nice 값 · 우선순위 {#nice}

- **정의**: 리눅스에서 일반 [스레드](#thread)에 매기는 "양보 정도". −20 ~ +19 정수(기본 0). 음수일수록 더 많은 CPU 시간을 받음, 양수일수록 양보. nice 차이는 산술이 아니라 가중치(대략 한 단계당 약 1.25배)로 [vruntime](#cfs) 누적 속도에 반영된다. 흔한 오해: nice 는 *일반(SCHED_OTHER)* 작업끼리의 상대적 양보일 뿐, 실시간 우선순위와는 다른 축이다.
- **비유**: 회의실 예약 우선권. 임원(낮은 nice)은 먼저 잡고, 일반 직원(높은 nice)은 양보.
- **왜 중요한가**: `nice` / `renice` 명령으로 조정 가능. 단, nice 를 음수(더 높은 우선순위)로 내리려면 보통 root 권한이 필요하다. 백그라운드 작업은 nice 값을 높여서 시스템 응답성을 지키는 게 정석 — CPU 폭주 프로세스를 죽이지 않고 `renice` 로 길들이는 완화책의 근거.
- **출처**: [sched(7): nice values and weights — Linux manual page](https://man7.org/linux/man-pages/man7/sched.7.html)

### 빨간-검정 트리 (Red-Black Tree) {#rb-tree}

- **정의**: 자가 균형 이진 탐색 트리의 한 종류. 색(빨강/검정) 규칙으로 트리 높이를 O(log n) 으로 묶어, 삽입·삭제·탐색이 모두 O(log n) 보장.
- **비유**: 자동으로 균형을 유지하는 분류 책장. 어느 책을 찾든 비슷한 시간에 찾을 수 있다.
- **왜 중요한가**: 리눅스 공정 스케줄러([CFS](#cfs))가 실행 가능한 [스레드](#thread)들을 vruntime 으로 정렬한 시간순 rbtree 로 관리한다. 가장 왼쪽(= vruntime 최소) 노드는 캐시되어 *다음 실행 대상 선택은 사실상 O(1)*, 삽입·재배치는 O(log n) 으로 끝난다. 그래서 스레드 수가 폭증해도 스케줄링 결정 비용이 안정적이다.
- **출처**: [CFS Scheduler — The Linux Kernel documentation](https://docs.kernel.org/scheduler/sched-design-CFS.html)

### POSIX Threads (pthread) {#pthread}

- **정의**: 유닉스 계열의 표준 [스레드](#thread) API. `pthread_create(&tid, attr, func, arg)` 로 생성, `pthread_join(tid, &ret)` 로 종료 대기, `pthread_mutex_*` 로 락 관리. 리눅스의 현행 구현은 NPTL(Native POSIX Thread Library)로, 각 사용자 스레드가 커널 스케줄링 단위 하나에 1:1 로 대응한다.
- **비유**: 모든 유닉스 식당이 따르는 "요리사 추가 채용 표준 매뉴얼". 식당마다 똑같은 양식으로 요리사를 부르고 보낼 수 있다.
- **왜 중요한가**: C/C++/Rust 등 시스템 언어의 멀티스레드는 거의 다 pthread 위에 얹혀 있다. Python `threading` 모듈도 내부적으로 pthread 를 부른다. 리눅스에서 `pthread_create` 는 메모리·파일·시그널 핸들러를 공유하도록 플래그를 켠 [`clone()`](#clone-syscall) 시스템 콜로 구현되고, 동기화(mutex·join)는 [`futex`](#futex) 위에서 동작한다.
- > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [nptl(7) — Linux manual page](https://man7.org/linux/man-pages/man7/nptl.7.html) · [pthreads(7) — Linux manual page](https://man7.org/linux/man-pages/man7/pthreads.7.html)

### TLS (Thread-Local Storage) {#tls}

- **정의**: 같은 [프로세스](#process) 안의 [스레드](#thread)들이 메모리를 공유하지만, *스레드별로 따로 존재하는* 변수를 두는 메커니즘. C 에서는 `__thread int x`(또는 C11 `_Thread_local`), Python 에서는 `threading.local()`. 각 스레드는 같은 이름의 변수에 접근해도 자기 전용 사본을 보므로 [락](#lock-mutex) 없이 안전하다.
- **비유**: 식당의 공용 주방에서 각 요리사가 자기만의 도마와 칼을 따로 갖는 것. 재료는 공유하지만 작업 도구는 개인용.
- **왜 중요한가**: 멀티스레드에서 "스레드마다 다른 컨텍스트(예: DB 커넥션, 로깅 prefix)" 가 필요할 때 [락](#lock-mutex) 없이도 안전하게 가질 수 있다. C 표준 라이브러리의 오류 코드 `errno` 도 멀티스레드에서 스레드별 변수로 정의되어, 한 스레드의 오류가 다른 스레드의 `errno` 를 덮어쓰지 않는다.
- > 더 깊이: [이론 지식](이론_지식.md)

---
- **출처**: [Thread-local storage — Wikipedia](https://en.wikipedia.org/wiki/Thread-local_storage) · [errno(3) — Linux manual page](https://man7.org/linux/man-pages/man3/errno.3.html)

## 6. 병렬 실행 모델

### GIL (Global Interpreter Lock) {#gil}

- **정의**: CPython(가장 흔한 Python 구현체) 인터프리터 전체를 하나의 거대한 [락](#lock-mutex)으로 묶는 메커니즘. 결과: **같은 프로세스 안에서는 한 번에 *오직 한 [스레드](#thread)만* Python 바이트코드를 실행할 수 있다**. GIL 은 영원히 한 스레드가 쥐고 있는 게 아니라, 인터프리터가 주기적으로(기본 *약 5ms* 의 "switch interval", `sys.setswitchinterval()` 로 조정 가능) 다른 스레드에 넘겨 *동시성의 환상* 을 만든다. 흔한 오해 둘: ① GIL 은 Python *언어* 의 속성이 아니라 *CPython 구현* 의 속성이다(Jython·IronPython 엔 GIL 이 없다). ② "GIL 때문에 스레드가 쓸모없다" 는 과장 — I/O 대기·`time.sleep`·많은 C 확장(NumPy 등)은 GIL 을 *놓아주므로* 그 구간은 진짜로 병렬이다.
- **비유**: 5인용 회의실에 마이크가 하나뿐인 회의. 5명이 들어와 있어도 마이크 가진 한 명만 발언 가능. 단, 발언자가 잠깐 물 마시러 가면(I/O 대기) 마이크를 내려놓아 다른 사람이 집을 수 있다.
- **왜 중요한가**: GIL 때문에 Python 멀티스레드는 **[CPU-bound](#cpu-vs-io-bound) 작업에는 효과가 없다** (코어가 8개여도 1개만 쓰는 셈). [I/O-bound](#cpu-vs-io-bound) 작업은 I/O 대기 중 GIL 이 풀려서 의미가 있음. CPU 병렬화가 필요하면 [멀티프로세싱](#py-multiprocessing) 으로 가야 한다. CPython 의 메모리 관리(참조 카운팅)가 스레드-안전하지 않아서 도입된 역사적 결정. 최신 흐름: Python 3.12 는 인터프리터별 독립 GIL(PEP 684)을, 3.13 은 GIL 을 끄는 *free-threaded* 실험 빌드(PEP 703)를 도입했고 3.14 에서 단일스레드 오버헤드가 5~10%대로 개선됐다 — 다만 표준 빌드는 여전히 GIL 이 켜져 있다. 미션에서 `MULTI_THREAD_ENABLE` 로 스레드를 켰는데도 [CPU](#cpu) 가 한 코어만 100% 라면 이 GIL 이 원인이다.
  > 더 깊이: [이론 지식 §4.5 — Python 의 GIL](이론_지식.md)
- **출처**: [sys — System-specific parameters and functions](https://docs.python.org/3/library/sys.html#sys.setswitchinterval) · [Python support for free threading](https://docs.python.org/3/howto/free-threading-python.html)

### multiprocessing (Python) {#py-multiprocessing}

- **정의**: Python 의 표준 멀티프로세스 라이브러리. `multiprocessing.Process`, `Pool`, `Queue` 등을 제공해 [GIL](#gil) 을 우회하는 진짜 병렬 실행을 가능하게 한다(자식 프로세스마다 자기 인터프리터·자기 GIL 을 가진다). *프로세스를 만드는 방식(start method)* 이 셋 있다 — `fork`(부모를 통째로 복제, Linux 전통 기본·빠르지만 멀티스레드 부모에선 위험), `spawn`(빈 인터프리터를 새로 띄움, 안전·느림), `forkserver`(중립 서버에서 fork). 흔한 오해: "항상 `fork` 를 쓴다" 는 옛말 — Windows·macOS 는 예전부터 `spawn` 이 기본이고, **Python 3.14 부터 macOS 를 제외한 Unix 의 기본값이 `fork` → `forkserver` 로 바뀌었다**.
- **비유**: 마이크 하나로 회의가 안 굴러가니, 회의실을 5개로 쪼개서 5개 회의를 동시에 열기. 회의실마다 마이크(GIL)가 따로 있으니 5명이 동시에 발언한다.
- **왜 중요한가**: CPU-bound Python 작업을 멀티코어로 가속하는 *표준* 방법. 단점은 [IPC](#ipc) 가 명시적이고 비싸다는 것 — 객체를 [pickle](https://docs.python.org/3/library/pickle.html) 로 직렬화해서 보내야 하고, `spawn`/`forkserver` 는 자식이 부모 모듈을 다시 import 한다(전역 코드가 또 실행됨에 유의). 미션에서 [메모리](#dram) 사용량이 *프로세스 수에 비례* 해 늘어나 [OOM](#oom-killer) 으로 이어질 수 있는 지점이 바로 여기다.
  > 더 깊이: [이론 지식 §4.2 — 멀티프로세싱](이론_지식.md)
- **출처**: [What's New In Python 3.14 — multiprocessing](https://docs.python.org/3/whatsnew/3.14.html)

### asyncio · 이벤트 루프 {#asyncio}

- **정의**: 단일 [스레드](#thread) 안에서 *수많은 동시 작업* 을 효율적으로 굴리기 위한 비동기 프로그래밍 모델. Python `asyncio`, JavaScript Node.js, C 의 libuv 등이 같은 원리.
  - **이벤트 루프**: "지금 깨어 있어야 할 작업" 들의 큐를 돌며 하나씩 실행하다가, I/O 대기에 들어가면 다음 작업으로 넘어가는 단일 스레드 스케줄러. 내부적으로 OS 의 다중화 호출(Linux `epoll`, BSD `kqueue`, 범용 `select`)로 "어떤 [파일 디스크립터](#file-descriptor)가 준비됐는지" 를 한 번에 묻는다.
  - **협력적(cooperative) 스케줄링**: [스레드](#thread)는 OS 가 강제로 뺏는(선점) 반면, 코루틴은 `await` 지점에서 *스스로* 제어권을 양보한다. `await` 가 없으면 양보도 없다 — 이게 아래 "블로킹" 문제의 뿌리다.
- **비유**: 1명의 손님 응대 직원이 손님 A 에게 "메뉴 보고 계세요" 하고 손님 B 응대를 시작, 그 사이 손님 A 가 결정하면 다시 가는 식. 한 명이 100명을 효율적으로 응대.
- **왜 중요한가**: I/O-bound 작업에 [스레드](#thread) 100개를 만드는 대신 *하나의 이벤트 루프* 로 처리하면 메모리·컨텍스트 스위치 비용이 훨씬 적다. 단, CPU-bound 작업이 끼거나 동기 블로킹 호출(예: `requests.get`, `time.sleep`)을 그대로 부르면 *모든 다른 작업이 같이 멈춘다* (이벤트 루프 블로킹). 미션에서 단일 프로세스의 [CPU](#cpu) 가 100% 인데 응답이 다 굳어버리면 이벤트 루프 블로킹을 의심하라.
  > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [asyncio — Asynchronous I/O](https://docs.python.org/3/library/asyncio.html) · [Developing with asyncio](https://docs.python.org/3/library/asyncio-dev.html)

### 암달의 법칙 (Amdahl's Law) {#amdahl}

- **정의**: 1967년 Gene Amdahl 이 AFIPS 학회에서 발표한 병렬화의 수학적 한계.
  - **Speedup ≤ 1 / ((1 − p) + p / N)** &nbsp;&nbsp; (p = 병렬화 가능한 코드 비율, N = 코어 수)
  - N → ∞ 로 보내면 분모의 `p/N → 0`, 즉 **최대 speedup 은 `1 / (1 − p)`** 에 수렴한다 — 직렬 부분이 천장을 정한다. 흔한 오해: "코어를 2배로 늘리면 2배 빨라진다" 는 p=1(완전 병렬)일 때만 참이다.
- **비유**: 부엌 일에서 *씻기·자르기* 는 사람을 늘려 빨라지지만 *국 한 솥을 끓이는 시간* 은 사람 수 늘려도 안 줄어든다. 끓이는 시간이 직렬 부분.
- **왜 중요한가**: 직렬 부분 5% 만 있어도(p=0.95) 코어를 *무한* 으로 늘려도 최대 20배. 50% 직렬이면 최대 2배. 그래서 병렬 시스템 설계의 핵심은 "직렬 구간을 어떻게 줄이느냐" 다. [락](#lock-mutex)으로 묶인 [임계구역](#critical-section)은 본질적으로 직렬 구간이므로, 락 경합이 심하면 코어를 늘려도 [CPU](#cpu) 활용률이 오르지 않는다.
  > 더 깊이: [이론 지식 §4.7 — 암달의 법칙](이론_지식.md)
- **출처**: [Amdahl's law — Wikipedia](https://en.wikipedia.org/wiki/Amdahl%27s_law)

### 스레드 풀 (Thread Pool) {#thread-pool}

- **정의**: [스레드](#thread)를 그때그때 만들지 않고, *미리 N개* 를 만들어 두고 작업을 분배하는 패턴. 작업이 끝난 스레드는 죽지 않고 다음 작업을 받아 재사용한다. 풀 크기 N 은 워크로드 성격에 맞춘다 — CPU-bound 면 대략 코어 수, I/O-bound 면 그보다 크게 잡는 게 보통이다.
- **비유**: 콜센터 — 전화 올 때마다 사람 뽑지 않고, 상담사 50명을 미리 대기시켜 두고 들어오는 전화를 분배.
- **왜 중요한가**: 스레드 생성·종료 비용이 작업당 비용보다 클 때 큰 차이가 난다. Python `concurrent.futures.ThreadPoolExecutor`, Java `ExecutorService` 등이 표준 구현. [멀티프로세싱](#py-multiprocessing) 에서도 `ProcessPoolExecutor`, `multiprocessing.Pool` 로 같은 패턴. 다만 풀 크기를 무한정 키우면 스레드/프로세스마다 [스택](#memory-segments)·[메모리](#dram)를 먹어 오히려 [OOM](#oom-killer) 이나 컨텍스트 스위치 폭증을 부른다 — 풀의 핵심은 *상한을 두는 것* 이다.
  > 더 깊이: [이론 지식 §4.6 — 동시 실행 패턴](이론_지식.md)
- **출처**: [concurrent.futures — Launching parallel tasks](https://docs.python.org/3/library/concurrent.futures.html#threadpoolexecutor)

### Producer-Consumer (생산자-소비자) {#producer-consumer}

- **정의**: 한 쪽은 작업을 [큐](#critical-section)에 *생산* 하고, 다른 쪽은 큐에서 꺼내 *소비* 한다. 큐가 비면 소비자가 대기, 가득 차면 생산자가 대기(이 "가득 차면 대기" 가 백프레셔/backpressure — 빠른 생산자가 느린 소비자를 압도해 메모리를 무한히 먹는 걸 막는다).
- **비유**: 카페에서 바리스타(생산자)가 음료를 카운터(큐)에 올리면 손님(소비자)이 가져간다. 카운터 자리가 다 차면 바리스타가 멈추고, 비어 있으면 손님이 기다린다.
- **왜 중요한가**: 멀티스레드·멀티프로세스 동시 실행의 가장 흔한 패턴. 큐 자체는 [락](#lock-mutex) + [컨디션 변수](#condition-variable) 로 구현된다(Python `queue.Queue`, `multiprocessing.Queue`). 데이터 파이프라인의 기본 형태. 큐에 *상한이 없으면*(`maxsize=0`) 느린 소비자 때문에 큐가 무한정 자라 [OOM](#oom-killer) 으로 직결되므로, 미션 메모리 분석에서 눈여겨볼 구조다.
  > 더 깊이: [이론 지식 §4.6 — 동시 실행 패턴](이론_지식.md)
- **출처**: [queue — A synchronized queue class](https://docs.python.org/3/library/queue.html)

### Fork-Join (분할-합치기) {#fork-join}

- **정의**: 큰 작업을 N 개로 쪼개서(fork) 병렬 실행 → 모두 끝날 때까지 기다렸다가(join) 결과를 모으는 패턴. 전체 소요 시간은 *가장 느린 한 조각* 에 묶인다(낙오자/straggler 문제) — 그래서 작업을 균등하게 쪼개는 게 중요하다.
- **비유**: 큰 시험지 채점을 5명에게 나눠 주고, 다 끝나면 점수를 합산. 한 명이 유독 두꺼운 묶음을 받으면 나머지 4명이 그를 기다린다.
- **왜 중요한가**: MapReduce, 병렬 정렬, 행렬 곱셈 등 대량 데이터 처리의 기본 구조. Java `ForkJoinPool`, Go 의 `sync.WaitGroup`, Python `concurrent.futures` 의 결과 수집 등이 표준 구현 제공. join 지점은 [암달의 법칙](#amdahl)의 직렬 구간이자, 한 워커가 락을 쥔 채 끝나지 않으면 [데드락](#coffman)으로 전체가 멈출 수 있는 합류점이다.
  > 더 깊이: [이론 지식 §4.6 — 동시 실행 패턴](이론_지식.md)

- **출처**: [ForkJoinPool — Java Platform SE API](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/ForkJoinPool.html)

## 7. 동시성 · 데드락

### Race Condition (경쟁 조건) {#race-condition}

- **정의**: 두 개 이상의 [스레드](#thread)(또는 [프로세스](#process))가 같은 자원을 동시에 만질 때, 연산의 *인터리빙(interleaving) 순서*에 따라 최종 결과가 달라지는 상태. 결과의 정확성이 스케줄러의 타이밍이라는 *통제 불가능한 변수*에 의존하게 된다. 흔한 오해는 "`x += 1` 한 줄이니 원자적(atomic)이겠지" 인데, 이 한 줄도 실제로는 *읽기 → 더하기 → 쓰기* 세 단계로 쪼개지므로 두 스레드가 끼어들면 증가분 하나가 통째로 사라진다(lost update). "버그가 가끔만 나타난다" 는 가장 무서운 종류의 버그.
- **비유**: 식당에서 두 종업원이 동시에 같은 주문서를 읽어 같은 음식을 두 번 만드는 사고.
- **왜 중요한가**: 한 번 만들고 끝나는 게 아니라, *발생 가능 조건* 만 잠재돼 있다. 테스트에서 99 번 통과해도 100 번째에 사고가 난다. [락](#lock-mutex)으로 [임계구역](#critical-section)을 보호하는 게 정석. 미션의 Deadlock·CPU 이상 분석에서 "왜 산발적으로만 재현되는가" 를 설명하는 출발점이다.
- > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [Race condition — Wikipedia](https://en.wikipedia.org/wiki/Race_condition)

### 임계구역 (Critical Section) {#critical-section}

- **정의**: 한 번에 *한 [스레드](#thread)만* 들어가야 하는 코드 구간. 공유 자원을 읽거나 쓰는 부분 전체를 가리킨다. 임계구역은 가능한 한 *짧게* 잡는 것이 원칙인데, 길수록 다른 스레드의 대기 시간이 늘어 동시성 이점이 사라지고(직렬화), 그 안에서 다른 락을 또 잡으면 [데드락](#coffman) 위험이 커지기 때문이다.
- **비유**: 일인용 탈의실. 한 명이 들어가면 다른 사람은 밖에서 기다려야 한다.
- **왜 중요한가**: 임계구역을 정확히 식별해서 [락](#lock-mutex)으로 둘러싸야 [race condition](#race-condition) 을 막을 수 있다. 식별이 너무 좁으면 보호가 새고, 너무 넓으면 성능이 죽는다.

### 락 · 뮤텍스 (Lock · Mutex) {#lock-mutex}

- **정의**:
  - **락**: 자원을 보호하는 잠금 장치. 잡으면(acquire) 다른 [스레드](#thread)는 못 잡고 대기.
  - **뮤텍스 (Mutex = Mutual Exclusion)**: 락의 가장 흔한 구현. 한 번에 한 명만 잡을 수 있는 가장 단순한 락. 핵심 규약은 *잠근 스레드가 직접 풀어야 한다*(소유권 개념)는 것 — 이 점이 아무나 풀 수 있는 [세마포어](#semaphore)와 다르다. 파이썬 `threading.Lock` 은 "locked / unlocked" 두 상태만 갖는 가장 원시적인 락으로, `acquire()` / `release()` 로 조작한다.
- **비유**: 탈의실 문 잠금. 한 명만 잠그면 다른 사람은 밖에서 기다린다. 안에 있는 사람이 풀어야 다음 사람 입장.
- **왜 중요한가**: 락을 잘못 쓰면 [데드락](#coffman)이 난다(획득 순서 불일치, release 누락 등). 미션에서 멀티스레드 프로세스가 멈춰 있을 때, 그 락의 내부 구현인 [futex](#futex) 대기로 잡히는지가 진단 단서다.
- **출처**: [threading — Lock Objects, Python docs](https://docs.python.org/3/library/threading.html#lock-objects)

### 세마포어 (Semaphore) {#semaphore}

- **정의**: 내부에 *정수 카운터*를 두는 동기화 장치. `acquire()` 마다 1 감소, `release()` 마다 1 증가하며, 카운터가 0이면 `acquire()` 가 블록된다. 초기값 N 이면 동시에 N 명까지 통과한다([뮤텍스](#lock-mutex)는 N=1 인 특수 경우로 볼 수 있다). 단, 엄밀히는 뮤텍스의 "소유권" 개념이 세마포어엔 없어서, 잡은 스레드가 아닌 다른 스레드가 `release` 할 수도 있다.
- **비유**: 좌석 5개짜리 회의실. 5명까지는 들어가고, 6번째는 한 명 나갈 때까지 대기.
- **왜 중요한가**: "DB 커넥션 풀 5개" 같은 동시 접속 수 제한 시나리오에 적합. 데드락 가능성은 뮤텍스와 동일하게 존재하며, `release` 누락 시 카운터가 새서 영구 블록(일종의 [기아](#starvation))이 생길 수 있다.
- **출처**: [threading — Semaphore Objects, Python docs](https://docs.python.org/3/library/threading.html#semaphore-objects)

### 컨디션 변수 (Condition Variable) {#condition-variable}

- **정의**: "특정 조건이 만족될 때까지 잠들었다가, 다른 [스레드](#thread)가 깨워줄 때 일어나는" 동기화 메커니즘. 항상 [뮤텍스](#lock-mutex)와 한 쌍으로 쓰며, `wait()` 는 *락을 원자적으로 풀고 잠든 뒤 깨어나면 다시 락을 잡는다*. 깨우는 쪽은 `notify()`(한 명)·`notify_all()`(전부)을 호출한다. C 에서 `pthread_cond_wait` / `pthread_cond_signal`, Python 에서 `threading.Condition`. 깨어난 뒤에도 조건을 *다시 검사*(`while` 루프)해야 한다 — 가짜 깨어남(spurious wakeup)과 다른 소비자가 먼저 채간 경우 때문.
- **비유**: 손님이 "음식 나오면 깨워 주세요" 하고 자리에서 잠든 뒤, 종업원이 와서 어깨를 두드려 깨우는 방식. [busy-wait](#busy-wait) 처럼 계속 확인하지 않아도 됨.
- **왜 중요한가**: [Producer-Consumer 패턴](#producer-consumer) 의 핵심 도구. 큐가 비었을 때 소비자가 *효율적으로 잠들었다가* 생산자가 깨우는 식이라 [CPU](#cpu)를 태우지 않는다(미션의 CPU 100% 진단에서 "busy-wait 대신 condition variable 을 썼는가" 가 갈림길). 락만으로는 못 만든다.
- **출처**: [threading — Condition Objects, Python docs](https://docs.python.org/3/library/threading.html#condition-objects)

### 데드락 · Coffman 조건 {#coffman}

- **정의**: 데드락 = 둘 이상의 스레드가 서로가 쥔 자원을 무한히 기다려 *전원이* 멈춘 상태(누구도 진행하지 못함). Coffman·Elphick·Shoshani 가 1971년 논문 *System Deadlocks* 에서 정리한 4가지 조건 — (1) 상호 배제(mutual exclusion), (2) 점유 대기(hold-and-wait), (3) 비선점(no preemption), (4) 순환 대기(circular wait) — 이 *동시에* 성립할 때만 발생한다. 즉 이 중 *하나만 깨도* 데드락은 원천 차단된다.
- **비유**: 좁은 골목에서 두 차가 마주쳐 서로 양보하지 않고 영원히 멈춰 있는 상황.
- **왜 중요한가**: "데드락 막으려면 어떤 조건을 깨야 하나" 의 체크리스트. 4개 다 외워야 데드락 분석을 자기 말로 할 수 있고, 미션 리포트의 *Root Cause Analysis* 에서 관찰된 멈춤이 어느 조건의 성립 때문인지 짚어 설명할 수 있다.
- > 더 깊이: [이론 지식](이론_지식.md)
- **출처**: [System Deadlocks (Coffman, Elphick, Shoshani, 1971)](https://uobdv.github.io/Design-Verification/Supplementary/System_Deadlocks-Four_necessary_and_sufficient_conditions_for_deadlock.pdf)

### lock-free 자료구조 {#lock-free}

- **정의**: [락](#lock-mutex) 없이도 멀티스레드가 안전하게 쓸 수 있도록 설계한 자료구조. CPU 의 원자적 명령인 `compare-and-swap`(CAS — "기대값과 같으면 새 값으로 바꾸기" 를 한 번에 수행)으로 동기화한다. 정확히는 *lock-free* 는 "최소 한 스레드는 항상 진행한다" 는 보장이고, 더 강한 *wait-free* 는 "모든 스레드가 유한 단계 안에 끝난다" 는 보장이다. CAS 기반 설계의 대표적 함정은 ABA 문제(값이 A→B→A 로 돌아오면 CAS 가 "안 변했다" 고 착각).
- **비유**: 일인용 탈의실 대신, 누가 들어가도 충돌 안 나는 자동 회전문.
- **왜 중요한가**: 락이 없으니 [데드락](#coffman) 자체가 사라진다(Coffman 의 상호 배제 조건을 깸). 단, 설계가 매우 어렵고 ABA·메모리 회수 문제 때문에 모든 자료구조에 만들 수 있는 건 아니다.
- **출처**: [Compare-and-swap — Wikipedia](https://en.wikipedia.org/wiki/Compare-and-swap)

### try_lock · 타임아웃 {#try-lock}

- **정의**: "[락](#lock-mutex)을 잡으려고 시도하되, 즉시(try) 또는 X 초 안에 못 잡으면 *블록하지 않고* 실패를 반환하고 포기" 하는 패턴. Python `Lock.acquire(blocking=False)` 또는 `acquire(timeout=…)`, C++ `try_lock` 이 그 예. 실패 시 호출 측이 이미 쥔 락을 풀고 재시도(backoff)할 길이 열린다.
- **비유**: 회의실 문이 잠겨 있으면 5분만 기다리고 다른 회의실 찾아가는 행동.
- **왜 중요한가**: [Coffman 조건](#coffman)의 "점유 대기/비선점" 을 깰 수 있는 실용적 방법. 무한 대기 대신 포기·재시도로 전환하므로, 미션에서 데드락 회피 설계의 근거로 인용하기 좋다.

### 식사하는 철학자 문제 (Dining Philosophers) {#dining-philosophers}

- **정의**: 동시성 학습용 고전 시나리오. 5명의 철학자가 원탁에 앉아 양옆 포크 *두 개를 동시에* 들어야 식사 가능 — 모두가 왼쪽 포크를 먼저 들면 오른쪽 포크를 영원히 기다려 데드락(순환 대기)에 빠진다. 원형은 1965년 다익스트라(Edsger Dijkstra)가 출제한 문제이며(원래 컴퓨터의 테이프 드라이브 경쟁 비유), 1971년 토니 호어(Tony Hoare)가 지금의 "철학자와 포크" 형태로 재구성했다. 자원 순서화(번호가 낮은 포크부터 들기) 등으로 순환 대기를 깨면 해결된다.
- **비유**: 5명이 원탁에 앉아 포크 5개를 사이에 두고 식사. 모두가 동시에 왼쪽 포크를 들고 오른쪽을 기다리면 영원히 멈춤.
- **왜 중요한가**: 운영체제 교과서마다 등장. 면접 단골 질문. 이번 미션 리포트의 *Root Cause Analysis* 에 인용하면 [Coffman 조건](#coffman)과 묶여 설득력이 확 올라간다.
- **출처**: [Dining philosophers problem — Wikipedia](https://en.wikipedia.org/wiki/Dining_philosophers_problem)

### 기아 (Starvation) {#starvation}

- **정의**: [데드락](#coffman)은 아니지만(시스템 전체는 진행함) *특정 [스레드](#thread)만* 영영 자원을 못 받아 진행하지 못하는 상태. 다른 스레드들이 계속 우선 처리돼서 차례가 안 온다. 데드락과의 결정적 차이: 데드락은 *아무도* 진행 못 하고 영구적이지만, 기아는 *일부만* 막히고 운(타이밍)에 따라 풀릴 수도 있다.
- **비유**: 식당에서 한 손님이 매번 우선순위 낮아서 새치기 당하다가 영원히 음식 못 받는 상황.
- **왜 중요한가**: 우선순위 기반 스케줄링이나 불공정한 락에서 자주 발생. 데드락과 다른 개념이라 진단 시 구분해야 한다(멈춘 스레드가 *전부*인지 *일부*인지가 1차 분기점).

### 프로세스 상태 코드 (STAT) {#stat-codes}

- **정의**: `ps` 의 `STAT` 컬럼에 찍히는 상태 코드. 첫 글자가 주 상태, 뒤따르는 글자는 부가 플래그다.
  - `R`: running or runnable (실행 중 또는 실행 큐에서 대기 가능)
  - `S`: interruptible sleep (이벤트 대기 중, 시그널로 깨울 수 있음)
  - `D`: uninterruptible sleep (보통 디스크 I/O 대기, 시그널로 강제 깨우기 불가 — 이래서 `D` 가 많으면 I/O 병목 신호)
  - `Z`: defunct/zombie (종료됐는데 부모가 회수(reap) 안 함)
  - `T`: stopped (job control 시그널, 예: Ctrl+Z 로 정지)
  - 뒤에 `l`: 멀티스레드(NPTL pthreads 등 `CLONE_THREAD`), `+`: 포그라운드 프로세스 그룹, `s`: 세션 리더, `<`: 높은 우선순위, `N`: 낮은 우선순위
- **비유**: 손님 상태 — 식사 중(R), 음식 기다림(S), 화장실(D), 가버림(Z).
- **왜 중요한가**: 데드락 의심 시 `Sl` 이 뜨면 *멀티스레드 프로세스가 인터럽트 가능 sleep* 에 잠겨 있다는 강한 단서. `R` 인데 안 끝나면 [무한 루프](#infinite-loop)/[busy-wait](#busy-wait)를, `D` 가 박혀 있으면 I/O 대기를 의심하는 식으로 1차 분류에 쓴다.
- **출처**: [ps(1) — Linux manual page](https://man7.org/linux/man-pages/man1/ps.1.html)

### futex (Fast Userspace muTEX) {#futex}

- **정의**: 리눅스에서 유저 공간 [락](#lock-mutex)을 효율적으로 구현하기 위한 커널 기반 메커니즘과 그 [시스템 콜](#syscall)(`futex(2)`). 핵심은 *경합 없는(uncontended) 경우를 빠르게* 만드는 것 — 락이 비어 있으면 유저 공간의 원자적 연산만으로 끝나 커널 진입이 아예 없고(나노초 수준), 경합이 생겼을 때만 `FUTEX_WAIT`/`FUTEX_WAKE` 시스템 콜로 커널에 잠들고 깨운다.
- **비유**: 회의실 문이 안 잠겨 있으면 그냥 들어가고(빠름), 잠겨 있을 때만 "노크하고 기다리는" 시스템.
- **왜 중요한가**: Python `threading.Lock`, C++ `std::mutex` 의 락 대기가 내부적으로 futex 로 내려간다. 그래서 데드락이 나면 [`wchan`](#wchan)에 `futex_wait_queue` 같은 이름이 찍혀 "이 스레드는 락을 기다리는 중" 이라는 결정적 단서가 된다.
- **출처**: [futex(7) — Linux manual page](https://man7.org/linux/man-pages/man7/futex.7.html)

### `wchan` (Wait Channel) {#wchan}

- **정의**: [스레드](#thread)가 sleep 상태일 때 *어느 커널 함수 안에서 잠들어 있는지*를 가리키는 심볼릭 이름. `/proc/<pid>/wchan` 파일이 "the symbolic name corresponding to the location in the kernel where the process is sleeping" 를 담으며, `ps -L -o wchan` 으로도 본다(읽기엔 ptrace 권한 필요). 깨어 있거나 잠들어 있지 않으면 `0`(또는 빈 값)으로 표시된다.
- **비유**: 잠든 사람이 "어떤 일 때문에 잠들었는지" 라벨을 붙여둔 것.
- **왜 중요한가**: `wchan` 만 보면 "이 스레드가 락 대기([futex](#futex))인가, I/O 대기인가, 단순 sleep 인가" 가 한눈에 구분된다. 외부(비침습)에서 코드를 안 봐도 데드락 vs CPU spike 를 가르는 핵심 컬럼이다.
- **출처**: [proc_pid_wchan(5) — Linux manual page](https://man7.org/linux/man-pages/man5/proc_pid_wchan.5.html)

### busy-wait (바쁜 대기) {#busy-wait}

- **정의**: 조건이 만족될 때까지 `while (!condition) {}` 식으로 *[CPU](#cpu) 를 계속 돌리면서* 폴링하며 기다리는 패턴(스핀 대기/spin-wait). 잠들지 않으므로 코어 하나를 100% 점유한다 — CPU 100% 점유의 가장 흔한 원인. 임계구역이 *극히 짧을 때만* 스핀락이 정당화되고, 그 외에는 거의 항상 [컨디션 변수](#condition-variable) 같은 블로킹 대기가 옳다.
- **비유**: 친구가 올 때까지 문 앞에서 계속 깡총깡총 뛰는 행동. 가만히 의자에 앉아 기다리면 될 일을.
- **왜 중요한가**: 올바른 동시성 코드는 [컨디션 변수](#condition-variable) 같은 메커니즘으로 잠들었다가 *깨어나는* 방식을 쓴다. busy-wait 는 거의 항상 버그 또는 미숙한 설계이며, 미션의 CPU spike 진단에서 `STAT=R` + 높은 `%CPU` + 비-`futex` `wchan` 조합으로 식별된다.

### 무한 루프 (Infinite Loop) {#infinite-loop}

- **정의**: 종료 조건이 영영 거짓이 되지 않아 끝나지 않는 반복문. `while True: pass`, `for(;;) {}` 같은 코드. 안에서 sleep·I/O·블로킹 호출이 없으면 한 코어를 100% 태운다.
- **비유**: "끝낼 때까지 계속" 만 정해두고 *끝낼 조건* 을 안 적은 일. 영원히 안 멈춘다.
- **왜 중요한가**: [CPU](#cpu) 100% 점유의 대표 원인. [busy-wait](#busy-wait) 도 무한 루프의 한 종류(차이: busy-wait 는 *조건을 돌면서* 계속 체크). 무한 루프는 [Watchdog](#watchdog) 같은 외부 감시가 깨주지 않으면 자체적으로 못 멈추므로 시스템에 *반드시* 안전망이 필요하다.

### 참조 카운팅 (Reference Counting) {#refcount}

- **정의**: 모든 객체에 "지금 나를 가리키는 참조가 몇 개냐" 라는 정수 카운터를 붙이고, 카운터가 0 이 되면 *즉시* 메모리를 해제하는 메모리 관리 방식. CPython 의 기본 [GC](#malloc-free-gc) 가 이 방식이며, 보충적으로 순환 참조(서로를 가리켜 카운터가 0이 안 되는 경우)를 잡는 *순환 가비지 컬렉터*를 함께 둔다. 일부 객체는 불멸(immortal)로 지정돼 카운터가 갱신되지 않는다.
- **비유**: 도서관 책에 "지금 누가 빌렸나" 카드를 붙여 두고, 카드가 비면 책장에 자동 반납.
- **왜 중요한가**: [GIL](#gil) 이 존재하는 *근본 이유*. 멀티스레드가 동시에 카운터를 ++/-- 하면 [race condition](#race-condition) 으로 카운터가 망가져 메모리 누수 또는 이중 해제(=크래시)가 일어난다. GIL 은 그걸 "한 번에 한 스레드만 바이트코드를 실행하게" 막는 가장 단순한 해결책으로, 객체 모델을 암묵적으로 스레드 안전하게 만든다(대신 멀티코어 병렬성을 희생).

---

## 8. 운영 도구

### `/proc` 가상 파일시스템 {#proc}

- **정의**: 실제 디스크에 데이터가 없는 **의사(pseudo) 파일시스템**. 커널 내부 자료구조를 텍스트 파일처럼 노출하는 인터페이스로, 보통 `/proc` 에 마운트된다. 실행 중인 프로세스마다 PID 이름의 디렉터리(`/proc/<pid>/`)가 생기고, 시스템 전체 정보는 `/proc/cpuinfo`·`/proc/meminfo`·`/proc/stat` 등에 있다. 파일 대부분은 읽기 전용이지만 일부(`/proc/sys/...`, sysctl)는 쓰기로 커널 파라미터를 런타임에 바꿀 수 있다. 흔한 오해: "파일이니 디스크 용량을 차지한다" — 아니다. 읽는 순간 커널이 값을 생성하는 가상 파일이라 `ls -l` 의 크기는 대부분 0 으로 보인다.
- **비유**: 빌딩 1층에 걸린 실시간 전광판. 지금 누가 어디 있는지 텍스트로 보여준다.
- **왜 중요한가**: `top`, `ps`, `free` 모두 결국 `/proc` 을 읽는다. 명령어가 막히거나 의심스러울 때 `cat /proc/<pid>/status` 로 RSS·스레드 수·State 같은 원본 진실에 직접 닿을 수 있다.
- > 더 깊이: [이론 지식 §6 시스템 관제 도구 정리](이론_지식.md)
- **출처**: [proc(5) — Linux manual page](https://man7.org/linux/man-pages/man5/proc.5.html) · [The /proc Filesystem — The Linux Kernel documentation](https://docs.kernel.org/filesystems/proc.html)

### `top` · `htop` · `ps` {#top-ps}

- **정의**:
  - `top`: [프로세스](#process) 사용량(CPU·메모리)을 *실시간* 으로 갱신(기본 3초 간격)해 보여주는 도구. `q` 로 종료, `top -H` 는 스레드 단위 표시.
  - `htop`: top 의 향상 버전. 색깔·트리뷰·마우스 지원. (별도 설치 필요)
  - `ps`: 그 순간의 *스냅샷* 만 한 번 찍어 표시. 출력이 고정 텍스트라 스크립트에서 파싱하기 좋음.
- **비유**: top = CCTV 라이브 화면, ps = 폴라로이드 사진 한 장.
- **왜 중요한가**: 데드락은 시간이 흐르지 않으므로 ps 한 장으로 충분(특히 `ps -L ... -o ...,wchan` 으로 멈춘 스레드의 대기 함수 확인). CPU 폭주는 실시간 변화를 봐야 하니 top(또는 `top -H`) 필요. 둘 다 내부적으로는 [`/proc`](#proc) 을 읽는 래퍼다.
- > 더 깊이: [이론 지식 §6 시스템 관제 도구 정리](이론_지식.md)
- **출처**: [top(1) — Linux manual page](https://man7.org/linux/man-pages/man1/top.1.html) · [ps(1) — Linux manual page](https://man7.org/linux/man-pages/man1/ps.1.html)

### `dmesg` {#dmesg}

- **정의**: 커널이 출력한 메시지를 **커널 링 버퍼(kernel ring buffer)** 에서 읽어 보여주는 명령어. 커널 코드는 `printk()` 로 메시지를 이 고정 크기 순환(circular) 버퍼에 쌓고, 가득 차면 가장 오래된 메시지부터 덮어쓴다. 따라서 부팅 이후의 하드웨어 변화·드라이버 로드·[OOM Killer](#oom-killer) 발동 같은 사건이 보이지만, 오래된 사건은 버퍼가 회전하며 사라질 수 있다. `dmesg -T` 로 사람이 읽는 타임스탬프, `dmesg -w` 로 실시간 추적.
- **비유**: 빌딩 관리실의 로그북(다만 페이지 수가 고정이라 꽉 차면 앞장부터 지워진다). 정전·화재경보·승강기 고장 같은 큰 사건이 기록됨.
- **왜 중요한가**: 앱 로그에 안 나오는 시스템 레벨 사건(특히 커널 OOM Killer 의 `Out of memory: Killed process ...` 줄)이 dmesg 에만 있을 수 있다. 영구 보존이 필요하면 [`journalctl -k`](#systemd-tools) 가 더 낫다.
- **출처**: [dmesg(1) — Linux manual page](https://man7.org/linux/man-pages/man1/dmesg.1.html)

### `vmstat` {#vmstat}

- **정의**: 프로세스·가상 메모리·페이징·블록 I/O·CPU 활동 통계를 보고하는 도구. `vmstat 1` 은 매초 한 줄씩 갱신한다. 단, 인자 없이 한 번만 호출하면 첫 줄은 *부팅 이후 누적 평균* 이라 순간값으로 착각하면 안 된다 — 둘째 줄부터가 실제 구간값이다. 핵심 컬럼: `si`(swap-in, 디스크→메모리로 초당 읽어들인 양), `so`(swap-out, 메모리→스왑으로 초당 쓴 양).
- **비유**: 자동차 계기판. 속도·연료·rpm 을 동시에 한 화면에서 모니터링.
- **왜 중요한가**: OOM 직전에 `si`/`so` 가 0 에서 폭증하면 [쓰래싱](#swap-thrashing)의 결정적 증거 — 시스템이 디스크 스왑에 매달려 실질 처리량이 무너지는 신호다.
- > 더 깊이: [이론 지식 §6 시스템 관제 도구 정리](이론_지식.md)
- **출처**: [vmstat(8) — Linux manual page](https://man7.org/linux/man-pages/man8/vmstat.8.html)

### 포트 · localhost · 0.0.0.0 {#port-localhost}

- **정의**:
  - **포트 (Port)**: 한 컴퓨터 안의 *프로그램별 통신 창구* 번호. TCP/UDP 가 쓰는 **16비트** 값이라 범위는 0~65535. IANA 는 0~1023 을 well-known(시스템) 포트, 1024~49151 을 registered, 49152~65535 를 dynamic/ephemeral 로 구분한다.
  - **localhost (127.0.0.1)**: "내 컴퓨터" 를 가리키는 루프백 주소. 패킷이 NIC 밖으로 *나가지 않고* 커널 안에서 되돌아온다. 정확히는 `127.0.0.0/8` 블록 전체가 루프백으로 예약돼 있고 127.0.0.1 이 그 대표 주소다.
  - **0.0.0.0**: 서버가 바인딩할 때 "이 호스트의 *모든* IPv4 인터페이스에서 들어오는 요청을 받겠다" 는 와일드카드 주소. 외부 접속도 허용된다(반대로 127.0.0.1 에만 바인딩하면 외부에서 접속 불가).
- **비유**: 아파트 호수. IP 주소 = 동, 포트 = 호수. 한 동(컴퓨터)에 여러 프로그램이 살고 각자 다른 호수(포트)에서 손님을 받는다.
- **왜 중요한가**: `agent-leak-app` 이 `0.0.0.0:15034` 에 바인딩되어 있다면 그 포트로 `curl` 을 쏘면 응답해야 정상 — [데드락](#coffman) 이면 timeout. `ss -tlnp` 또는 `netstat -tlnp` 로 어떤 프로세스가 어떤 포트를 LISTEN 중인지 확인.
- **출처**: [RFC 6335 — IANA Procedures for the Port Number Registry](https://www.rfc-editor.org/rfc/rfc6335.html) · [Reserved IP addresses — Wikipedia](https://en.wikipedia.org/wiki/Reserved_IP_addresses)

### HTTP · `curl` · `wget` {#http-curl}

- **정의**:
  - **HTTP (HyperText Transfer Protocol)**: 웹 서버와 클라이언트가 텍스트로 주고받는 표준 양식. `GET /path` 같은 요청 메서드와 3자리 상태 코드 응답(`2xx` 성공, `4xx` 클라이언트 오류, `5xx` 서버 오류 — 예 `200 OK`, `404 Not Found`, `500 Internal Server Error`)으로 이뤄진다.
  - **curl**: 명령행에서 URL 로 데이터를 전송하는 도구(libcurl 기반). `curl --max-time 3 http://...` 로 타임아웃을 걸 수 있고, 기본 출력은 표준출력이다.
  - **wget**: GNU 프로젝트의 비대화형 다운로더. HTTP/HTTPS/FTP 를 지원하고 기본으로 파일을 저장하며, 재귀 다운로드·끊긴 전송 재개·백그라운드 다운로드에 강하다.
- **비유**: HTTP = 손님이 웨이터에게 주문하는 공통 양식. curl = 그 양식대로 주문서를 쓰는 도구.
- **왜 중요한가**: 외부 health check 의 표준 방법. `curl --max-time 3 http://0.0.0.0:15034` 가 timeout 된다면 *서버 프로세스가 살아있어도 응답을 못 한다* 는 외부 증거 — [데드락](#coffman) 분석의 6대 증거 중 하나(프로세스는 떠 있는데 소켓 응답만 멈춤).
- **출처**: [HTTP response status codes — MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status) · [curl(1) manual page](https://curl.se/docs/manpage.html) · [GNU Wget Manual — Overview](https://www.gnu.org/software/wget/manual/wget.html)

### grep · awk · sed {#text-tools}

- **정의**: 셸의 텍스트 처리 3대장. 셋 다 *줄 단위 스트림* 을 처리하며 정규표현식을 쓴다.
  - **grep**: 패턴에 맞는 *줄* 만 골라냄. 이름 자체가 ed 편집기 명령 `g/re/p`(global / regular expression / print)에서 왔다. `grep -i "error" app.log`
  - **awk**: 입력을 *레코드(줄)와 필드(컬럼)* 로 쪼개 패턴-액션 규칙으로 추출/계산하는 데이터 처리 언어. `awk '{print $2, $5}'` 는 2·5번째 컬럼.
  - **sed**: 스트림 편집기. 줄 단위로 *치환·삭제* 등을 수행. `sed 's/foo/bar/g'` 는 각 줄의 모든 foo→bar.
- **비유**: 책에서 grep = "특정 단어가 있는 페이지만 표시", awk = "각 페이지의 2번째 줄만 모음", sed = "이 단어를 저 단어로 일괄 교체".
- **왜 중요한가**: `monitor.log` 같은 텍스트 로그에서 원하는 정보만 뽑는 표준 도구. `tail -f app.log | grep CRITICAL` 처럼 [파이프](#pipe-redirect)와 결합해서 라이브 모니터링도 한다.
- **출처**: [grep(1) — Linux manual page](https://man7.org/linux/man-pages/man1/grep.1.html) · [The GNU Awk User's Guide](https://www.gnu.org/software/gawk/manual/gawk.html) · [sed, a stream editor — GNU sed Manual](https://www.gnu.org/software/sed/manual/sed.html)

### `systemctl` · `journalctl` {#systemd-tools}

- **정의**: systemd(리눅스의 표준 init 시스템, PID 1) 관리 도구.
  - **systemctl**: 서비스([데몬](#daemon)) 유닛의 시작/정지/상태 조회 및 부팅 활성화 제어. `systemctl status sshd`, `systemctl restart nginx`, `systemctl enable agent-leak`.
  - **journalctl**: systemd 의 로깅 데몬(`systemd-journald`)이 한곳에 모은 구조화 로그 조회. `journalctl -k`(커널만), `journalctl -u sshd`(특정 서비스), `journalctl --since "5 min ago"`. 기본은 재부팅 시 휘발(`/run`)이지만 `/var/log/journal/` 이 있으면 영구 저장된다.
- **비유**: systemctl = 빌딩 자동제어 패널(시설별 on/off). journalctl = 그 빌딩의 전자 로그북.
- **왜 중요한가**: 데몬 미션에서 `systemctl status` 로 살아 있는지 보고, `journalctl -k | grep -i oom` 로 [커널 OOM Killer](#oom-killer) 흔적을 확인. [`dmesg`](#dmesg)와 비슷하지만 영구 저장·서비스/시간 단위 필터링이 강하다.

- **출처**: [systemctl(1) — freedesktop.org](https://www.freedesktop.org/software/systemd/man/latest/systemctl.html) · [journalctl(1) — freedesktop.org](https://www.freedesktop.org/software/systemd/man/latest/journalctl.html)

### 데드락 vs 라이브락 vs 기아

| 현상 | 정의 | 진단 |
| --- | --- | --- |
| [데드락](#coffman) | 서로의 자원을 기다리며 영원히 정지 | CPU 0%, [wchan](#wchan) = futex_*, 로그 멈춤 |
| 라이브락 | 서로 양보만 하느라 진척이 없음 (실제로는 CPU 를 쓰며 도는데 결과 없음) | CPU 높음, 로그는 계속 찍히는데 같은 패턴 반복 |
| [기아 (Starvation)](#starvation) | 특정 스레드만 영영 자원을 못 받음 | 한 스레드의 진행률만 멈춰 있음, 다른 건 정상 |

### 시스템 호출 추적 도구

| 도구 | 무엇을 보여주나 |
| --- | --- |
| `strace -f -p $PID` | 그 프로세스(+자식들)가 *지금* 호출하는 [시스템 콜](#syscall)을 실시간으로 출력. 데드락 시 모든 스레드가 `futex(...)` 에서 멈춰 있는 모습이 직접 보인다. |
| `ltrace -p $PID` | 라이브러리 함수 호출을 추적. `pthread_mutex_lock` 같은 함수가 어디서 멈췄는지 보임. |
| `perf top` | CPU 가 어느 함수에서 시간을 가장 많이 쓰는지 실시간 프로파일. CPU 스파이크 원인 후보를 좁혀준다. |
| `pidstat -t 1` | 스레드 단위 CPU/메모리 초당 변화. `top -H` 보다 스크립트 친화적. |
