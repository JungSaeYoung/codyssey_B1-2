## 1. 컴퓨터 구조 기초

### 시스템 콜 (System Call) {#syscall}

- **정의**: 일반 프로그램(앱)이 운영체제(OS)에게 "이 일 좀 대신 해 줘" 라고 부탁하는 공식 통로. `malloc` 으로 메모리 달라고 하기, `open` 으로 파일 열기, `read/write` 로 디스크 읽기/쓰기 등 — 결국 다 시스템 콜이다.
- **비유**: 식당에서 손님(앱)이 주방(OS)에 "물 좀 더 주세요" 라고 말하는 행위. 손님이 주방에 직접 들어가지는 않고, 정해진 호출 창구로만 요청한다.
- **왜 중요한가**: 모든 자원(메모리·CPU 시간·디스크·네트워크)은 OS 가 관리하므로, 앱이 무슨 사고를 치든 그 흔적은 결국 시스템 콜 어딘가에 남는다. 장애 분석의 출발점이 된다.

### 운영체제 · 커널 · 유저 공간 {#os-kernel-userspace}

- **정의**:
  - **OS (운영체제)**: 컴퓨터의 모든 자원(CPU·메모리·디스크·네트워크)을 관리하고, 앱들이 안전하게 공유하도록 중재하는 거대한 프로그램. 리눅스·윈도우·macOS 가 OS 다.
  - **커널 (Kernel)**: OS 의 핵심 부분으로, 가장 강한 권한을 가지고 하드웨어를 직접 다루는 코드. "리눅스 커널" 이라고 부르는 그것.
  - **유저 공간 (User Space)**: 일반 앱이 실행되는 영역. 권한이 약해서 하드웨어를 직접 못 만지고, [시스템 콜](#syscall)로 커널에 부탁해야 한다.
- **비유**: 운영체제 = 빌딩 전체 시스템. 커널 = 시설관리실(전기·수도·엘리베이터 직접 통제). 유저 공간 = 사무실(직원이 일하지만 시설은 못 만짐).
- **왜 중요한가**: "내가 짠 앱" 은 유저 공간에서 돈다. 메모리 할당·스레드 생성·파일 쓰기 등은 다 커널에 시스템 콜로 부탁하는 거다. 그래서 *앱 로그* 와 *커널 로그(dmesg)* 두 군데를 다 봐야 장애 전모를 본다.

### 폰 노이만 구조 (von Neumann architecture) {#von-neumann}

- **정의**: 1945년 수학자 John von Neumann 이 정리한 컴퓨터 모델로, "코드와 데이터를 같은 메모리에 두고, CPU 가 그 메모리에서 한 단어씩 가져와 실행한다" 는 설계.
- **비유**: 요리사(CPU)와 레시피 + 재료를 같은 책상(메모리) 위에 올려놓고, 요리사가 책상에서 한 줄씩 레시피를 읽으며 재료를 가져다 쓰는 모습.
- **왜 중요한가**: 우리가 지금 쓰는 거의 모든 컴퓨터(노트북·서버·스마트폰)가 이 모델을 따르므로, "코드가 곧 메모리이고, 메모리가 모자라면 코드 실행도 멈춘다" 는 사실이 자연스러워진다. OOM 이 왜 시스템 마비로 이어지는지의 근본 원인.

### CPU (Central Processing Unit) {#cpu}

- **정의**: 컴퓨터의 "두뇌". 메모리에 적힌 명령어를 하나씩 꺼내 실행하는 하드웨어 칩.
- **비유**: 식당의 요리사. 주문(명령어)을 받아 정해진 동작(더하기·곱하기·메모리 읽기 등)을 수행한다.
- **왜 중요한가**: CPU 가 100% 점유되면 다른 작업이 못 들어간다. 이번 미션의 CPU Spike 가 그 결과다.

### 스루풋 (Throughput) vs 지연시간 (Latency) {#throughput-vs-latency}

- **정의**:
  - **스루풋**: 단위 시간당 처리 가능한 작업량 (예: 초당 요청 1만 건)
  - **지연시간**: 작업 하나가 끝나는 데 걸리는 시간 (예: 요청 한 건당 5ms)
- **비유**: 식당에서 스루풋 = "1시간에 손님 몇 명 받나", 지연시간 = "손님 한 명이 음식 받기까지 얼마나 걸리나".
- **왜 중요한가**: 둘이 다른 개념인데 흔히 헷갈린다. CPU 코어가 많아지면 스루풋이 늘지만 한 작업의 지연시간은 그대로일 수 있다.

### 시간 단위 (ns / µs / ms / s) {#time-units}

- **정의**:
  - 1 초 (s) = 1,000 밀리초 (ms)
  - 1 밀리초 (ms) = 1,000 마이크로초 (µs)
  - 1 마이크로초 (µs) = 1,000 나노초 (ns)
  - 즉 1 초 = 1,000,000,000 나노초.
- **비유**: 1초가 1년이라면, 1ms = 8시간, 1µs = 30초, 1ns = 0.03초.
- **왜 중요한가**: 레지스터(0.3ns)와 HDD(10ms)의 차이는 **3300만 배**다. 작은 단위라고 무시하면 안 된다.

### 동시성 (Concurrency) vs 병렬성 (Parallelism) {#concurrency-vs-parallelism}

- **정의**:
  - **동시성**: 여러 일이 진행 중인 *것처럼* 보이는 상태. 코어가 1개여도 [스케줄러](#scheduler)가 짧은 시간 단위로 [스레드](#thread)들을 번갈아 돌리면 *겉으로는* 동시 같다.
  - **병렬성**: *진짜로* 같은 시각에 여러 일이 실행되는 상태. 멀티코어에서만 가능.
- **비유**: 동시성 = 1명의 요리사가 라면·김밥·떡볶이를 30초마다 번갈아 만들기. 병렬성 = 요리사 3명이 진짜 동시에 만들기.
- **왜 중요한가**: 데드락은 *동시성* 이 만드는 문제이지 *병렬성* 이 만드는 게 아니다. 1 코어 머신에서도 멀티스레드면 데드락 가능. 집합 관계는 동시성 ⊃ 병렬성.

### CPU-bound vs I/O-bound {#cpu-vs-io-bound}

- **정의**:
  - **CPU-bound 작업**: 대부분의 시간을 *계산* 에 쓰는 작업. 이미지 처리, 압축, 암호화, 수치 시뮬레이션.
  - **I/O-bound 작업**: 대부분의 시간을 *기다림* 에 쓰는 작업. HTTP 요청, DB 쿼리, 파일 읽기/쓰기, 네트워크 송수신.
- **비유**: CPU-bound = 시험 문제를 계속 푸는 학생. I/O-bound = 식당에서 주문하고 음식 나오기를 기다리는 손님.
- **왜 중요한가**: 어느 쪽인지에 따라 최적의 병렬화 전략이 다르다. CPU-bound 는 [멀티프로세싱](#py-multiprocessing) 이 답이고, I/O-bound 는 [asyncio](#asyncio) 나 멀티스레드가 답. Python 의 [GIL](#gil) 은 CPU-bound 멀티스레드를 막지만 I/O-bound 는 허용한다.

### bit · byte · 이진수 (binary) {#bit-byte}

- **정의**:
  - **bit (비트)**: 0 또는 1 — 컴퓨터의 가장 작은 정보 단위.
  - **byte (바이트)**: 8 bit 묶음. 영어 한 글자(ASCII)가 1 byte.
  - **이진수 (binary)**: 0/1 만으로 모든 숫자·문자·이미지·동영상을 표현하는 방식.
- **비유**: bit = 전기 스위치 1개(on/off). byte = 스위치 8개 묶음. 컴퓨터는 결국 스위치 수십억 개로 모든 일을 한다.
- **왜 중요한가**: KB·MB·GB 모두 byte 의 배수. `top`, `ps`, `free` 가 보여주는 모든 메모리 수치는 결국 byte 단위다. 4 KB = 4096 byte 라는 사실이 [페이지](#page) 크기와도 연결된다.

## 1.5 셸 · 터미널 · 로그 (비전공자 보충)

이 섹션은 미션 도중 마주치는 *명령행 환경* 의 기본기를 비전공자 관점에서 한 번 정리한다.

### 셸 (Shell) · bash · zsh {#shell}

- **정의**: 사용자가 키보드로 친 명령어를 받아 OS 에 전달·실행하는 *프로그램*. 리눅스 기본은 `bash`, macOS 최신은 `zsh`. 명령어를 해석하고, [환경변수](#env-var)를 관리하고, [파이프](#pipe-redirect)를 묶어주는 역할을 한다.
- **비유**: 호텔 컨시어지. "5층 가게 해 주세요" 같은 사람의 요청을 받아 엘리베이터·청소·룸서비스 같은 실제 시스템에 전달.
- **왜 중요한가**: 우리가 치는 `ps`, `top`, `curl`, `kill` 모두 셸이 받아서 [fork+exec](#fork-exec) 로 새 [프로세스](#process)를 만들어 돌린다. 셸 자체도 하나의 프로세스([PID](#pid) 있음).

### 터미널 (Terminal) {#terminal}

- **정의**: 셸을 띄우는 *창*. macOS Terminal, Windows Terminal, GNOME Terminal 등이 모두 터미널 *에뮬레이터* — 옛날의 물리 단말기(VT100 등) 를 화면으로 흉내낸 것.
- **비유**: 셸이 컨시어지라면 터미널은 그 컨시어지가 앉아있는 *데스크*. SSH 로 원격 서버에 접속하는 것은 "다른 데스크에서 같은 컨시어지(셸)와 대화" 하는 것.
- **왜 중요한가**: 터미널은 셸을 *보여주는 도구* 일 뿐 셸 자체는 아니다. 둘을 구분해야 "터미널을 닫으면 왜 [`nohup`](#daemon) 없이는 프로세스도 죽나" 같은 질문에 답할 수 있다.

### CLI vs GUI {#cli-gui}

- **정의**:
  - **CLI (Command Line Interface)**: 텍스트 명령어로 컴퓨터를 다루는 방식. [셸](#shell)이 대표.
  - **GUI (Graphical User Interface)**: 아이콘·창·마우스로 다루는 방식. 윈도우 탐색기, macOS Finder.
- **비유**: CLI = 점원에게 말로 주문. GUI = 메뉴판 사진 보고 손가락으로 가리키기.
- **왜 중요한가**: 서버 운영·자동화·스크립트는 거의 CLI. 이번 미션의 모든 도구(`ps`, `top`, `vmstat`, `dmesg`, `curl`) 가 CLI 다.

### 표준 입출력 (stdin · stdout · stderr) {#std-streams}

- **정의**: 모든 [프로세스](#process)가 시작할 때 자동으로 받는 3개의 데이터 통로. 각각 [파일 디스크립터](#file-descriptor) 0/1/2 번을 차지.
  - **stdin (표준 입력, FD 0)**: 키보드 또는 다른 명령의 출력에서 *들어오는* 데이터.
  - **stdout (표준 출력, FD 1)**: 화면으로 보내는 일반 출력.
  - **stderr (표준 에러, FD 2)**: 화면으로 보내는 에러·경고. stdout 과 *분리되어 있어* 따로 잡을 수 있다.
- **비유**: 식당 카운터에 창구가 3개 — 주문(stdin), 음식(stdout), 컴플레인(stderr).
- **왜 중요한가**: `command > out.log 2> err.log` 처럼 정상 출력과 에러를 분리할 수 있는 건 둘이 다른 FD 라서 가능. monitor.sh 같은 로깅 스크립트가 자주 쓴다.

### 파이프 · 리다이렉트 (`|`, `>`, `>>`, `<`) {#pipe-redirect}

- **정의**:
  - **파이프 (`|`)**: 왼쪽 명령의 [stdout](#std-streams) 을 오른쪽 명령의 [stdin](#std-streams) 으로 *흘려보낸다*. `ps -ef | grep python` 처럼.
  - **리다이렉트**: `>` 는 stdout 을 *파일로 덮어쓰기*, `>>` 는 *이어쓰기*, `<` 는 파일을 stdin 으로 *읽기*, `2>` 는 stderr 만 따로.
- **비유**: 파이프 = 한 요리사가 만든 반찬을 옆 요리사 도마로 바로 올리는 컨베이어 벨트. 리다이렉트 = 다 만든 음식을 그릇(파일)에 담아 보관.
- **왜 중요한가**: 셸 스크립트의 거의 모든 표현력이 이 두 가지에서 나온다. monitor.log 같은 결과 파일도 결국 리다이렉트의 산물이다. [IPC](#ipc) 의 가장 단순한 형태이기도 함.

### 파일 디스크립터 (File Descriptor, FD) {#file-descriptor}

- **정의**: [프로세스](#process)가 열어 둔 파일·소켓·파이프에 OS 가 붙이는 작은 정수 번호. 0/1/2 는 [stdin/stdout/stderr](#std-streams) 가 차지하고, 새로 연 것은 3, 4, 5… 로 받음. `ls -l /proc/$PID/fd/` 로 그 프로세스가 지금 무엇을 열고 있는지 다 볼 수 있다.
- **비유**: 도서관에서 빌린 책마다 받는 번호표. 책 자체가 아니라 번호표만 들고 다닌다.
- **왜 중요한가**: 멀티스레드에서 FD 는 *공유* 자원이라 한 [스레드](#thread)가 닫으면 모두에게 영향. "Too many open files" 에러는 한 프로세스가 가질 수 있는 FD 수가 한계(`ulimit -n`)에 닿았다는 뜻.

### exit code (종료 코드) · `$?` {#exit-code}

- **정의**: 명령어가 끝날 때 OS 에 반환하는 0~255 정수. 관례: **0 = 성공, 그 외 = 실패**. 직전 명령의 종료 코드는 셸 변수 `$?` 에 담긴다.
- **비유**: 시험지를 내고 받는 점수. 0 점이 *만점*(다 풀었다 = 성공), 다른 숫자는 어디서 막혔는지 코드.
- **왜 중요한가**: 셸의 `if`, `&&`, `||` 가 모두 exit code 로 판단. [시그널](#signal)로 죽으면 `128 + 시그널번호` 가 된다 — SIGKILL(9) → 137, SIGTERM(15) → 143. 로그에 `Exited with 137` 가 보이면 *누가 SIGKILL 을 보냈다* 의 강력한 단서.

### root · sudo · 권한 (Permission) {#root-sudo}

- **정의**:
  - **root (= superuser, uid 0)**: 시스템 *모든 권한* 을 가진 특수 계정. 어떤 파일이든 읽고/쓰고, 어떤 프로세스든 [SIGKILL](#signal) 할 수 있음.
  - **sudo**: 일반 사용자가 *한 명령만 일시적으로* root 권한으로 실행하는 도구. `sudo apt install ...` 같은 식.
  - **권한**: 파일·디렉토리에 붙은 *r(읽기) · w(쓰기) · x(실행)* 표시. `ls -l` 의 `rwxr-xr-x` 9 칸이 (소유자·그룹·기타) × (r/w/x).
- **비유**: 빌딩 마스터키(root) vs 직원 카드(일반 사용자). 마스터키는 모든 문을 열지만, 직원 카드는 자기 사무실만. 청소부가 잠깐 마스터키를 빌리는 것 = sudo.
- **왜 중요한가**: `dmesg`, 남의 프로세스 [`kill`](#signal), `/proc/<남의 pid>/...` 등 시스템 깊은 곳에 닿으려면 root 가 필요. 보안 원칙: *기본은 일반 사용자, 필요할 때만 sudo*.

### 로그 · 로그 레벨 {#log-level}

- **정의**: 프로그램이 동작 중 남기는 텍스트 기록. 보통 5단계 레벨로 분류:
  - **DEBUG**: 개발 중 추적용 (운영에선 보통 꺼둠)
  - **INFO**: 정상 진행 상황 알림
  - **WARNING**: 잘못될 수 있는 상황
  - **ERROR**: 일부 기능 실패
  - **CRITICAL** (= FATAL): 시스템 자체가 위험 — 곧 종료/장애
- **비유**: 일기. 평범한 날(INFO), 좀 이상한 날(WARNING), 사고 난 날(ERROR), 큰 사건(CRITICAL).
- **왜 중요한가**: `MemoryGuard` 가 띄우는 `[CRITICAL]` 은 *가장 높은 위험 레벨* 이므로 장애 1차 단서가 된다. `grep CRITICAL` 만 해도 평소 안 보이던 사건만 솎아낼 수 있다.

## 2. 저장 계층 (Storage Hierarchy)

### 레지스터 (Register) {#register}

- **정의**: CPU 칩 *안에* 있는 가장 빠른 저장 공간. 보통 64비트짜리가 십수 개 있다.
- **비유**: 요리사의 양손. 지금 이 순간 만지고 있는 재료 한두 개만 들 수 있다.
- **왜 중요한가**: CPU 가 진짜로 계산하는 건 레지스터 위에서다. 메모리에서 값을 꺼내 레지스터에 올리고, 계산하고, 다시 메모리에 적는 순환을 끝없이 반복한다.

### 캐시 · L1 / L2 / L3 (Cache) {#cache}

- **정의**: CPU 와 DRAM 사이에 끼워 놓은 작은 고속 메모리. L1 이 가장 작고(수십 KB) 빠르고, L3 가 가장 크고(수십 MB) 느리다.
- **비유**: 책상 서랍 3단. 1단 서랍(L1)은 손 뻗으면 닿지만 작고, 3단(L3)은 약간 멀지만 더 많이 들어간다. 책장([DRAM](#dram))까지 가는 건 더 멀다.
- **왜 중요한가**: 같은 데이터를 반복해 쓰면 캐시 덕분에 무지 빠르다. 반대로 메모리 누수로 캐시가 자주 갈리면 (= "캐시 미스") CPU 가 매번 DRAM 까지 가야 하니 느려진다.

### DRAM (= RAM) {#dram}

- **정의**: 우리가 흔히 "메모리" 라 부르는 그것. 전원이 꺼지면 내용이 사라지는 휘발성 저장소.
- **비유**: 책장. 책상보다는 크지만 멀고, 정전되면 책이 다 사라지는 마법의 책장.
- **왜 중요한가**: 모든 실행 중인 프로그램과 데이터가 여기에 있다. `free`, `%MEM` 이 가리키는 게 바로 DRAM 사용량. 가득 차면 OS 가 디스크로 내쫓기 시작한다.

### 디스크 · SSD · HDD {#disk}

- **정의**:
  - **HDD (Hard Disk Drive)**: 회전판에 자기로 데이터를 적는 전통적 디스크. 느림.
  - **SSD (Solid State Drive)**: 플래시 메모리 기반. HDD 보다 약 100배 빠름.
  - 둘 다 전원이 꺼져도 내용이 남는 비휘발성 저장소.
- **비유**: 도서관. 책장에 없는 책을 찾으러 가는 곳. SSD 는 가까운 도서관, HDD 는 다른 도시 도서관.
- **왜 중요한가**: [DRAM](#dram) 이 다 차면 OS 가 페이지를 SSD/HDD 의 스왑 영역으로 옮긴다. 옮긴 페이지가 다시 필요하면 SSD/HDD 에서 읽어와야 하니 갑자기 100~10000배 느려진다 (= [쓰래싱](#swap-thrashing)).

### 페이지 교체 · 스왑 · 쓰래싱 {#swap-thrashing}

- **정의**:
  - **페이지 교체**: [DRAM](#dram) 이 모자라면 OS 가 자주 안 쓰는 [페이지](#page)(4KB 덩어리)를 골라 디스크로 옮기는 행위.
  - **스왑 (Swap)**: 그렇게 디스크로 옮겨두는 전용 영역. 리눅스에서는 swap 파일/파티션.
  - **쓰래싱 (Thrashing)**: 페이지를 디스크로 옮기자마자 다시 필요해져서, "DRAM ↔ 디스크" 왕복만 반복하며 진짜 일은 못 하는 상태.
- **비유**: 책상이 좁아서 안 쓰는 책을 도서관에 반납했는데, 반납하자마자 그 책이 또 필요해져서 다시 빌리러 가는 사람. 결국 책상에서 공부는 못 하고 도서관만 왔다갔다 한다.
- **왜 중요한가**: 쓰래싱이 시작되면 *겉으로는* "메모리는 아직 100%는 아닌데 CPU 도 멀쩡한데 시스템이 멈췄다" 처럼 보인다. monitor.log 의 MEM 이 95% 근처에서 CPU 도 같이 치솟는다면 쓰래싱 의심.

## 3. 프로세스와 메모리 구조

### 프로세스 (Process) {#process}

- **정의**: 실행 중인 프로그램 한 인스턴스. 디스크에 있는 실행 파일을 OS 가 메모리에 올려서 돌리면, 그 살아 있는 단위 하나가 프로세스.
- **비유**: 책장에 꽂힌 요리책(실행 파일) vs 그 레시피로 지금 요리하고 있는 요리사 한 명(프로세스). 같은 책으로 두 요리사가 동시에 요리하면 프로세스 2개.
- **왜 중요한가**: `ps`, `top`, `kill <PID>` 등 우리가 다루는 모든 단위가 프로세스다. 각 프로세스는 자기만의 [가상 주소 공간](#virtual-address-space)을 가진다.

### PID (Process ID) {#pid}

- **정의**: 실행 중인 모든 [프로세스](#process)에 OS 가 붙이는 고유 번호.
- **비유**: 식당 손님의 대기번호. 같은 번호는 같은 시각에 두 명 못 받는다.
- **왜 중요한가**: 데드락 미션에서 "PID 가 살아있다(`ps -ef | grep`)" 가 1차 증거다. 프로세스가 죽었으면 PID 도 사라진다.

### PPID · 부모-자식 프로세스 {#parent-child}

- **정의**:
  - **PPID (Parent PID)**: 어떤 [프로세스](#process)를 *만든* 부모 프로세스의 [PID](#pid). `ps -o pid,ppid,comm` 으로 확인.
  - 모든 프로세스는 [fork](#fork-exec) 로 만들어지므로 *반드시* 부모가 있다. 시스템 부팅 직후 만들어지는 `init` 또는 `systemd`(PID 1) 가 모든 프로세스의 시조.
  - 자식이 끝나면 부모가 `wait()` 시스템 콜로 회수해야 자식의 [PCB](#pcb) 가 깨끗이 정리된다.
- **비유**: 가족 관계도 — 모든 사람에게 부모가 있고, 거슬러 올라가면 한 시조에 닿는다.
- **왜 중요한가**: 부모-자식 관계는 [좀비/고아](#zombie-orphan) 를 이해하는 토대이며, 셸이 `nohup` 으로 자식을 *부모와 분리* 해 살려두는 것도 이 관계를 끊는 행위다.

### 좀비 · 고아 프로세스 {#zombie-orphan}

- **정의**:
  - **좀비 (Zombie)**: 자식이 종료됐는데 [부모](#parent-child)가 `wait()` 으로 회수하지 않아 [PCB](#pcb) 만 남은 프로세스. [`STAT=Z`](#stat-codes). CPU·메모리는 거의 0 이지만 [PID](#pid) 슬롯을 차지한다.
  - **고아 (Orphan)**: 부모가 *먼저* 죽어서 [PPID](#parent-child)가 1번(init/systemd)으로 재지정된 자식. 사고는 아니며 정상 동작.
- **비유**: 좀비 = 출석부에는 있는데 졸업 처리가 안 된 학생(자리 차지). 고아 = 담임이 바뀐 학생(여전히 학생).
- **왜 중요한가**: 좀비가 쌓이면 시스템 PID 가 고갈된다. ⚠ 단, 이번 데드락 미션의 *"살아있는 좀비"* 표현은 진짜 좀비(STAT=Z)가 아니라 *비유적* 표현 — 정상 작동이 멈춘 채 PID 만 유지되는 데드락 프로세스를 말한다.

### 데몬 (Daemon) · 서비스 {#daemon}

- **정의**: 부팅 시 자동으로 떠서 *백그라운드로 계속 돌면서* 어떤 서비스를 제공하는 [프로세스](#process). 이름 끝에 `d` 가 붙는 관례(`sshd`, `httpd`, `crond`, `systemd`). 보통 터미널과 분리되어 부모가 init 인 [고아](#zombie-orphan) 가 되도록 만들어진다.
- **비유**: 24시간 편의점 점원. 손님이 오든 안 오든 자리를 지키고 있다가 누가 들어오면 응대.
- **왜 중요한가**: 리눅스의 거의 모든 시스템 서비스(웹서버·DB·SSH·시간동기화)가 데몬으로 실행된다. `agent-leak-app` 도 일종의 데몬처럼 종일 동작하다 정책 위반 시 자체 종료. `nohup ... &` 로 일반 명령을 데몬화할 수 있다.

### 가상 주소 공간 (Virtual Address Space) {#virtual-address-space}

- **정의**: [프로세스](#process)마다 OS 가 만들어주는 "자기 전용 메모리 환상". 실제 [DRAM](#dram) 의 어디에 있든, 프로세스는 0번지부터 거대한 연속 공간을 자기가 다 쓰는 것처럼 느낀다.
- **비유**: 호텔 객실. 모든 객실은 "방번호 1번" 부터 시작하는 듯한 도면을 갖지만, 실제로는 빌딩의 다른 층, 다른 위치에 있다. 옆방의 도면을 봐도 자기 방 안은 못 본다(메모리 격리).
- **왜 중요한가**: 가상 주소 공간 덕분에 (1) 프로세스 간 격리 (한 앱 버그가 다른 앱 메모리를 못 망침), (2) DRAM 보다 큰 메모리도 마치 있는 것처럼 쓸 수 있음(스왑 활용) 이 가능하다.

### Heap · Stack · Data · BSS · Text {#memory-segments}

- **정의 (한 줄씩)**:
  - **Text (Code)**: 프로그램의 기계어 명령. 읽기 전용.
  - **Data**: `int x = 5;` 처럼 초기값 있는 전역/정적 변수.
  - **BSS**: `int y;` 처럼 초기값 없는 전역/정적 변수. (0 으로 자동 초기화)
  - **Heap (힙)**: 프로그램이 실행 중에 동적으로 할당받는 영역. `malloc()`, `new`, Python 객체 생성 등.
  - **Stack (스택)**: 함수가 호출될 때마다 새 칸이 위에 쌓이는 영역. 지역 변수·매개변수·반환 주소가 들어감.
- **비유**: 식당 주방의 구역들. Text = 메뉴판(못 바꿈), Data/BSS = 주방장이 미리 만들어둔 소스 통, Heap = 손님 주문대로 그때그때 만드는 음식 보관함(계속 변함), Stack = 지금 만드는 한 요리에 쓰는 임시 도마(만들고 치움).
- **왜 중요한가**: 메모리 누수는 *거의 항상* Heap 에서 일어난다. Stack 은 함수 끝나면 자동 정리되니까. 그래서 누수 분석은 Heap 사용량(= [RSS](#vsz-vs-rss))추이만 봐도 충분.

### malloc / free / 가비지 컬렉터 (GC) {#malloc-free-gc}

- **정의**:
  - `malloc()`: C 언어에서 [Heap](#memory-segments)에 메모리 달라는 함수. ("malloc" = memory allocate)
  - `free()`: 다 쓴 메모리를 반납하는 함수. C 에서는 *손으로* 호출해야 함.
  - **GC (Garbage Collector)**: Python/Java 등 일부 언어가 자동으로 "이 객체 더 이상 쓸 사람 없네" 라고 판단해 메모리를 회수해주는 시스템.
- **비유**: 도서관에서 책을 빌리는 것(malloc) ↔ 반납하는 것(free). GC 는 "한 달 동안 안 본 책은 자동 반납됨" 같은 자동화 시스템.
- **왜 중요한가**: GC 가 있어도 *어딘가에서 그 객체를 참조하고 있으면* GC 는 못 치운다. 전역 리스트에 자꾸 append 하면서 안 비우면 GC 도 "이거 누가 쓰고 있는 거 같은데?" 하고 그냥 둔다 → 메모리 누수.

### ELF (Executable and Linkable Format) {#elf}

- **정의**: 리눅스의 표준 실행 파일 형식. 윈도우의 `.exe` 에 해당.
- **비유**: 실행 파일이 담긴 도시락 박스. 표지에 "이 도시락 안에 코드는 여기, 데이터는 여기" 식의 색인이 있어 OS 가 그걸 보고 메모리에 펼친다.
- **왜 중요한가**: `agent-leak-app` 도 ELF 다 (`file agent-leak-app` 으로 확인 가능). 부팅 시 로더가 ELF 의 [Text/Data 영역](#memory-segments)을 [가상 메모리](#virtual-address-space)에 펼친다.

### 페이지 · 4 KB {#page}

- **정의**: 가상 메모리 ↔ 물리 메모리 매핑을 관리하는 최소 단위. 거의 모든 현대 OS 에서 4096 바이트(4 KB).
- **비유**: 책의 한 페이지. 책을 한 글자 단위로 도서관에서 빌리고 반납하면 비효율적이니, "페이지 단위로 빌린다" 고 정한 셈.
- **왜 중요한가**: OS 는 메모리를 페이지 단위로 할당·회수·스왑한다. 1 바이트만 써도 4 KB 가 할당된다(= 내부 단편화). 메모리 누수도 결국 페이지 단위 누적.

### MMU (Memory Management Unit) {#mmu}

- **정의**: CPU 칩 안에 있는 하드웨어로, [프로세스](#process)가 보는 가상 주소를 실제 [DRAM](#dram) 의 물리 주소로 *매번* 자동 변환해주는 장치.
- **비유**: 호텔 객실 번호(가상) → 실제 빌딩 층/방 번호(물리) 를 즉시 알려주는 안내원.
- **왜 중요한가**: 이게 없다면 [가상 메모리](#virtual-address-space)도 없고, 프로세스 격리도 없고, 스왑도 없다. 모든 메모리 시스템의 토대.

### 페이지 테이블 · TLB {#page-table-tlb}

- **정의**:
  - **페이지 테이블**: "가상 페이지 N 번은 물리 페이지 M 번에 있다" 같은 매핑 정보가 적힌 거대한 표. 메모리에 있다.
  - **TLB (Translation Lookaside Buffer)**: 페이지 테이블 검색이 너무 자주 일어나니, [MMU](#mmu) 가 최근 매핑 결과를 작은 캐시에 저장. 캐시 히트면 빠르게 변환.
- **비유**: 페이지 테이블 = 호텔 객실 도면(분량이 크다). TLB = 안내원이 머리에 외워둔 최근 객실 위치 메모.
- **왜 중요한가**: 메모리를 새로 많이 할당하면 TLB 가 자주 갈리고(= miss), 변환만으로도 느려진다. 메모리 누수가 성능 저하를 가속하는 또 하나의 이유. [컨텍스트 스위치](#context-switch) 시 TLB 가 통째로 비워지므로 [멀티프로세싱](#py-multiprocessing) 이 [멀티스레딩](#pthread) 보다 비싼 이유이기도 하다.

### 페이지 폴트 (Page Fault) {#page-fault}

- **정의**: [프로세스](#process)가 어떤 가상 주소를 건드렸는데, 해당 [페이지](#page)가 아직 [DRAM](#dram) 에 없거나 매핑이 안 돼 있을 때 발생하는 이벤트. CPU 가 커널에게 "이 페이지 좀 채워줘" 라고 호출.
- **비유**: 호텔 객실 도면에 있는 방인데 실제로는 아직 가구가 없는 상태로 가니, 안내원이 즉시 가구를 옮겨와 채워주는 행위.
- **왜 중요한가**: 페이지 폴트는 *정상* 동작이다. 처음 할당받은 페이지에 처음 쓰는 순간 항상 발생. 하지만 너무 자주 발생하면(예: 스왑된 페이지를 다시 가져오는 경우) 시스템이 느려진다. [Copy-on-Write](#cow) 도 페이지 폴트 메커니즘 위에 얹혀 있다.

### lazy allocation (지연 할당) {#lazy-allocation}

- **정의**: [프로세스](#process)가 `malloc(1GB)` 라고 해도 OS 는 진짜 1GB 물리 메모리를 안 준다. *프로세스가 그 페이지를 실제로 읽거나 쓰는 순간* 에만 물리 페이지를 배정한다.
- **비유**: 식당에서 "10인분 예약" 했다고 음식을 미리 다 만들어두지 않는다. 손님이 와서 "이제 주세요" 할 때마다 한 사람분씩 만든다.
- **왜 중요한가**: [VSZ vs RSS](#vsz-vs-rss) 의 차이를 만드는 핵심 메커니즘. 누수 추적 시 *RSS* 만 봐야 하는 이유.

### VSZ vs RSS {#vsz-vs-rss}

- **정의**:
  - **VSZ (Virtual Size)**: [프로세스](#process)가 OS 에 "이만큼 줘" 라고 *요청한* 가상 메모리 총량 (KB 단위). [lazy allocation](#lazy-allocation) 때문에 실제 사용량은 더 작을 수 있다.
  - **RSS (Resident Set Size)**: 그 중에서 *지금 [DRAM](#dram) 에 진짜 올라와 있는* 양. [스왑](#swap-thrashing)으로 나간 페이지는 빠진다.
- **비유**: 호텔 예약(VSZ) vs 지금 실제 묵고 있는 손님 수(RSS). 1000명 예약해도 80명만 와 있으면 RSS 는 80명분.
- **왜 중요한가**: 메모리 누수의 가장 직접적 지표는 RSS 다. VSZ 는 늘었다 줄었다 안 하므로 누수 감지에 둔감하다.

### fork / exec {#fork-exec}

- **정의**:
  - **`fork()`**: 현재 [프로세스](#process) 를 *자기 복제* 하는 시스템 콜. 호출 한 번에 거의 똑같은 프로세스 두 개(부모·자식)가 생긴다. 메모리 상태·열린 파일·환경변수 모두 같지만 [PID](#pid) 만 다르다.
  - **`exec()`**: 현재 프로세스의 메모리 이미지를 *다른 프로그램으로 교체* 하는 시스템 콜. PID 는 유지되지만 코드는 완전히 새것.
- **비유**: fork = 나 자신을 복사해서 쌍둥이를 만든다. exec = 그 쌍둥이가 옷·기억·이름표를 한꺼번에 새것으로 바꾼다(PID 라는 사번만 유지).
- **왜 중요한가**: 셸이 명령어를 실행할 때마다 `fork → exec` 패턴을 쓴다. `bash` 가 `ls` 를 돌리려면 fork 로 자식을 만들고, 자식이 exec 로 ls 가 된다. 멀티프로세싱 구현의 가장 밑바닥.

### Copy-on-Write (COW) {#cow}

- **정의**: [fork](#fork-exec) 후에 부모와 자식이 같은 물리 페이지를 *공유* 하다가, 어느 한쪽이 페이지에 *쓰려는 순간* 비로소 페이지를 복사해 둘로 분리하는 최적화. 읽기만 하면 영원히 공유.
- **비유**: 같은 책을 둘이 같이 보다가, 한 명이 한 페이지에 메모를 적으려는 순간 그 페이지만 복사해 가져가는 방식.
- **왜 중요한가**: 1 GB 짜리 프로세스를 fork 해도 실제 메모리는 거의 안 늘어난다. 그래서 멀티프로세싱이 *생각보다* 메모리 효율적이다. 단, 시간이 흐르며 페이지가 분리되면 결국 메모리는 늘어난다.

### clone() 시스템 콜 {#clone-syscall}

- **정의**: 리눅스에서 [`fork`](#fork-exec) 도 [`pthread_create`](#pthread) 도 결국 모두 같은 `clone()` 시스템 콜의 다른 옵션이다. `CLONE_VM`·`CLONE_FS`·`CLONE_FILES`·`CLONE_SIGHAND` 등 플래그로 "메모리도 공유? 파일도 공유? 시그널 핸들러도 공유?" 를 결정.
- **비유**: 한 종류의 "분신술" 시스템에서 분신과 본체가 무엇을 공유할지 옵션으로 고를 수 있는 것. fork = 아무것도 공유 안 함. thread = 거의 다 공유.
- **왜 중요한가**: 리눅스에서 프로세스와 스레드의 경계가 *유연* 하다는 의미. 그래서 `ps -L` 이 스레드 별 정보를 보여줄 수 있다. 커널 입장에서는 같은 [`task_struct`](#pcb).

## 4. 종료 · 보호 정책 · 환경변수

### OOM Killer · `oom_score` {#oom-killer}

- **정의**:
  - **OOM Killer**: 시스템 전체 메모리가 부족해 더는 안 되겠다 싶을 때 리눅스 커널이 발동하는 비상 장치. 누군가 한 명을 죽여서 살아남는다.
  - **`oom_score`**: 커널이 각 프로세스에 매기는 "이 놈이 죽으면 메모리가 많이 회수될까" 점수. [RSS](#vsz-vs-rss) 가 큰 프로세스일수록 높음. `/proc/<pid>/oom_score` 에서 확인.
- **비유**: 응급 상황에서 함선 무게를 줄여야 할 때, 가장 무거운 짐 한 개를 골라 바다에 던지는 함장.
- **왜 중요한가**: OOM Killer 가 발동하면 *우리가 죽이고 싶지 않은 중요한 프로세스* 가 죽을 수 있다. 그래서 앱 내부에 MemoryGuard 같은 자체 보호 정책을 두는 게 일반적이다.

### 시그널 · SIGKILL · SIGTERM {#signal}

- **정의**: [프로세스](#process)에게 보내는 짧은 알림. 종류마다 번호와 의미가 다름.
  - **SIGTERM (15)**: "끝낼 준비해" — 프로세스가 받아서 정상 종료 절차(파일 닫기, 락 풀기 등) 수행 가능.
  - **SIGKILL (9)**: "지금 즉시 죽여" — 프로세스가 거부할 수도, 정리할 수도 없다. 무조건 즉사.
  - **SIGINT (2)**: Ctrl+C 가 보내는 시그널. SIGTERM 비슷한 정상 종료.
- **비유**: SIGTERM = "퇴근 준비하세요" / SIGKILL = "당장 나가" / SIGINT = "Ctrl+C 누른 사용자의 인내심 끝".
- **왜 중요한가**: MemoryGuard 는 보통 SIGKILL 을 쓰고, [Watchdog](#watchdog) 은 SIGTERM 을 쓰는 차이가 있다. 시그널이 뭐였는지를 알면 어떤 정책이 발동했는지 추론 가능.

### 환경변수 (Environment Variable) {#env-var}

- **정의**: 셸/프로세스가 자식 프로세스에게 전달하는 키=값 쌍. `export NAME=value` 로 설정하고, 프로그램에서 `getenv("NAME")` 로 읽음.
- **비유**: 신입사원이 출근할 때 받는 가이드. "사무실 전화번호는 X, 이메일은 Y" 같은 정보를 미리 알려준다.
- **왜 중요한가**: `agent-leak-app` 은 `MEMORY_LIMIT`, `CPU_MAX_OCCUPY` 같은 환경변수를 부팅 시 읽고, 이후 행동의 임계치로 쓴다. 환경변수를 바꾸면 코드 재컴파일 없이 정책이 바뀐다.

### Watchdog (감시견) {#watchdog}

- **정의**: 어떤 자원의 사용량을 주기적으로 감시하다가, 임계치를 넘으면 미리 정의된 보호 조치(주로 종료)를 발동하는 패턴. 하드웨어 워치독, OS 워치독, 앱 워치독 등 다양한 층위에서 존재.
- **비유**: 집의 누전 차단기. 전기를 너무 많이 쓰면 자동으로 차단해서 화재를 막는다.
- **왜 중요한가**: "내가 폭주하면 알아서 자살" 패턴은 시스템 전체를 지키는 방어선. 이번 미션의 핵심 정책 중 하나.

## 5. 스레드 · 스케줄링

### 스레드 (Thread) {#thread}

- **정의**: [프로세스](#process) 안에서 따로 실행되는 작업 단위. 같은 프로세스의 메모리·파일·코드를 공유하지만 자기만의 실행 흐름과 [Stack](#memory-segments)을 가짐.
- **비유**: 한 식당(프로세스) 안의 여러 요리사(스레드). 같은 주방과 재료(메모리)를 공유하면서 각자 다른 요리를 동시에 만든다.
- **왜 중요한가**: 멀티스레드는 빠르지만, 같은 자원을 동시에 만지면 충돌([race condition](#race-condition), [데드락](#coffman))이 생긴다. 이번 미션 3번째 장애의 원인.

### PCB · `task_struct` {#pcb}

- **정의**: 프로세스/스레드의 상태(레지스터 값, [PID](#pid), 메모리 정보, 열린 파일 목록 등)를 담아두는 OS 내부 자료구조. 리눅스에서는 `task_struct` 라는 C 구조체.
- **비유**: 식당 손님 카드. 손님 번호, 주문 내역, 결제 정보 등을 한 장에 모아둔 것.
- **왜 중요한가**: OS 가 [컨텍스트 스위치](#context-switch) (다음 스레드로 바꾸기)를 할 때 PCB 의 정보를 저장·복원한다.

### 컨텍스트 스위치 (Context Switch) {#context-switch}

- **정의**: CPU 가 지금 돌리던 [스레드](#thread)를 멈추고 다른 스레드로 바꾸는 행위. 현재 스레드의 모든 레지스터 값을 [PCB](#pcb) 에 저장하고, 새 스레드의 값을 PCB 에서 꺼내 레지스터에 채운다.
- **비유**: 요리사가 음식 만들기를 잠시 멈추고, 도마/칼/재료 위치를 메모지에 적어둔 뒤, 다른 요리를 시작. 나중에 돌아오면 그 메모지를 보고 정확히 멈춘 자리부터 재개.
- **왜 중요한가**: 컨텍스트 스위치는 공짜가 아니다(수십 µs 비용). 너무 자주 일어나면 정작 일하는 시간보다 바꾸는 시간이 길어진다. 또 *프로세스 간* 스위치는 [TLB](#page-table-tlb) 까지 비워야 해서 *스레드 간* 스위치보다 훨씬 비싸다.

### IPC (Inter-Process Communication) {#ipc}

- **정의**: 서로 격리된 [프로세스](#process)끼리 데이터를 주고받는 방법들. 파이프·소켓·공유 메모리·시그널 등.
- **비유**: 다른 사무실에 있는 동료에게 메모를 보내거나(파이프), 전화하거나(소켓), 공용 게시판에 적거나(공유 메모리), 사내 방송으로 알리는 것([시그널](#signal)).
- **왜 중요한가**: 멀티프로세스 설계에서 필수. 메모리가 격리돼 있어 변수 공유가 안 되므로 명시적 통신이 필요. [멀티스레딩](#pthread) 에 비해 통신이 *명시적이고 비싼* 게 멀티프로세싱의 트레이드오프.

### CPU 코어 · 하이퍼스레딩 {#core-smt}

- **정의**:
  - **코어**: CPU 칩 안에 들어 있는 독립적 처리 유닛. 코어 하나당 동시에 한 [스레드](#thread) 실행 가능.
  - **하이퍼스레딩 (Hyper-Threading, SMT)**: 한 물리 코어가 마치 2개 코어인 것처럼 보이게 하는 인텔/AMD 기술. 4코어 + HT = OS 가 보기엔 8 "논리 코어".
- **비유**: 코어 = 식당 요리사 인원. HT 는 한 명이 두 일을 번갈아 하는 "1.5인분" 효과.
- **왜 중요한가**: `top` 에서 %CPU 가 400% 찍혔다면 "4코어 다 쓰고 있다" 라는 뜻. 시스템 전체 사용률은 (총합 / 코어 수) 로 나눠서 봐야 함. [병렬성](#concurrency-vs-parallelism) 의 상한은 결국 코어 수.

### idle (유휴 시간) {#idle}

- **정의**: [CPU](#cpu) 가 할 일이 없어서 쉬고 있는 시간의 비율. `top` 의 `id` 컬럼.
- **비유**: 식당 요리사가 주문이 없어 그냥 서 있는 시간의 비율.
- **왜 중요한가**: CPU 사용률 = 100% - idle%. `monitor.sh` 가 사용률을 구하는 공식이 그것.

### 스케줄러 (Scheduler) {#scheduler}

- **정의**: 여러 [스레드](#thread)가 동시에 실행 가능할 때, "다음으로 어느 스레드에게 CPU 를 줄까" 를 결정하는 커널의 핵심 모듈.
- **비유**: 식당 매니저. 손님이 줄을 서면 누구를 어느 테이블에 보낼지 결정한다.
- **왜 중요한가**: 스케줄러 정책이 바뀌면 같은 코드도 다른 속도로 돌 수 있다. "왜 내 작업이 안 끝나지?" 의 답이 스케줄러에 있을 때가 많다.

### 타임 슬라이스 (Time Slice / Quantum) {#time-slice}

- **정의**: 한 [스레드](#thread)가 [CPU](#cpu) 를 잡고 있을 수 있는 최대 시간. 보통 수 ms 단위. 이 시간이 지나면 [스케줄러](#scheduler)가 강제로 다음 스레드로 바꾼다.
- **비유**: 회의실 예약 시간. 30분 슬롯이 다 차면 다음 팀에게 양보해야 한다.
- **왜 중요한가**: 타임 슬라이스가 짧으면 응답성↑ [컨텍스트 스위치](#context-switch)↑(오버헤드↑). 길면 처리량↑ 응답성↓. 트레이드오프의 대표 사례.

### CFS · `vruntime` {#cfs}

- **정의**:
  - **CFS (Completely Fair Scheduler)**: 리눅스 2.6.23 부터 기본 [스케줄러](#scheduler). "모든 [스레드](#thread)를 가능한 한 *공정하게* 다루겠다".
  - **`vruntime`**: 각 스레드가 지금까지 받은 CPU 시간을 가중치로 보정한 "가상 실행 시간". 작을수록 우선 실행 대상.
- **비유**: CFS = 식당 매니저가 "오늘 가장 적게 음식 받은 손님에게 다음 음식을 준다" 정책. vruntime = 손님이 받은 음식 양.
- **왜 중요한가**: 공정성을 추구하므로 Round-Robin 과 외형이 비슷하지만, 일부 스레드가 자발적으로 sleep 하거나 [nice 값](#nice)이 다르면 비율이 달라진다.

### nice 값 · 우선순위 {#nice}

- **정의**: 리눅스에서 [스레드](#thread)에 매기는 "양보 정도". -20 ~ +19 정수. 음수일수록 더 많은 CPU 시간을 받음, 양수일수록 양보.
- **비유**: 회의실 예약 우선권. 임원(낮은 nice)은 먼저 잡고, 일반 직원(높은 nice)은 양보.
- **왜 중요한가**: `nice` / `renice` 명령으로 조정 가능. 백그라운드 작업은 nice 값을 높여서 시스템 응답성을 지키는 게 정석.

### 빨간-검정 트리 (Red-Black Tree) {#rb-tree}

- **정의**: 자가 균형 이진 탐색 트리의 한 종류. 삽입·삭제·탐색이 모두 O(log n) 보장.
- **비유**: 자동으로 균형을 유지하는 분류 책장. 어느 책을 찾든 비슷한 시간에 찾을 수 있다.
- **왜 중요한가**: [CFS](#cfs) 가 "vruntime 가장 작은 [스레드](#thread)" 를 빨리 찾기 위해 이 자료구조를 쓴다. 스케줄링 결정이 O(log n) 안에 끝나서 스레드 수가 많아도 안정적.

### POSIX Threads (pthread) {#pthread}

- **정의**: 유닉스 계열의 표준 [스레드](#thread) API. `pthread_create(&tid, attr, func, arg)` 로 생성, `pthread_join(tid, &ret)` 로 종료 대기, `pthread_mutex_*` 로 락 관리.
- **비유**: 모든 유닉스 식당이 따르는 "요리사 추가 채용 표준 매뉴얼". 식당마다 똑같은 양식으로 요리사를 부르고 보낼 수 있다.
- **왜 중요한가**: C/C++/Rust 등 시스템 언어의 멀티스레드는 거의 다 pthread 위에 얹혀 있다. Python `threading` 모듈도 내부적으로 pthread 를 부른다. 리눅스에서는 [`clone()`](#clone-syscall) 시스템 콜로 구현.

### TLS (Thread-Local Storage) {#tls}

- **정의**: 같은 [프로세스](#process) 안의 [스레드](#thread)들이 메모리를 공유하지만, *스레드별로 따로 존재하는* 변수를 두는 메커니즘. C 에서는 `__thread int x`, Python 에서는 `threading.local()`.
- **비유**: 식당의 공용 주방에서 각 요리사가 자기만의 도마와 칼을 따로 갖는 것. 재료는 공유하지만 작업 도구는 개인용.
- **왜 중요한가**: 멀티스레드에서 "스레드마다 다른 컨텍스트(예: DB 커넥션, 로깅 prefix)" 가 필요할 때 [락](#lock-mutex) 없이도 안전하게 가질 수 있다. `errno` 도 TLS 로 구현돼 있어 멀티스레드에서 충돌 안 남.

## 6. 병렬 실행 모델

### GIL (Global Interpreter Lock) {#gil}

- **정의**: CPython(가장 흔한 Python 구현체) 인터프리터 전체를 하나의 거대한 [락](#lock-mutex)으로 묶는 메커니즘. 결과: **같은 프로세스 안에서는 한 번에 *오직 한 [스레드](#thread)만* Python 바이트코드를 실행할 수 있다**.
- **비유**: 5인용 회의실에 마이크가 하나뿐인 회의. 5명이 들어와 있어도 마이크 가진 한 명만 발언 가능.
- **왜 중요한가**: GIL 때문에 Python 멀티스레드는 **[CPU-bound](#cpu-vs-io-bound) 작업에는 효과가 없다** (코어가 8개여도 1개만 쓰는 셈). [I/O-bound](#cpu-vs-io-bound) 작업은 I/O 대기 중 GIL 이 풀려서 의미가 있음. CPU 병렬화가 필요하면 [멀티프로세싱](#py-multiprocessing) 으로 가야 한다. CPython 의 메모리 관리(참조 카운팅)가 스레드-안전하지 않아서 도입된 역사적 결정.

### multiprocessing (Python) {#py-multiprocessing}

- **정의**: Python 의 표준 멀티프로세스 라이브러리. `multiprocessing.Process`, `Pool`, `Queue` 등을 제공해 [GIL](#gil) 을 우회하는 진짜 병렬 실행을 가능하게 한다. 내부적으로 [`fork`](#fork-exec) 시스템 콜을 호출.
- **비유**: 마이크 하나로 회의가 안 굴러가니, 회의실을 5개로 쪼개서 5개 회의를 동시에 열기.
- **왜 중요한가**: CPU-bound Python 작업을 멀티코어로 가속하는 *표준* 방법. 단점은 [IPC](#ipc) 가 명시적이고 비싸다는 것 — 객체를 [pickle](https://docs.python.org/3/library/pickle.html) 로 직렬화해서 보내야 함.

### asyncio · 이벤트 루프 {#asyncio}

- **정의**: 단일 [스레드](#thread) 안에서 *수많은 동시 작업* 을 효율적으로 굴리기 위한 비동기 프로그래밍 모델. Python `asyncio`, JavaScript Node.js, C 의 libuv 등이 같은 원리.
  - **이벤트 루프**: "지금 깨어 있어야 할 작업" 들의 큐를 돌며 하나씩 실행하다가, I/O 대기에 들어가면 다음 작업으로 넘어가는 단일 스레드 스케줄러.
- **비유**: 1명의 손님 응대 직원이 손님 A 에게 "메뉴 보고 계세요" 하고 손님 B 응대를 시작, 그 사이 손님 A 가 결정하면 다시 가는 식. 한 명이 100명을 효율적으로 응대.
- **왜 중요한가**: I/O-bound 작업에 [스레드](#thread) 100개를 만드는 대신 *하나의 이벤트 루프* 로 처리하면 메모리·컨텍스트 스위치 비용이 훨씬 적다. 단, CPU-bound 작업이 끼면 *모든 다른 작업이 같이 멈춘다* (이벤트 루프 블로킹).

### 암달의 법칙 (Amdahl's Law) {#amdahl}

- **정의**: 1967년 Gene Amdahl 이 정리한 병렬화의 수학적 한계.
  - **Speedup ≤ 1 / ((1 − p) + p / N)** &nbsp;&nbsp; (p = 병렬화 가능한 코드 비율, N = 코어 수)
- **비유**: 부엌 일에서 *씻기·자르기* 는 사람을 늘려 빨라지지만 *국 한 솥을 끓이는 시간* 은 사람 수 늘려도 안 줄어든다. 끓이는 시간이 직렬 부분.
- **왜 중요한가**: 직렬 부분 5% 만 있어도 코어를 *무한* 으로 늘려도 최대 20배. 50% 직렬이면 최대 2배. 그래서 병렬 시스템 설계의 핵심은 "직렬 구간을 어떻게 줄이느냐" 다.

### 스레드 풀 (Thread Pool) {#thread-pool}

- **정의**: [스레드](#thread)를 그때그때 만들지 않고, *미리 N개* 를 만들어 두고 작업을 분배하는 패턴. 작업이 끝난 스레드는 다음 작업을 받아 재사용.
- **비유**: 콜센터 — 전화 올 때마다 사람 뽑지 않고, 상담사 50명을 미리 대기시켜 두고 들어오는 전화를 분배.
- **왜 중요한가**: 스레드 생성·종료 비용이 작업당 비용보다 클 때 큰 차이가 난다. Python `concurrent.futures.ThreadPoolExecutor`, Java `ExecutorService` 등이 표준 구현. [멀티프로세싱](#py-multiprocessing) 에서도 `ProcessPoolExecutor`, `multiprocessing.Pool` 로 같은 패턴.

### Producer-Consumer (생산자-소비자) {#producer-consumer}

- **정의**: 한 쪽은 작업을 [큐](#critical-section)에 *생산* 하고, 다른 쪽은 큐에서 꺼내 *소비* 한다. 큐가 비면 소비자가 대기, 가득 차면 생산자가 대기.
- **비유**: 카페에서 바리스타(생산자)가 음료를 카운터(큐)에 올리면 손님(소비자)이 가져간다. 카운터 자리가 다 차면 바리스타가 멈추고, 비어 있으면 손님이 기다린다.
- **왜 중요한가**: 멀티스레드·멀티프로세스 동시 실행의 가장 흔한 패턴. 큐 자체는 [락](#lock-mutex) + [컨디션 변수](#condition-variable) 로 구현된다. 데이터 파이프라인의 기본 형태.

### Fork-Join (분할-합치기) {#fork-join}

- **정의**: 큰 작업을 N 개로 쪼개서 병렬 실행 → 모두 끝나면 결과를 모으는 패턴.
- **비유**: 큰 시험지 채점을 5명에게 나눠 주고, 다 끝나면 점수를 합산.
- **왜 중요한가**: MapReduce, 병렬 정렬, 행렬 곱셈 등 대량 데이터 처리의 기본 구조. Java `ForkJoinPool`, Go 의 `sync.WaitGroup` 등이 표준 구현 제공.

## 7. 동시성 · 데드락

### Race Condition (경쟁 조건) {#race-condition}

- **정의**: 두 개 이상의 [스레드](#thread)가 같은 자원을 동시에 만질 때, 누가 먼저 도달하느냐에 따라 결과가 달라지는 상태. "버그가 가끔만 나타난다" 는 가장 무서운 종류의 버그.
- **비유**: 식당에서 두 종업원이 동시에 같은 주문서를 읽어 같은 음식을 두 번 만드는 사고.
- **왜 중요한가**: 한 번 만들고 끝나는 게 아니라, *발생 가능 조건* 만 잠재돼 있다. 테스트에서 99 번 통과해도 100 번째에 사고가 난다. [락](#lock-mutex)으로 예방하는 게 정석.

### 임계구역 (Critical Section) {#critical-section}

- **정의**: 한 번에 *한 [스레드](#thread)만* 들어가야 하는 코드 구간. 공유 자원을 만지는 부분 전체.
- **비유**: 일인용 탈의실. 한 명이 들어가면 다른 사람은 밖에서 기다려야 한다.
- **왜 중요한가**: 임계구역을 정확히 식별해서 [락](#lock-mutex)으로 둘러싸야 [race condition](#race-condition) 을 막을 수 있다.

### 락 · 뮤텍스 (Lock · Mutex) {#lock-mutex}

- **정의**:
  - **락**: 자원을 보호하는 잠금 장치. 잡으면(acquire) 다른 [스레드](#thread)는 못 잡고 대기.
  - **뮤텍스 (Mutex = Mutual Exclusion)**: 락의 가장 흔한 구현. 한 번에 한 명만 잡을 수 있는 가장 단순한 락.
- **비유**: 탈의실 문 잠금. 한 명만 잠그면 다른 사람은 밖에서 기다린다. 안에 있는 사람이 풀어야 다음 사람 입장.
- **왜 중요한가**: 락을 잘못 쓰면 [데드락](#coffman)이 난다. 모든 동시성 버그의 90% 가 락 관리 실수.

### 세마포어 (Semaphore) {#semaphore}

- **정의**: [뮤텍스](#lock-mutex)의 확장판. 동시에 N 명까지 잡을 수 있는 락. N=1 이면 뮤텍스와 동일.
- **비유**: 좌석 5개짜리 회의실. 5명까지는 들어가고, 6번째는 한 명 나갈 때까지 대기.
- **왜 중요한가**: "동시 접속자 수 제한" 같은 시나리오에 적합. 데드락 가능성은 뮤텍스와 동일하게 존재.

### 컨디션 변수 (Condition Variable) {#condition-variable}

- **정의**: "특정 조건이 만족될 때까지 잠들었다가, 다른 [스레드](#thread)가 깨워줄 때 일어나는" 동기화 메커니즘. 보통 [뮤텍스](#lock-mutex)와 한 쌍으로 쓴다. C 에서 `pthread_cond_wait` / `pthread_cond_signal`, Python 에서 `threading.Condition`.
- **비유**: 손님이 "음식 나오면 깨워 주세요" 하고 자리에서 잠든 뒤, 종업원이 와서 어깨를 두드려 깨우는 방식. [busy-wait](#busy-wait) 처럼 계속 확인하지 않아도 됨.
- **왜 중요한가**: [Producer-Consumer 패턴](#producer-consumer) 의 핵심 도구. 큐가 비었을 때 소비자가 *효율적으로 잠들었다가* 생산자가 깨우는 식. 락만으로는 못 만든다.

### 데드락 · Coffman 조건 {#coffman}

- **정의**: 데드락 = 서로의 자원을 무한히 기다려 멈춤. E.G. Coffman 이 1971년 논문에서 정리한 4가지 필요조건(상호 배제·점유 대기·비선점·순환 대기)이 동시에 성립할 때만 일어난다.
- **비유**: 좁은 골목에서 두 차가 마주쳐 서로 양보하지 않고 영원히 멈춰 있는 상황.
- **왜 중요한가**: "데드락 막으려면 어떤 조건을 깨야 하나" 의 체크리스트. 4개 다 외워야 데드락 분석을 자기 말로 할 수 있다.

### lock-free 자료구조 {#lock-free}

- **정의**: [락](#lock-mutex) 없이도 멀티스레드가 안전하게 쓸 수 있도록 설계한 자료구조. CPU 의 원자적 명령(`compare-and-swap`) 으로 동기화.
- **비유**: 일인용 탈의실 대신, 누가 들어가도 충돌 안 나는 자동 회전문.
- **왜 중요한가**: [데드락](#coffman) 자체가 없어진다. 단, 설계가 매우 어렵고 모든 자료구조에 만들 수 있는 건 아니다.

### try_lock · 타임아웃 {#try-lock}

- **정의**: "[락](#lock-mutex)을 잡으려고 시도하되, X 초 안에 못 잡으면 포기" 패턴.
- **비유**: 회의실 문이 잠겨 있으면 5분만 기다리고 다른 회의실 찾아가는 행동.
- **왜 중요한가**: [Coffman 조건](#coffman)의 "비선점" 을 깬다. 데드락 회피의 실용적 방법 중 하나.

### 식사하는 철학자 문제 (Dining Philosophers) {#dining-philosophers}

- **정의**: 1965년 다익스트라(Edsger Dijkstra)가 만든 동시성 학습용 가상 시나리오. 5명의 철학자가 원탁에 앉아 양옆 포크를 동시에 들어야 식사 가능 — 모두 왼쪽 포크부터 들면 데드락.
- **비유**: 5명이 원탁에 앉아 포크 5개를 사이에 두고 식사. 모두가 동시에 왼쪽 포크를 들고 오른쪽을 기다리면 영원히 멈춤.
- **왜 중요한가**: 운영체제 교과서마다 등장. 면접 단골 질문. 이번 미션 리포트의 *Root Cause Analysis* 에 인용하면 설득력이 확 올라간다.

### 기아 (Starvation) {#starvation}

- **정의**: [데드락](#coffman)은 아니지만 *특정 [스레드](#thread)만* 영영 자원을 못 받는 상태. 다른 스레드들이 계속 우선 처리돼서.
- **비유**: 식당에서 한 손님이 매번 우선순위 낮아서 새치기 당하다가 영원히 음식 못 받는 상황.
- **왜 중요한가**: Priority 스케줄링에서 자주 발생. 데드락과 다른 개념이라 구분해서 진단.

### 프로세스 상태 코드 (STAT) {#stat-codes}

- **정의**: `ps` 의 `STAT` 컬럼에 한 글자씩 찍히는 상태 코드.
  - `R`: Running (실행 중 또는 실행 가능)
  - `S`: Sleeping (이벤트 대기 중, 인터럽트 가능)
  - `D`: Uninterruptible sleep (디스크 I/O 대기 등, 강제 깨우기 불가)
  - `Z`: Zombie (종료됐는데 부모가 회수 안 함)
  - `T`: Stopped (Ctrl+Z 등으로 정지)
  - 뒤에 `l`: 멀티스레드, `+`: 포그라운드 등
- **비유**: 손님 상태 — 식사 중(R), 음식 기다림(S), 화장실(D), 가버림(Z).
- **왜 중요한가**: 데드락 의심 시 `Sl` 이 뜨면 멀티스레드가 잠들어 있다는 강한 단서.

### futex (Fast Userspace muTEX) {#futex}

- **정의**: 리눅스에서 유저 공간 [락](#lock-mutex)을 효율적으로 구현하기 위한 [시스템 콜](#syscall). 락이 경합 없으면 시스템 콜 없이 빠르게, 경합 있으면 커널에 잠들기.
- **비유**: 회의실 문이 안 잠겨 있으면 그냥 들어가고(빠름), 잠겨 있을 때만 "노크하고 기다리는" 시스템.
- **왜 중요한가**: Python `threading.Lock`, C++ `std::mutex` 등이 내부적으로 futex 를 쓴다. 그래서 데드락이 나면 [`wchan=futex_*`](#wchan) 가 결정적 단서.

### `wchan` (Wait Channel) {#wchan}

- **정의**: [스레드](#thread)가 잠들었을 때 그 호출이 일어난 커널 함수의 이름. `/proc/<pid>/wchan` 또는 `ps -L -o wchan`.
- **비유**: 잠든 사람이 "어떤 일 때문에 잠들었는지" 라벨을 붙여둔 것.
- **왜 중요한가**: `wchan` 만 보면 "이 스레드가 락 대기인가, I/O 대기인가, 단순 sleep 인가" 가 한눈에 구분된다. 데드락 분석의 핵심 컬럼.

### busy-wait (바쁜 대기) {#busy-wait}

- **정의**: 조건이 만족될 때까지 `while (!condition) {}` 식으로 *[CPU](#cpu) 를 계속 돌리면서* 기다리는 패턴. CPU 100% 점유의 가장 흔한 원인.
- **비유**: 친구가 올 때까지 문 앞에서 계속 깡총깡총 뛰는 행동. 가만히 의자에 앉아 기다리면 될 일을.
- **왜 중요한가**: 올바른 동시성 코드는 [컨디션 변수](#condition-variable) 같은 메커니즘으로 잠들었다가 *깨어나는* 방식을 쓴다. busy-wait 는 거의 항상 버그 또는 미숙한 설계.

### 무한 루프 (Infinite Loop) {#infinite-loop}

- **정의**: 종료 조건이 영영 거짓이 되지 않아 끝나지 않는 반복문. `while True: pass`, `for(;;) {}` 같은 코드.
- **비유**: "끝낼 때까지 계속" 만 정해두고 *끝낼 조건* 을 안 적은 일. 영원히 안 멈춘다.
- **왜 중요한가**: [CPU](#cpu) 100% 점유의 대표 원인. [busy-wait](#busy-wait) 도 무한 루프의 한 종류(차이: 조건을 *돌면서* 계속 체크). 무한 루프는 [Watchdog](#watchdog) 같은 외부 감시가 깨주지 않으면 자체적으로 못 멈추므로 시스템에 *반드시* 안전망이 필요.

### 참조 카운팅 (Reference Counting) {#refcount}

- **정의**: 모든 객체에 "지금 나를 가리키는 변수가 몇 개냐" 라는 정수 카운터를 붙이고, 카운터가 0 이 되면 *즉시* 메모리를 해제하는 메모리 관리 방식. CPython 의 기본 [GC](#malloc-free-gc) 가 이 방식.
- **비유**: 도서관 책에 "지금 누가 빌렸나" 카드를 붙여 두고, 카드가 비면 책장에 자동 반납.
- **왜 중요한가**: [GIL](#gil) 이 존재하는 *근본 이유*. 멀티스레드가 동시에 카운터를 ++/-- 하면 [race condition](#race-condition) 으로 카운터가 망가져 메모리 누수 또는 이중 해제(=크래시)가 일어난다. GIL 은 그걸 "한 번에 한 스레드만 객체를 만지게" 막는 가장 단순한 해결책이다.

## 8. 운영 도구

### `/proc` 가상 파일시스템 {#proc}

- **정의**: 실제 디스크에 없는 가짜 파일시스템. 커널이 현재 시스템·프로세스 상태를 텍스트 파일처럼 보여주는 인터페이스. `cat /proc/cpuinfo`, `cat /proc/<pid>/status` 등.
- **비유**: 빌딩 1층에 걸린 실시간 전광판. 지금 누가 어디 있는지 텍스트로 보여준다.
- **왜 중요한가**: `top`, `ps`, `free` 모두 결국 `/proc` 을 읽는다. 명령어가 막힐 때 직접 `/proc` 을 보면 원본 진실에 닿을 수 있다.

### `top` · `htop` · `ps` {#top-ps}

- **정의**:
  - `top`: [프로세스](#process) 사용량을 *실시간* 으로 갱신해 보여주는 도구. `q` 로 종료.
  - `htop`: top 의 향상 버전. 색깔·트리뷰·마우스 지원. (별도 설치 필요)
  - `ps`: 그 순간의 *스냅샷* 만 한 번 찍어 표시. 스크립트에서 파싱하기 좋음.
- **비유**: top = CCTV 라이브 화면, ps = 폴라로이드 사진 한 장.
- **왜 중요한가**: 데드락은 시간이 흐르지 않으므로 ps 한 장으로 충분. CPU 폭주는 실시간 변화를 봐야 하니 top 필요.

### `dmesg` {#dmesg}

- **정의**: 커널이 부팅 이후 출력한 메시지를 모아 보여주는 명령어. 하드웨어 변화·드라이버·[OOM Killer](#oom-killer) 발동 등이 다 여기에 있다.
- **비유**: 빌딩 관리실의 로그북. 정전·화재경보·승강기 고장 같은 큰 사건이 다 기록됨.
- **왜 중요한가**: 앱 로그에 안 나오는 시스템 레벨 사건(특히 OOM Killer)이 dmesg 에만 있을 수 있다.

### `vmstat` {#vmstat}

- **정의**: 가상 메모리·CPU·디스크 I/O 의 초당 변화를 한 줄씩 보여주는 도구. `vmstat 1` 로 매초 갱신.
- **비유**: 자동차 계기판. 속도·연료·rpm 을 동시에 한 화면에서 모니터링.
- **왜 중요한가**: OOM 직전에 `si`(swap in) / `so`(swap out) 가 폭증하면 [쓰래싱](#swap-thrashing)의 결정적 증거.

### 포트 · localhost · 0.0.0.0 {#port-localhost}

- **정의**:
  - **포트 (Port)**: 한 컴퓨터 안의 *프로그램별 통신 창구* 번호. 0~65535 정수.
  - **localhost (127.0.0.1)**: "내 컴퓨터" 를 가리키는 특수 주소. 패킷이 *외부로 안 나간다*.
  - **0.0.0.0**: 서버가 "모든 네트워크 인터페이스에서 들어오는 요청을 받겠다" 는 의미로 바인딩하는 주소. 외부 접속도 허용.
- **비유**: 아파트 호수. IP 주소 = 동, 포트 = 호수. 한 동(컴퓨터)에 여러 프로그램이 살고 각자 다른 호수(포트)에서 손님을 받는다.
- **왜 중요한가**: `agent-leak-app` 이 `0.0.0.0:15034` 에 바인딩되어 있다면 그 포트로 `curl` 을 쏘면 응답해야 정상 — [데드락](#coffman) 이면 timeout. `ss -tlnp` 또는 `netstat -tlnp` 로 어떤 프로세스가 어떤 포트를 잡고 있는지 확인.

### HTTP · `curl` · `wget` {#http-curl}

- **정의**:
  - **HTTP (HyperText Transfer Protocol)**: 웹 서버와 클라이언트가 텍스트로 주고받는 표준 양식. `GET /path` 같은 요청과 `200 OK` / `500 Internal Server Error` 같은 응답.
  - **curl**: 명령행에서 HTTP 요청을 보내는 도구. `curl --max-time 3 http://...`
  - **wget**: 파일 다운로드 특화 HTTP 클라이언트. 비슷한 용도지만 백그라운드 다운로드에 강함.
- **비유**: HTTP = 손님이 웨이터에게 주문하는 공통 양식. curl = 그 양식대로 주문서를 쓰는 도구.
- **왜 중요한가**: 외부 health check 의 표준 방법. `curl --max-time 3 http://0.0.0.0:15034` 가 timeout 된다면 *서버가 살아있어도 응답 못 한다* 는 외부 증거 — [데드락](#coffman) 분석의 6대 증거 중 하나.

### grep · awk · sed {#text-tools}

- **정의**: 셸의 텍스트 처리 3대장.
  - **grep**: 패턴에 맞는 *줄* 만 골라냄. `grep -i "error" app.log`
  - **awk**: *필드(컬럼)* 단위로 잘라서 추출/계산. `awk '{print $2, $5}'` 는 2·5번째 컬럼.
  - **sed**: *문자열 치환·삭제*. `sed 's/foo/bar/g'` 는 모든 foo→bar.
- **비유**: 책에서 grep = "특정 단어가 있는 페이지만 표시", awk = "각 페이지의 2번째 줄만 모음", sed = "이 단어를 저 단어로 일괄 교체".
- **왜 중요한가**: `monitor.log` 같은 텍스트 로그에서 원하는 정보만 뽑는 표준 도구. `tail -f app.log | grep CRITICAL` 처럼 [파이프](#pipe-redirect)와 결합해서 라이브 모니터링도 한다.

### `systemctl` · `journalctl` {#systemd-tools}

- **정의**: systemd(리눅스의 표준 init 시스템) 관리 도구.
  - **systemctl**: [데몬](#daemon) 시작/정지/상태 조회. `systemctl status sshd`, `systemctl restart nginx`, `systemctl enable agent-leak`.
  - **journalctl**: systemd 가 한곳에 모은 시스템 로그 조회. `journalctl -k` (커널만), `journalctl -u sshd` (특정 서비스), `journalctl --since "5 min ago"`.
- **비유**: systemctl = 빌딩 자동제어 패널(시설별 on/off). journalctl = 그 빌딩의 전자 로그북.
- **왜 중요한가**: 데몬 미션에서 `systemctl status` 로 살아 있는지 보고, `journalctl -k | grep -i oom` 로 [커널 OOM Killer](#oom-killer) 흔적을 확인. [`dmesg`](#dmesg)와 비슷하지만 영구 저장·필터링이 강함.

### 멀티프로세싱 vs 멀티스레딩 한 줄 요약

| 항목 | 멀티프로세싱 | 멀티스레딩 |
| --- | --- | --- |

| 한 명 죽으면 | 다른 명 무사 | 전체 종료 |
| Python GIL 우회 | 가능 | 불가 |
| 적합한 작업 | [CPU-bound](#cpu-vs-io-bound) (Python), 격리 필요 | [I/O-bound](#cpu-vs-io-bound), 데이터 공유 잦음 |

### 데드락 vs 라이브락 vs 기아

| 현상 | 정의 | 진단 |
| --- | --- | --- |
| [데드락](#coffman) | 서로의 자원을 기다리며 영원히 정지 | CPU 0%, [wchan](#wchan) = futex_*, 로그 멈춤 |
| 라이브락 | 서로 양보만 하느라 진척이 없음 (실제로는 CPU 를 쓰며 도는데 결과 없음) | CPU 높음, 로그는 계속 찍히는데 같은 패턴 반복 |
| [기아 (Starvation)](#starvation) | 특정 스레드만 영영 자원을 못 받음 | 한 스레드의 진행률만 멈춰 있음, 다른 건 정상 |

### 시스템 호출 추적 도구

| 도구 | 무엇을 보여주나 |
| --- | --- |
| `strace -f -p $PID` | 그 프로세스(+자식들)가 *지금* 호출하는 [시스템 콜](#syscall)을 실시간으로 출력. 데드락 시 모든 스레드가 `futex(...)` 에서 멈춰 있는 모습이 직접 보인다. |
| `ltrace -p $PID` | 라이브러리 함수 호출을 추적. `pthread_mutex_lock` 같은 함수가 어디서 멈췄는지 보임. |
| `perf top` | CPU 가 어느 함수에서 시간을 가장 많이 쓰는지 실시간 프로파일. CPU 스파이크 원인 후보를 좁혀준다. |
| `pidstat -t 1` | 스레드 단위 CPU/메모리 초당 변화. `top -H` 보다 스크립트 친화적. |
