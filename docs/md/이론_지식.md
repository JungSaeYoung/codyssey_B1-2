# 이론 지식 — 컴퓨터 구조 · 메모리 · CPU · 동시성 · 데드락

> 이 문서는 B1-2 미션(OOM / CPU Spike / Deadlock 분석)을 수행할 때 **로그·관제 수치 뒤에서 무슨 일이 벌어지고 있는가** 를 학습자가 스스로 추론할 수 있도록, 컴퓨터 구조 → OS 동작 → 앱 정책의 흐름을 한 줄로 잇기 위한 이론 지식을 정리한다.
>
> "MEM% 가 96% 까지 올라가서 SELF-TERMINATED 됐다" 같은 한 줄을 **누가, 어떤 자료구조 위에서, 어떤 정책에 의해 종료시켰는가** 로 풀어 설명할 수 있어야 한다.
>
> **이 문서 읽는 법**: 본문에서 처음 등장하는 모든 전문 용어는 [용어집](용어집.md)으로 가는 하이퍼링크로 표시된다. 링크를 클릭하면 용어집의 해당 항목으로 이동해 정의·비유·왜 중요한가 를 확인할 수 있고, 좌측 사이드바의 "이론 지식" 을 다시 클릭하면 본문으로 돌아온다.

## 0. 왜 이론을 알아야 하는가

쉘 명령어 한 줄(`top`, `ps`, `kill`)만 외워서 장애를 해결하는 단계와, **장애가 왜 일어났는지 운영체제 동작 원리로 설명할 수 있는 단계**는 분명히 다르다.

이번 미션의 본질은 다음 한 문장이다:

> **"외부에서 관측 가능한 데이터(로그·관제·시스템 도구)만으로, 프로세스 내부와 OS 내부에서 일어난 일을 역추론한다."**

따라서 학습자는 아래 3개 레이어가 한 줄에 연결되는 멘탈 모델을 가져야 한다.

```
[Application 정책]  MemoryGuard / Watchdog / Lock 획득 순서
        ↑↓ 시스템 콜 (malloc, mmap, futex, sched_yield…)
[Operating System]  가상 메모리 / OOM Killer / 스케줄러 / 뮤텍스
        ↑↓ 명령어 실행
[Hardware]          CPU 코어 · 캐시 · DRAM · 디스크
```

OOM 로그 한 줄을 보면 학습자는 머릿속에서 **하단부터** 거꾸로 올라가며 묻는 습관이 생겨야 한다.

- DRAM 의 어디까지 채워졌나? (Hardware)
- 가상 메모리는 어떤 페이지를 디스크로 내쫓고 있었나? (OS)
- 그래서 앱 정책이 어느 임계치에서 자신을 죽였나? (Application)

상위 레이어가 하위 레이어에게 일을 시키는 방법이 [시스템 콜](용어집.md#syscall)이고, 우리가 외부에서 관찰하는 모든 수치는 결국 [운영체제·커널](용어집.md#os-kernel-userspace)이 자기 내부 상태를 노출해주는 흔적이다.

## 1. 컴퓨터 구조 기초 — 폰 노이만 모델

### 1.1 모든 것이 메모리에 있다

현대 컴퓨터의 기본 가정은 **명령어(코드)와 데이터가 같은 메모리에 들어 있고, [CPU](용어집.md#cpu) 는 그 메모리에서 한 단어씩 꺼내 실행한다** 는 [폰 노이만 구조](용어집.md#von-neumann)다. 이 가정에서 두 가지 시스템 자원이 곧바로 도출된다.

| 자원 | 역할 | 이번 미션에서 관찰되는 모습 |
| --- | --- | --- |
| **CPU** | 메모리에서 명령어를 꺼내 실행 | `top` 의 `%CPU` 컬럼, `CPU_MAX_OCCUPY` |
| **메모리** | 코드/데이터 보관 | `free`, `top` 의 `%MEM` 컬럼, `MEMORY_LIMIT` |

CPU 가 빠를수록 단위 시간당 더 많은 명령어를 처리하고([스루풋](용어집.md#throughput-vs-latency) ↑), 메모리가 클수록 동시에 살아 있을 수 있는 객체(데이터)의 수가 늘어난다. **이 두 자원 모두 유한** 하다는 것이 이번 미션 3개 장애의 공통 출발점이다.

### 1.2 폰 노이만 병목 — "메모리는 항상 느리다"

| 저장 계층 | 접근 [지연](용어집.md#time-units) (rough) | 비유 |
| --- | --- | --- |
| [레지스터](용어집.md#register) | 0.3 ns | 책상 위 |
| [L1 / L2 / L3 캐시](용어집.md#cache) | 1 ~ 20 ns | 책상 서랍 |
| [DRAM (RAM)](용어집.md#dram) | ~ 100 ns | 책장 |
| [SSD](용어집.md#disk) | ~ 100 µs (= 100,000 ns) | 도서관 |
| [HDD](용어집.md#disk) | ~ 10 ms (= 10,000,000 ns) | 다른 도시 도서관 |

- 한 단계 내려갈 때마다 보통 **100배~1000배** 느려진다.
- 따라서 OS 는 자주 쓰는 데이터를 위쪽 계층에 두려고 발버둥치고, 그 행위가 곧 캐시, [페이지 교체, 스왑](용어집.md#swap-thrashing) 같은 메커니즘이다.
- 메모리 누수가 무서운 이유는 단지 "공간이 모자란다" 가 아니라, **DRAM 이 가득 차면 OS 가 디스크(스왑)로 페이지를 내쫓기 시작하고**, 디스크는 DRAM 보다 *수만 배* 느리기 때문에 시스템 전체가 갑자기 멈춰버리는 데 있다. 이 현상을 [쓰래싱(thrashing)](용어집.md#swap-thrashing)이라 부른다.

> 🔑 **미션 연결점**: `agent-leak-app` 의 메모리 누수가 임계치에 도달하면 OS 가 페이지 교체로 발버둥치다 결국 OOM Killer 가 발동한다. 즉 "메모리 부족 → CPU 도 같이 느려진다" 가 결합돼 나타날 수 있다. monitor.log 에서 MEM% 가 90% 를 넘어가는 순간 CPU% 도 같이 튀어 오르는 패턴이 보이는지 살펴보면 쓰래싱의 흔적을 잡을 수 있다.

## 2. 메모리 구조 — 가상 메모리부터 OOM 까지

### 2.1 프로세스의 가상 메모리 레이아웃

리눅스는 모든 [프로세스](용어집.md#process)에게 **자기만 쓰는 [가상 주소 공간](용어집.md#virtual-address-space)** 을 환상으로 제공한다(64bit 에서는 사실상 무한처럼 보임). 이 주소 공간 안에는 정해진 자리에 영역들이 배치된다.

```
높은 주소
┌─────────────────────────────┐
│           Stack             │  ← 자동 변수, 함수 호출 프레임 (↓ 자라남)
│             ↓               │
│                             │
│             ↑               │
│           Heap              │  ← malloc / new / 동적 객체  (↑ 자라남)
├─────────────────────────────┤
│           BSS               │  ← 초기화 안된 전역/정적 변수
│           Data              │  ← 초기화된 전역/정적 변수
│           Text (Code)       │  ← 기계어 명령
└─────────────────────────────┘
낮은 주소
```

[각 영역의 의미](용어집.md#memory-segments)는 다음과 같이 나뉜다.

| 영역 | 누가 채움 | 누가 비움 | 이번 미션에서 |
| --- | --- | --- | --- |
| **Text** | 로더가 [ELF](용어집.md#elf) 에서 로드 | 종료 시 | 거의 안 변함 |
| **Data / BSS** | 로더 | 종료 시 | 거의 안 변함 |
| **Heap** | [`malloc`](용어집.md#malloc-free-gc) (Python 객체 등) | `free` 또는 GC | **여기가 새는 곳** |
| **Stack** | 함수 호출 시 push | 함수 반환 시 pop | 보통 일정 — 단 무한 재귀 시 오버플로 |

> 🔑 **메모리 누수의 정체**: "사용 안 하지만 참조가 남아 있어서 GC/`free()` 가 회수하지 못하는 객체가 Heap 에 계속 누적되는 상태." 즉 **누수는 Heap 영역에서만 일어난다**(거의 항상). Python 같은 [GC](용어집.md#malloc-free-gc) 언어에서도 전역 리스트에 append 만 하고 pop 안 하면 객체가 계속 살아 있으니 누수가 된다.

### 2.2 가상 메모리(Virtual Memory) — OS 가 보여주는 환상

프로세스가 보는 주소는 **가상 주소** 다. 실제 DRAM 의 물리 주소는 OS 와 [MMU](용어집.md#mmu) (CPU 안의 하드웨어)가 둘 사이를 매핑한다.

- 매핑 단위는 [페이지(Page)](용어집.md#page) — 보통 4 KB.
- 매핑 정보는 [페이지 테이블](용어집.md#page-table-tlb)에 들어 있고, 자주 쓰는 매핑은 CPU 안 [TLB](용어집.md#page-table-tlb)에 캐시된다.
- 프로세스가 새 페이지를 처음 건드릴 때 → [페이지 폴트](용어집.md#page-fault) → 커널이 물리 페이지를 할당해 매핑.

이 구조가 가져다 주는 두 가지 사실:

1. **물리 메모리는 처음부터 할당되지 않는다 ([lazy allocation](용어집.md#lazy-allocation))** — `malloc(1GB)` 만 해서는 RAM 1GB 가 안 줄어든다. 그 메모리를 *건드려야* RAM 이 줄어든다.
2. **RAM 이 모자라면 OS 는 자주 안 쓰는 페이지를 디스크로 내쫓는다 (swap out)** — 이걸 [페이지 교체](용어집.md#swap-thrashing)라 한다.

### 2.3 RSS vs VSZ — `ps`, `top` 의 메모리 컬럼

[VSZ 와 RSS](용어집.md#vsz-vs-rss) 의 차이는 누수 분석에서 가장 중요한 구분이다.

| 컬럼 | 의미 | 이번 미션에서 봐야 할 것 |
| --- | --- | --- |
| **VSZ** | 가상 메모리 총량 (예약 포함) | 거의 안 변하면 정상. 급격히 늘면 mmap 이슈 |
| **RSS** | Resident Set Size — **실제로 DRAM 에 올라가 있는** 양 | **이게 누수의 진짜 지표** |
| **%MEM** | RSS / 전체 DRAM 비율 | monitor.log 에 찍히는 값과 동일 의미 |

`ps -o pid,vsz,rss,pcpu,comm -p $PID` 로 같이 보면 누수 패턴이 명확해진다 — VSZ 는 일정한데 RSS 만 우상향이면 명백한 누수다.

### 2.4 OOM (Out-Of-Memory) — 두 가지 트리거

리눅스에서 "메모리 부족으로 종료" 는 **두 가지 다른 메커니즘** 으로 일어날 수 있다.

#### 1) 커널의 [OOM Killer](용어집.md#oom-killer)

- DRAM + 스왑까지 다 차서 더는 페이지를 못 받을 때, 커널이 **점수(`oom_score`)가 가장 높은 프로세스를 [SIGKILL](용어집.md#signal)** 로 강제 종료.
- 흔적: `dmesg | grep -i oom` 또는 `journalctl -k | grep -i oom-killer` 에서 `Killed process … total-vm:… anon-rss:…` 라인.
- 학습자의 프로세스 *바깥*(커널) 에서 일어나는 일이라 앱 로그에는 흔적이 안 남는다.

#### 2) 앱 내부의 메모리 보호 정책 (이번 미션 — `MemoryGuard`)

- `agent-leak-app` 은 자신의 RSS 를 주기적으로 체크하고, [`MEMORY_LIMIT` 환경변수](용어집.md#env-var)를 넘으면 **자기 자신을 종료** 한다(`SELF-TERMINATED`). 즉 *self-protection* 패턴.
- 흔적: 앱 로그에 `[CRITICAL] [MemoryGuard] Memory limit exceeded (…MB >= …MB)` 처럼 명시적으로 남음. 커널 dmesg 에는 안 남는다.

> 🔑 **이번 미션은 (2) 다.** OOM Killer 가 발동되기 *전에* 앱이 스스로 죽는 구조. 학습자가 OOM Killer 와 헷갈리지 않는 게 중요하다 — 같은 "OOM 으로 죽었다" 라도 누가 죽였는지를 로그로 구분해야 한다.

| 구분 | 트리거 | [시그널](용어집.md#signal) | 로그 위치 |
| --- | --- | --- | --- |
| 커널 OOM Killer | 시스템 전체 RAM 부족 | SIGKILL (9) | `dmesg`, `/var/log/syslog` |
| MemoryGuard (앱) | 프로세스 RSS > `MEMORY_LIMIT` | 자체 종료 | `$AGENT_LOG_DIR/agent-leak-app.log` |

### 2.5 환경변수 `MEMORY_LIMIT` 의 의미

이 미션에서 `MEMORY_LIMIT=256` 의 의미는 "256 MB 까지는 누수가 쌓여도 봐주겠다" 이다. 즉 **누수를 막는 게 아니라, 한계까지의 생존 시간을 늘리는 임시 조치** 다.

리포트의 *Workaround* 항목에 적어야 할 핵심:
- `MEMORY_LIMIT` 상향은 **근본 해결이 아니다.** 새는 속도가 일정하다면 단지 죽기까지의 시간을 산술적으로 늘릴 뿐.
- 근본 해결은 누수 지점(전역 리스트, 캐시, 참조 사이클 등)을 코드에서 끊는 것.

## 3. CPU — 프로세스 · 스레드 · 스케줄링

### 3.1 프로세스 vs 스레드

| 구분 | [프로세스](용어집.md#process) | [스레드](용어집.md#thread) |
| --- | --- | --- |
| 주소 공간 | 자기 것 (격리) | **부모 프로세스와 공유** |
| 생성 비용 | 크다 (fork + 새 [PCB](용어집.md#pcb)) | 작다 (스택만 따로) |
| 통신 | [IPC](용어집.md#ipc) (파이프, 소켓, shm) | 같은 메모리 — 변수 직접 공유 |
| 한 명 죽으면? | 다른 프로세스 무사 | **같은 프로세스 안 모두 종료** |

리눅스 커널은 사실 프로세스/스레드를 거의 같은 자료구조([`task_struct`](용어집.md#pcb))로 다룬다 — 그래서 `ps -L` 또는 `top -H` 로 스레드 단위 관찰이 가능하다. 두 스레드를 번갈아 실행시키는 데 드는 비용이 [컨텍스트 스위치](용어집.md#context-switch)이고, 너무 자주 일어나면 그 자체가 성능 저하의 원인이 된다.

> 🔑 **데드락 미션 연결**: 데드락은 **스레드들이 같은 메모리를 공유하면서 락을 서로 잡고 있을 때** 발생한다. 그래서 `MULTI_THREAD_ENABLE=false` 로 멀티스레드를 끄면 자동으로 회피된다. 멀티프로세스로 같은 일을 하면 (메모리가 격리돼 있으므로) 같은 종류의 락 문제는 거의 안 일어난다.

### 3.2 CPU 사용률(%CPU)이 100% 가 넘는 이유

- 1 [코어](용어집.md#core-smt) 1 시간 = 100% × 1시간
- 2 코어 1 시간 = **200%** × 1시간

`top` 의 `%CPU` 는 **한 코어를 100% 로 잡아 먹는 비율** 을 합산한 값이다. 4 코어 머신에서 한 프로세스가 멀티스레드로 4개 코어를 다 점유하면 400% 까지 찍힐 수 있다.

`agent-leak-app` 이 `CPU_MAX_OCCUPY=80` 이면 보통 "단일 코어 기준 80% 까지 봐줌" 으로 해석된다. monitor.sh 의 CPU% 는 **시스템 전체** 의 사용률([idle](용어집.md#idle) 의 반대) 이므로 두 수치를 헷갈리지 말 것.

### 3.3 스케줄러 — Linux CFS 의 핵심 직관

리눅스의 기본 [스케줄러](용어집.md#scheduler)는 [**CFS(Completely Fair Scheduler)**](용어집.md#cfs) 다. 한 줄로 요약하면:

> "모든 실행 가능한(runnable) [스레드](용어집.md#thread) 중에서, **지금까지 가장 적게 CPU 를 받은** 놈에게 다음 [타임 슬라이스](용어집.md#time-slice)를 준다."

- 각 스레드는 `vruntime` 이라는 가상 실행 시간을 누적한다.
- 스케줄러는 `vruntime` 이 가장 작은 스레드를 [빨간-검정 트리](용어집.md#rb-tree)에서 골라 실행시킨다.
- 그 스레드가 자기 몫만큼 돌고 나면 `vruntime` 이 증가 → 트리 안에서 위치가 뒤로 밀림 → 다른 스레드가 골라짐.

겉으로 보기엔 **Round-Robin 처럼 보이지만**, 엄밀히는 *공정성 기반 가중치 스케줄링* 이다. 가중치는 [nice 값](용어집.md#nice)으로 조정한다.

#### 보너스 과제와의 연결

`agent-leak-app` 의 워커 스레드 로그에서 A → B → C → A → B … 식으로 돌아가며 진행률이 찍힌다면, 학습자가 외부에서 관측한 결과는 **Round-Robin 처럼 보이는 패턴** 이다. 그게 정확히 RR 인지 CFS 의 결과인지, FCFS 가 아니고 Priority 도 아닌 근거가 무엇인지를 로그 타임스탬프와 진행률 변화로 짚어내는 것이 보너스 미션의 핵심이다.

| 알고리즘 | 한 줄 정의 | 외부 관측 시그니처 |
| --- | --- | --- |
| **FCFS** (First-Come First-Served) | 도착 순서대로 끝까지 | 한 스레드가 완료될 때까지 다른 스레드 진행률 0% |
| **Round-Robin** | 각자 정해진 타임퀀텀만큼 → 다음 | A → B → C → A → B → C … 균등 교대 |
| **Priority** | 우선순위 높은 놈 먼저 | 특정 스레드만 빠르게 진행, 나머지 정체 |
| **CFS** | `vruntime` 가장 작은 놈 우선 | RR 과 비슷하나 nice 값/부하에 따라 비율이 미세 변동 |

### 3.4 Watchdog — CPU 과점유 보호 정책

`agent-leak-app` 의 [`Watchdog`](용어집.md#watchdog) 은 자기 자신의 CPU 사용률을 주기적으로 측정하다가 `CPU_MAX_OCCUPY` 를 넘기면 [**SIGTERM**](용어집.md#signal) 으로 자신을 종료시킨다. 이게 OS 의 `softlockup detector` 가 아니라 **앱 내부 정책** 이라는 점을 분명히 해야 한다.

| 구분 | 트리거 | 시그널 |
| --- | --- | --- |
| 커널 softlockup | CPU 코어 하나가 일정 시간 idle 0% | 보통 경고 메시지 (`watchdog: BUG: soft lockup`) |
| Watchdog (앱) | 프로세스 자신의 CPU% > `CPU_MAX_OCCUPY` | SIGTERM → 자체 종료 |

> 🔑 **SIGTERM vs SIGKILL**: SIGTERM(15) 은 "정상 종료 요청" — 프로세스가 신호를 받고 cleanup 할 시간이 있음. SIGKILL(9) 은 "지금 즉시 죽임" — 무시 불가, cleanup 못 함. `MemoryGuard` 가 SIGKILL 을 쓰는 이유는 메모리가 이미 한계에 도달해 cleanup 도 추가 메모리를 요구할 수 있어 위험하기 때문. `Watchdog` 이 SIGTERM 인 이유는 CPU 만 문제이므로 정상 정리 시간을 줘도 안전하기 때문.

## 4. 동시성 · 락 · 데드락

### 4.1 왜 락이 필요한가 — 공유 자원 문제

[스레드](용어집.md#thread)가 같은 메모리를 공유하면 좋은 점(빠른 통신)과 나쁜 점([**Race Condition**](용어집.md#race-condition))이 함께 온다.

```python
balance = 100      # 공유 변수

# 스레드 A          # 스레드 B
tmp = balance       tmp = balance
tmp += 50           tmp += 50
balance = tmp       balance = tmp
```

둘 다 동시에 실행되면 `balance` 가 200 이 되어야 할 것 같지만 150 으로 끝날 수 있다 — 두 스레드가 같은 `tmp = balance` 시점의 값을 읽었기 때문. 이를 막으려고 [**임계구역(Critical Section)을 락(mutex)으로 보호**](용어집.md#lock-mutex)한다.

```python
with lock:
    balance += 50
```

동시에 N 명까지 허용해야 한다면 [세마포어(Semaphore)](용어집.md#semaphore)를 쓴다.

### 4.2 데드락의 4대 필요조건 (Coffman conditions)

[데드락](용어집.md#coffman)은 **네 조건이 동시에 모두 성립할 때만** 일어난다. 하나라도 깨면 데드락은 발생 불가능.

| 조건 | 의미 | 깨는 방법(예) |
| --- | --- | --- |
| **상호 배제** (Mutual Exclusion) | 자원을 한 번에 한 스레드만 쓸 수 있음 | 락 없이 동시 접근 가능한 자료구조 사용 ([lock-free](용어집.md#lock-free)) |
| **점유 대기** (Hold and Wait) | 락을 잡은 채 다른 락을 또 기다림 | 필요한 락을 한꺼번에 잡거나 하나도 안 잡고 시작 |
| **비선점** (No Preemption) | 잡힌 락은 강제로 뺏을 수 없음 | 타임아웃 적용 ([`try_lock`](용어집.md#try-lock)) — 일정 시간 후 포기 |
| **순환 대기** (Circular Wait) | A→B→A 같은 사이클이 생김 | **락에 전역 순서를 매기고 항상 작은 번호부터 잡기** |

> 🔑 **이번 미션의 데드락**: `MULTI_THREAD_ENABLE=true` 일 때 두 워커 스레드가 락 A 와 B 를 서로 다른 순서로 잡으려 해서 순환 대기가 형성된다. `false` 로 끄면 멀티스레드 자체가 없으므로 4대 조건의 첫 번째인 상호 배제도 일어나지 않아 자동으로 회피된다.

### 4.3 식사하는 철학자 문제 — 데드락의 표준 비유

5명이 원탁에 앉아 있고 사이사이에 포크 5개가 놓여 있다. 식사는 양손에 포크를 들어야 가능. 모두가 동시에 "왼쪽 포크를 먼저 든다" 라고 하면 모두가 왼쪽을 든 채 오른쪽을 무한히 기다린다 → 데드락.

이 비유는 다익스트라(1965)가 만든 [**식사하는 철학자 문제**](용어집.md#dining-philosophers)이고, 위의 4대 조건이 모두 등장한다:

- 포크는 1명만 쓸 수 있다 → 상호 배제
- 왼쪽을 든 채 오른쪽을 기다린다 → 점유 대기
- 들고 있는 포크를 강제로 뺏지 않는다 → 비선점
- 1→2→3→4→5→1 의 사이클 → 순환 대기

해결책: **포크에 번호를 매기고 모두 작은 번호부터 잡게 한다** (순환 대기 차단). 참고로 데드락과 헷갈리기 쉬운 개념으로 [기아(Starvation)](용어집.md#starvation)가 있는데, 이는 *전체* 가 멈추는 게 아니라 *특정 스레드만* 영영 자원을 못 받는 상태로 구분된다.

### 4.4 어떻게 외부에서 데드락을 진단하는가

학습자의 핵심 도전 — 소스 코드 디컴파일이 금지된 상황에서 **외부 관측만으로** 데드락을 증명해야 한다.

| 증거 | 어떻게 수집 | 무엇을 의미하는가 |
| --- | --- | --- |
| PID 존재 | `ps -ef \| grep agent-leak` | 프로세스는 죽지 않았다 (단순 크래시가 아님) |
| 스레드 CPU 0% | `top -H -p $PID` 또는 `ps -L -p $PID -o lwp,pcpu` | 스레드들이 CPU 를 쓰지 않고 자고 있다 |
| [`wchan`](용어집.md#wchan) = `futex_wait_queue_me` | `ps -L -p $PID -o lwp,wchan:30` | 커널 안에서 [futex](용어집.md#futex) 대기 중 (= 락 대기) |
| [`stat`](용어집.md#stat-codes) = `Sl` | `ps -p $PID -o stat` | Sleeping + multi-thread |
| 마지막 로그 = `WAITING/BLOCKED` | `tail -n 30 $AGENT_LOG_DIR/*.log` | 락 획득 시도에서 멈췄다 |
| 외부 요청 무응답 | `curl --max-time 3 http://0.0.0.0:15034` | 이벤트 루프도 멈춰 응답조차 못 한다 |

위 6 가지가 동시에 성립할 때 비로소 **"교착 상태"** 라는 결론이 정당화된다. 단순히 "응답이 없다" 만으로는 부족하다 — 살아있는 좀비, 무한 루프, 외부 I/O 대기 등 다른 원인일 수 있다.

### 4.5 `wchan` 이 결정적인 이유

[`wchan` 컬럼](용어집.md#wchan)은 그 스레드가 **커널의 어느 함수에서 잠들었는가** 를 보여준다.

| `wchan` 값 | 의미 |
| --- | --- |
| `futex_wait_queue_me`, `do_futex` | **유저 공간 락(mutex) 대기** ← 데드락 후보 |
| `pipe_wait`, `sock_recvmsg` | I/O 대기 (정상) |
| `do_nanosleep`, `hrtimer_nanosleep` | sleep() 호출 중 (정상) |
| `-` | 실행 중이거나 종료 |

여러 스레드가 동시에 `futex_*` 에 머물러 있다 = "**서로 다른 락을 기다리며 영원히 멈춤**" 의 강력한 증거가 된다. 이는 Python `threading.Lock`, C++ `std::mutex` 등 거의 모든 유저 공간 락이 내부적으로 [futex](용어집.md#futex)를 쓰기 때문에 가능한 분석이다.

## 5. 시스템 관제 도구 정리 (이번 미션에서 자주 쓰는 것 위주)

| 도구 | 무엇을 본다 | 미션에서 활용 |
| --- | --- | --- |
| [`top` / `top -H`](용어집.md#top-ps) | CPU·메모리 실시간, `-H` 는 스레드 단위 | OOM/CPU/Deadlock 모두 — 특히 데드락은 `-H` 필수 |
| `ps -ef \| grep` | 프로세스 존재 / cmdline | PID 살아 있는지 확인 (데드락 1차 증거) |
| `ps -L -p $PID -o lwp,pcpu,stat,wchan:30` | 스레드 별 wchan | **데드락 증명의 결정적 증거** |
| `ps -o pid,vsz,rss,pcpu,pmem -p $PID` | 프로세스 메모리/CPU 정밀 | OOM 시 RSS 우상향 패턴 |
| `free -m` | 시스템 전체 메모리 | OOM 시 swap 사용 여부 |
| [`vmstat 1`](용어집.md#vmstat) | CPU/MEM/Swap/IO 초당 변화 | OOM 직전 swap 폭증, [쓰래싱](용어집.md#swap-thrashing) 확인 |
| [`dmesg \| grep -i kill`](용어집.md#dmesg) | 커널 메시지 | 커널 OOM Killer 발동 여부 (이번엔 MemoryGuard 이므로 안 보일 가능성 큼) |
| `/proc/$PID/status` | 프로세스 모든 상태(Vm*, Threads, State) | RSS, 스레드 수, 상태를 한 화면에서 |
| `/proc/$PID/stack` | 현재 커널 콜 스택 | 데드락 시 락 함수 호출 흔적 |

> 🔑 **모든 도구의 베이스는 [`/proc`](용어집.md#proc)**. `top`, `ps`, `free` 는 결국 `/proc/<pid>/...` 와 `/proc/meminfo`, `/proc/stat` 의 텍스트를 파싱해 보여주는 래퍼다. 한 번씩 `cat /proc/$PID/status` 로 원본을 봐 보면 도구가 어디서 숫자를 가져오는지가 손에 잡힌다.

## 6. 미션 3대 장애 — 이론과 관제 데이터의 매핑

학습자가 머릿속에서 다음 표를 그대로 떠올릴 수 있어야 한다.

### 6.1 OOM (Memory Leak)

| 레이어 | 무엇이 일어나는가 | 관제 시그니처 |
| --- | --- | --- |
| 앱 | 전역 컨테이너에 객체가 계속 append, 참조 해제 없음 | 로그에 일반 메시지 (눈에 안 띔) |
| OS | RSS 가 우상향 → DRAM 한계 근접 → swap-out 시작 → 쓰래싱 | `vmstat` 의 `si/so` 가 0 이상으로 튀어 오름 |
| 앱 정책 | RSS > `MEMORY_LIMIT` 감지 → `SELF-TERMINATED` 로그 후 자체 SIGKILL | `[MemoryGuard] Memory limit exceeded …` |

### 6.2 CPU Spike

| 레이어 | 무엇이 일어나는가 | 관제 시그니처 |
| --- | --- | --- |
| 앱 | 무한 루프 또는 [busy-wait](용어집.md#busy-wait) 발생, sleep/yield 없음 | 로그 정상 진행 후 갑자기 침묵 직전 |
| OS | 해당 스레드가 자기 타임 슬라이스를 끝까지 사용, 다른 작업 대기 → CFS 가 균형 잡으려 하지만 한 코어는 사실상 점유 | `top` 의 그 프로세스 `%CPU` 가 ~100% 근처 (단일 코어) |
| 앱 정책 | `%CPU` > `CPU_MAX_OCCUPY` 감지 → SIGTERM 자기 종료 | `[Watchdog] CPU threshold exceeded … SIGTERM` |

### 6.3 Deadlock

| 레이어 | 무엇이 일어나는가 | 관제 시그니처 |
| --- | --- | --- |
| 앱 | 스레드 A 가 lock1 잡고 lock2 기다림, 스레드 B 가 lock2 잡고 lock1 기다림 | 로그가 `WAITING/BLOCKED` 에서 정지 |
| OS | 두 스레드 모두 futex 시스템콜 안에서 무한 슬립 — CPU 도 안 쓰고 메모리도 안 변함 | `top -H` 의 스레드 `%CPU = 0`, `wchan = futex_wait_queue_me` |
| 앱 정책 | 데드락 감지기 없음 → 영원히 살아있는 좀비 | 외부 `curl` 타임아웃, 마지막 로그 갱신 정지 |

## 7. 학습 체크리스트 — 미션 끝난 뒤 스스로 확인

리포트를 다 쓰고 나서, 아래 질문에 **자기 말로** 답할 수 있어야 한다. 답이 막히면 그 부분을 다시 읽고 코드/로그를 재확인하거나, 해당 [용어집](용어집.md) 항목을 다시 읽을 것.

### 7.1 메모리

- [ ] Heap, Stack, Data 영역의 차이를 한 줄씩 설명할 수 있다.
- [ ] RSS 와 VSZ 의 차이를 사례로 설명할 수 있다.
- [ ] 가상 메모리가 왜 "환상" 인지, 페이지 폴트가 무엇인지 설명할 수 있다.
- [ ] 커널 OOM Killer 와 이번 미션의 MemoryGuard 가 다른 메커니즘임을 구분할 수 있다.
- [ ] `MEMORY_LIMIT` 상향이 왜 임시 조치이지 근본 해결이 아닌지 설명할 수 있다.
- [ ] 쓰래싱이 무엇이고 왜 시스템 마비로 이어지는지 설명할 수 있다.
- [ ] lazy allocation 때문에 VSZ 와 RSS 가 다르다는 사실을 사례로 설명할 수 있다.

### 7.2 CPU / 스케줄링

- [ ] 프로세스와 스레드의 차이를 5가지 항목으로 비교할 수 있다.
- [ ] `top` 의 `%CPU` 가 왜 100% 를 넘을 수 있는지 설명할 수 있다.
- [ ] CFS 의 핵심 직관("vruntime 가장 작은 놈") 을 한 줄로 말할 수 있다.
- [ ] FCFS / RR / Priority 의 외부 관측 시그니처를 구분할 수 있다.
- [ ] SIGTERM 과 SIGKILL 의 차이, Watchdog 이 왜 SIGTERM 인지 설명할 수 있다.
- [ ] 컨텍스트 스위치가 무엇이고 왜 비용이 드는지 설명할 수 있다.
- [ ] 타임 슬라이스의 길이가 응답성과 처리량에 어떤 트레이드오프를 만드는지 설명할 수 있다.

### 7.3 동시성 / 데드락

- [ ] Race condition 의 예시를 코드로 그려 보일 수 있다.
- [ ] 데드락 4대 조건을 외울 수 있고, 각각을 깨는 전략을 하나씩 말할 수 있다.
- [ ] 식사하는 철학자 문제로 데드락을 설명할 수 있다.
- [ ] 외부에서 데드락을 증명하기 위해 어떤 6 가지 증거를 모아야 하는지 말할 수 있다.
- [ ] `wchan = futex_wait_queue_me` 가 왜 결정적 단서인지 설명할 수 있다.
- [ ] 뮤텍스와 세마포어의 차이를 사례로 설명할 수 있다.
- [ ] 데드락과 기아(starvation)가 다른 개념임을 설명할 수 있다.

### 7.4 운영 도구

- [ ] `ps`, `top`, `/proc` 이 어떤 관계인지 (`/proc` 이 원본) 설명할 수 있다.
- [ ] 데드락 진단에 `top -H` 와 `ps -L` 이 필수인 이유를 말할 수 있다.
- [ ] `monitor.sh` 가 모은 데이터가 왜 장애 분석의 "최초 단서" 가 되는지 설명할 수 있다.
- [ ] `vmstat` 의 `si/so` 가 무엇이고 쓰래싱 감지에 어떻게 쓰는지 설명할 수 있다.
- [ ] `dmesg` 에서 무엇을 찾아야 하는지 (특히 OOM Killer) 설명할 수 있다.
