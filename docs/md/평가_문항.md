## 항목 2 — 관제·도구 활용 (구술) {#item2}

> 핵심: 명령을 *외워서* 친 게 아니라, **무엇을 보려고 그 도구·옵션을 골랐는지** 를 말로 설명할 수 있는가.

### Q2-1. `monitor.sh`(monitor.log)에서 메모리 증가 패턴을 추적한 명령·추출 방법은? {#q2-1}

- **채점 포인트**: ① 라이브 관찰(`tail -f`)과 ② 수치 추출(`grep`/`awk`/`report.sh`)을 구분하는가. 리포트의 표/그래프 수치가 어디서 나왔는지 댈 수 있는가.
- **모범답안 요지**:
  - 라이브 추세는 `tail -f /var/log/agent-app/monitor.log` 로 MEM% 가 분 단위로 오르는 걸 본다.
  - 리포트에 넣을 **수치 시계열** 은 해당 PID 라인만 뽑아 MEM% 만 잘라낸다:
    ```bash
    grep "PID:<pid>" /var/log/agent-app/monitor.log | awk -F'MEM:' '{print $2}' | awk -F'%' '{print $1}'
    ```
  - Max/도달 시각·평균 같은 **통계** 는 `bash src/report.sh "<START>" "<END>"` 로 구간을 지정해 `STATISTICS REPORT`(`[Memory] Maximum 96.8% at 14:10:00 …`)를 얻는다.
  - 핵심 해석: CPU 는 평탄(1~2%)한데 **MEM 만 단조 증가** → "계산 폭주가 아니라 메모리 누수" 라는 1차 분기.
- **근거**: [실험 절차서 §1-2](실험_절차서.md) · `src/report.sh` · [README §5.4](README.md) · 용어집 [proc](용어집.md#proc) · [top·ps](용어집.md#top-ps)

### Q2-2. CPU 사용률을 확인하려 고른 도구와 옵션의 의미는? {#q2-2}

- **채점 포인트**: `top`/`ps` 의 옵션을 *왜* 붙였는지, per-process %CPU 의 정규화(1코어=100%)를 아는지, `%us` vs `%sy` 를 구분하는지.
- **모범답안 요지**:
  - `top -bn1 -o %CPU` — `-b`(배치, 비대화형) `-n1`(1회) `-o %CPU`(CPU 내림차순 정렬)로 폭주 프로세스를 맨 위에 띄운다.
  - `ps -o pid,pcpu,pmem,stat,etime,cmd -p <pid>` — `stat`(상태) `etime`(가동시간)까지 한 줄에. **`STAT=R`(Running)** = I/O 대기 없이 계속 실행 = busy-loop 시그니처.
  - top 헤더의 **`%Cpu(s) 92.3 us`** 는 사용자 모드 CPU(커널 시간 아님) → "앱 코드가 도는 중". per-process `%CPU 91.2` 는 **한 코어의 91%**(4-vCPU 시스템 전체로는 ≈22.8%)임에 주의 — 시스템 전체 압박은 `load average` 로 본다.
- **근거**: [02 리포트 §2-3](02_cpu_report.md) · `cpu_top_ps.txt` · [이론 §3.2 %CPU가 100%를 넘는 이유](이론_지식.md#32-cpu-cpu-100) · 용어집 [STAT 코드](용어집.md#stat-codes) · [busy-wait](용어집.md#busy-wait)

### Q2-3. "살아있지만 멈춰 있는" 상태를 진단한 도구와 순서, 판단 흐름은? {#q2-3}

- **채점 포인트**: 무작정 명령을 나열하는 게 아니라 **"죽었나? → 일하나? → 왜 멈췄나?"** 의 *논리 순서* 로 도구를 골랐는지.
- **모범답안 요지** (진단 6단계):
  1. `ps -ef | grep agent` — **PID 가 살아있나?** (있음 → crash 아님)
  2. `ps -p <pid> -o stat,etime` — **`STAT=Sl`**(Sleeping+multi-thread) → 죽은 게 아니라 *잠듦*
  3. `top -H -bn1 -p <pid>` — **스레드별 `%CPU=0`**, `TIME+` 누적 정지 → 일을 *안 함*
  4. `ps -L -p <pid> -o lwp,stat,wchan` — **`wchan=futex_wait_queue_me`** → 사용자 공간 락 대기
  5. `tail` 앱 로그 — 마지막 라인이 **`WAITING for LOCK_*`** 에서 정지
  6. `curl --max-time 3 http://127.0.0.1:15034` — **타임아웃** → 외부에서도 무응답
  - 결론 흐름: *살아있다(1) → 잠들었다(2,3) → 락을 기다린다(4) → 서로를 기다린다(5) → 밖에서도 죽은 셈(6)*.
- **근거**: [03 리포트 §2](03_deadlock_report.md) · `deadlock_ps_top.txt` · [이론 §5.4 외부에서 데드락 진단](이론_지식.md#54) · [이론 §5.5 wchan](이론_지식.md#55-wchan) · 용어집 [wchan](용어집.md#wchan) · [futex](용어집.md#futex)

## 항목 3 — 이론 이해도 (구술) {#item3}

### Q3-1. 메모리 누수 시 앱의 메모리 보호 정책이 프로세스를 강제 종료하는 *이유* 는? {#q3-1}

- **채점 포인트**: 커널 **OOM Killer** 와 앱 **MemoryGuard** 를 구분하는가. "왜 미리 스스로 죽나"(시스템 전체 보호)를 설명하는가.
- **모범답안 요지**:
  - 힙에 쌓인 객체를 해제하지 않으면 [RSS](용어집.md#vsz-vs-rss)가 단조 증가한다. OS 엔 정상 점유로 보이므로, 끝내 RAM 이 고갈되면 **커널 OOM Killer** 가 가장 큰 점유자를 `SIGKILL`(즉사·`dmesg` 흔적)로 죽인다 — *우리가 죽이고 싶지 않은 프로세스* 까지 끌려갈 수 있다.
  - 그래서 앱은 커널 OOM 이 발동하기 **전에** `RSS > MEMORY_LIMIT` 를 스스로 감지해 `SELF-TERMINATED` 로그를 남기고 **graceful 종료(self-terminate)** 한다 → 한 프로세스만 질서있게 정리해 **시스템 전체 불안정을 예방**.
  - 핵심 구분: *정돈된 `[CRITICAL]`·`SELF-TERMINATED` 로그가 남았다 = SIGKILL(로깅 불가)이 아니라 앱이 스스로 종료한 증거.*
- **근거**: [이론 §2.4 OOM 두 트리거](이론_지식.md#24-oom-out-of-memory) · [이론 §7.1](이론_지식.md#71-oom-memory-leak) · 용어집 [OOM Killer](용어집.md#oom-killer) · [시그널](용어집.md#signal) · [01 리포트 §3](01_oom_report.md)

### Q3-2. CPU 과점유 시 단일 프로세스를 종료하는 것이 시스템 보호에 필요한 *근거* 는? {#q3-2}

- **채점 포인트**: 한 프로세스의 폭주가 *다른 작업의 응답성* 에 미치는 영향(스케줄러 관점), `SIGTERM`(graceful)을 쓰는 이유를 설명하는가.
- **모범답안 요지**:
  - 리눅스 [CFS](용어집.md#cfs)는 Runnable 프로세스에 CPU 시간을 *공평히* 나눈다. 한 프로세스가 busy-loop 로 한 코어를 100% 점유하면 같은 우선순위의 다른 작업이 받을 몫이 줄어 **시스템 응답성이 저하**(load average 상승)된다.
  - 앱 **Watchdog** 은 `%CPU > CPU_MAX_OCCUPY` 가 일정 시간 지속되면 **자기 자신에게 `SIGTERM`** 을 보내 graceful shutdown 한다. 메모리와 달리 CPU 만 문제이므로 정상 정리 시간을 줘도 안전 → `Caught signal SIGTERM` 정리 로그를 남길 수 있다.
  - 임시 조치(`CPU_MAX_OCCUPY` 상향)는 임계치만 미룰 뿐 폭주 루프 자체를 멈추지 못한다(여전히 종료됨).
- **근거**: [이론 §3.3 CFS](이론_지식.md#33-linux-cfs) · [이론 §3.4 Watchdog](이론_지식.md#34-watchdog-cpu) · [이론 §7.2](이론_지식.md#72-cpu-spike) · 용어집 [Watchdog](용어집.md#watchdog) · [02 리포트 §3](02_cpu_report.md)

### Q3-3. 교착상태(Deadlock)를 "상호 배제" 와 "순환 대기" 로 설명하면? {#q3-3}

- **채점 포인트**: Coffman 4조건을 *이번 사례에 대응시켜* 설명하는가. 특히 ①상호배제·④순환대기를 그래프로 짚는가.
- **모범답안 요지**:
  - **상호 배제**: `LOCK_X`·`LOCK_Y` 는 한 번에 한 스레드만 쥘 수 있는 [뮤텍스](용어집.md#lock-mutex).
  - **점유 대기**: Worker-A 는 `LOCK_X` 를 *쥔 채* `LOCK_Y` 를 요청, Worker-B 는 그 반대.
  - **비선점**: 외부에서 락을 강제 회수하지 않음(보유자가 자발적으로 release 해야).
  - **순환 대기**: `A → (LOCK_Y 대기) → B → (LOCK_X 대기) → A` 의 닫힌 사이클 → 둘 다 영원히 대기.
  - 4조건은 *동시에* 성립해야 하므로 *하나만 깨면* 회피된다. 임시 조치(`MULTI_THREAD_ENABLE=false`)는 상호배제 속성을 없애는 게 아니라 **직렬화로 동시 경쟁을 제거** → 점유대기·순환대기가 성립 못 함.
- **근거**: [이론 §5.2 Coffman 4조건](이론_지식.md#52-4-coffman-conditions) · [이론 §5.3 식사하는 철학자](이론_지식.md#53) · 용어집 [Coffman](용어집.md#coffman) · [식사하는 철학자](용어집.md#dining-philosophers) · [03 리포트 §3](03_deadlock_report.md)

### Q3-4. 로그에서 스레드 간 순환 의존(A→B, B→A)을 어떻게 *파악* 했는지 추적 과정은? {#q3-4}

- **채점 포인트**: 마지막 4줄 로그만으로 락 의존 그래프를 *복원* 하는 추론을 보이는가. `wchan` 으로 외부에서 확증하는가.
- **모범답안 요지**:
  - 앱 로그 마지막 4줄: `[A] acquired LOCK_X` / `[B] acquired LOCK_Y` / `[A] WAITING for LOCK_Y` / `[B] WAITING for LOCK_X` → 각 스레드가 *보유한 락* 과 *기다리는 락* 을 적어 그래프를 만들면 `A:보유 X·대기 Y`, `B:보유 Y·대기 X` → **순환**.
  - 외부 확증: `ps -L -o wchan` 의 두 워커가 모두 `futex_wait_queue_me` → "사용자 공간 락 대기 중" 이 코드 없이도 입증된다. 커널엔 데드락 자동 탐지가 없으므로 **외부 관측자가 '모두 잠듦 + 진행 없음'** 을 감지해야 한다.
- **근거**: [이론 §5.5 wchan이 결정적인 이유](이론_지식.md#55-wchan) · [이론 §7.3](이론_지식.md#73-deadlock) · `deadlock_app.log` · `deadlock_ps_top.txt` · [03 리포트 §3-1](03_deadlock_report.md)

## 항목 4 — 응용·심화 (가정형) {#item4}

> 정해진 정답이 없다. **근거를 댄 판단** 이면 된다. 아래는 이 저장소의 증거·이론으로 뒷받침되는 *모범 방향* 이다.

### Q4-1. 운영 서버였다면, 메모리 누수를 *장애 전에* 탐지하도록 `monitor.sh` 를 어떻게 개선? {#q4-1}

- **채점 포인트**: 절대 임계(MEM 90%)는 *이미 늦다* 는 점을 아는가. *추세/예측* 기반 경보를 제안하는가.
- **모범답안 방향**:
  - **추세(기울기) 경보**: 최근 N개 샘플의 MEM% 가 *단조 증가* 하거나 회귀 기울기가 양(+)으로 지속되면 경보 — 이번 누수는 3분당 +30%p 라 5%→35% 구간에서 이미 탐지 가능.
  - **도달 예측(ETA)**: 현재 기울기로 `MEMORY_LIMIT` 도달 시각을 추정해 "약 N분 후 OOM 예상" 알림.
  - **변화율 임계**: 분당 +X%p 초과 상승 시 WARNING(절대값과 별개).
  - **평탄도 경보**(보너스): 데드락처럼 *너무 평탄한 것* 도 이상신호 → 분산이 0 에 수렴하면 freeze 의심.
- **근거**: `evidence/oom_monitor.log`(단조 선형) · `src/report.sh`(이미 Max/평균 집계 → 기울기 확장 용이) · [이론 §7.1](이론_지식.md#71-oom-memory-leak)

### Q4-2. 세 장애 중 실서비스에서 가장 치명적인 것은? 이유와 근본 예방책은? {#q4-2}

- **채점 포인트**: "죽는 장애" 와 "조용히 멈추는 장애" 의 운영상 차이를 아는가.
- **모범답안 방향** (권장: **Deadlock**):
  - OOM·CPU 는 앱이 **스스로 종료** 하며 명확한 로그를 남긴다 → 재시작·알림·crash-loop 감지가 쉽다("죽었다" 는 분명함).
  - Deadlock 은 **PID 살아있고 포트도 LISTEN** → 존재/포트 기반 **헬스체크를 통과**, CPU/MEM 도 임계 미달이라 WARNING 도 안 뜬다 → 일반 관제가 **놓치는 '조용한 장애'**. 사용자 요청만 조용히 타임아웃된다.
  - **근본 예방**: ① 락 획득 순서 통일(순환대기 차단) ② `lock.acquire(timeout)` ③ **진행성(liveness) 헬스체크** — 단순 포트가 아니라 "최근 N초 내 작업 완료 로그가 있는가" 로 freeze 를 잡는다.
  - (반론 여지: 메모리 고갈은 *시스템 전체* 를 위협하므로 OOM 이 가장 치명적이라는 답도 근거가 분명하면 인정 — 영향 범위 vs 탐지 난이도 중 무엇을 기준 삼느냐의 문제.)
- **근거**: `deadlock_monitor.log`(헬스체크 통과·임계 미달 주석) · [이론 §7.3 "데드락 감지기 없음"](이론_지식.md#73-deadlock)

### Q4-3. 같은 서버에서 OOM 과 Deadlock 이 *동시에* 발생하면 트리아지 순서는? {#q4-3}

- **채점 포인트**: 우선순위의 *근거*(시간 압박·영향 범위)를 대는가.
- **모범답안 방향** (권장: **OOM 먼저**):
  - OOM 은 *임박한 강제 종료* + 메모리는 시스템 전체 공유 자원이라 고갈 시 **커널 OOM Killer 가 무고한 프로세스까지** 죽일 수 있다 → 시간 압박·전체 영향 모두 큼. 먼저 메모리를 확보(재시작/한도조정)해 시스템을 안정화.
  - Deadlock 은 자원을 *추가로 더 먹지 않고* 정지 상태라 당장 번지지 않는다(프로세스도 살아있음) → 시스템 안정화 후 `ps`/`top`/`wchan` 으로 *차분히* 진단·재시작.
  - **예외**: 데드락이 *공유 핵심 락/자원* 을 쥐고 다른 서비스까지 멈춘다면 영향 범위가 역전 → 데드락 우선. *영향 범위로 판단* 하는 게 원칙.
- **근거**: [이론 §1.2 메모리는 항상 느리다/공유 자원](이론_지식.md#12) · [이론 §2.4 OOM Killer가 최대 점유자 SIGKILL](이론_지식.md#24-oom-out-of-memory)

### Q4-4. 소스 코드를 고칠 수 있다면, 장애 유형별 *코드 레벨* 개선은? {#q4-4}

- **채점 포인트**: 임시 조치(환경변수)와 근본 해결(코드)을 구분하고, 장애별로 다른 처방을 내는가.

| 장애 | 코드 레벨 개선 |
| ---- | -------------- |
| **OOM** | 처리 끝난 데이터 `del`/`pop`/`clear()`, `collections.deque(maxlen=…)`·캐시 TTL/LRU, 장기 데이터는 외부 캐시(Redis)로 오프로드, `tracemalloc`/`memray` 로 누수 객체 식별, RSS 회귀 테스트 가드 |
| **CPU** | `heavy_compute()` 반복 상한/청크 처리 후 `time.sleep(0)`·`await` 양보, 알고리즘 최적화(O(n²)→O(n log n)), busy-wait→[컨디션 변수](용어집.md#condition-variable), `py-spy` 프로파일로 폭주 함수 식별 |
| **Deadlock** | **락 획득 순서 통일**(자원 순서화 → 순환대기 제거), [`lock.acquire(timeout)`](용어집.md#try-lock) 후 backoff, 단일 통합 락/[lock-free](용어집.md#lock-free), 데드락 감지 옵저버(진행 로그 정지 시 재시작) |

- **근거**: 각 리포트의 §4-3 "근본 해결을 위한 제안" — [01](01_oom_report.md) · [02](02_cpu_report.md) · [03](03_deadlock_report.md)

### Q4-5. 다시 처음부터 수행한다면 트러블슈팅에서 무엇을 다르게? {#q4-5}

- **채점 포인트**: 회고가 *구체적 개선* 으로 이어지는가(막연한 반성 X).
- **모범답안 방향**:
  - 절대 임계 대신 **추세·평탄도 기반 관제** 를 처음부터 설계(Q4-1).
  - `tail -f` 라이브 관찰에만 의존하지 말고 **`report.sh` 로 수치 추출을 자동화** — Evidence 표를 손이 아니라 도구로.
  - **가설→증거 매핑 표** 를 먼저 세우고 명령을 실행(역추론 훈련) — 명령 나열이 아니라 "무엇을 확인하려는가" 를 먼저.
  - Before/After 를 **2회 이상 반복** 측정해 분산까지 확인.
  - Deadlock 의 *조용한 장애* 특성을 먼저 인지하고 **진행성 헬스체크** 를 우선 점검 대상에 둠.
- **근거**: [실험 절차서 §5 환경변수 조합표](실험_절차서.md) · 각 리포트 §4 · [README §4 분석 흐름](README.md)

## 항목 5 — 보너스 (스케줄링 추론) {#item5}

- **채점 포인트**: 로그 타임스탬프·진행률 패턴으로 FCFS·Priority 를 *기각* 하고 Round-Robin 을 *추론* 하는 논리를 갖췄는가.
- **모범답안 요지**:
  - **FCFS 기각**: A 가 100% 끝나기 전 B 가 진입(progress 20%↔10%) → 먼저 온 작업을 끝까지 처리하지 않음.
  - **Priority 기각**: 세 워커의 누적 CPU 시간(`TIME+`)이 ±0.03s 로 동일 → 독점 없음.
  - **Round-Robin 결론**: 약 50ms 고정 할당량 + `A→B→C` 순환 교체 + `TIME+` 공평 → RR(할당량 ≈ 50ms).
  - 한계: 진짜 Preemptive RR 인지 Cooperative yield 인지는 "할당량을 다 못 채우고 양보하는 케이스" 유무로 추가 검증.
- **근거**: [04 스케줄링 분석](04_scheduling_analysis.md) · `scheduling_workers.log`(51ms 교체·A→B→C) · `scheduling_top_h.txt`(TIME+ 편차 <0.03s) · [이론 §3.3 스케줄러 외부 관측 시그니처](이론_지식.md#33-linux-cfs)
